# Codebase Structure

**Analysis Date:** 2026-02-27

## Directory Layout

```
~/.nanoclaw/
├── src/                      # Orchestrator source (Node.js)
├── container/                # Agent container Dockerfile and scripts
├── container/agent-runner/   # Container entrypoint code
├── skills-engine/            # MCP skill definitions
├── groups/                   # Per-group persistent state
├── data/                     # Runtime data (messages, sessions, IPC)
├── store/                    # WhatsApp auth credentials
├── dist/                     # Compiled JavaScript (generated)
├── docs/                     # Documentation
├── .planning/codebase/       # This directory (architecture analysis)
├── package.json              # Node.js dependencies
└── tsconfig.json             # TypeScript config
```

## Directory Purposes

**`src/`:**
- Purpose: Orchestrator source code (runs on host)
- Contains: TypeScript modules for channels, database, container management, routing
- Key files: `index.ts` (main entry), `config.ts` (constants), `types.ts` (interfaces)

**`src/channels/`:**
- Purpose: Messaging platform integrations
- Contains: `whatsapp.ts` (Baileys SDK), `telegram.ts` (Grammy framework), tests
- Pattern: Each exports a class implementing `Channel` interface

**`container/`:**
- Purpose: Docker image definition and build scripts
- Contains: `Dockerfile`, `build.sh`, `agent-runner/` (TypeScript entrypoint)
- Generated: Container image named `nanoclaw-agent:latest` by `./container/build.sh`

**`container/agent-runner/src/`:**
- Purpose: Code running inside the container, communicates with orchestrator
- Contains: Argument parsing, session management, IPC file writing for cross-group ops
- Not committed: `dist/` folder (built on container creation)

**`container/skills/`:**
- Purpose: MCP skill implementations (tools available to agents)
- Contains: `agent-browser/` (Puppeteer), `google-workspace/`, `gtasks/`
- Mounted: Read-only into container at `/workspace/skills/`

**`skills-engine/`:**
- Purpose: MCP skill loader and registry (runs in container)
- Contains: TypeScript modules that dynamically load skills based on manifest files

**`groups/`:**
- Purpose: Per-group persistent state and agent memory
- Contains: Per-group folders (e.g., `groups/main/`, `groups/project-a/`)
- Structure:
  - `groups/main/` - Main group data (orchestrator can see all groups)
  - `groups/{folder}/` - Individual group state
  - `groups/global/` - Read-only shared state (visible to all groups)
- Mounted: Into containers at `/workspace/group/` (read-write) and `/workspace/global/` (read-only)

**`groups/{folder}/` contents:**
- `CLAUDE.md` - Group-specific agent memory/instructions
- `conversations/` - Conversation logs
- `knowledge/` - Group knowledge base
- `tasks/` - Task definitions
- `daily/` - Daily briefings/summaries
- `plans/` - Planning documents
- `tacit/` - Implicit group context
- `logs/` - Container execution logs

**`data/`:**
- Purpose: Runtime state (not checked in)
- Contains: IPC pipes, session data, environment configs
- Directories:
  - `data/ipc/{groupFolder}/` - Inter-process communication files (messages, tasks)
  - `data/ipc/{groupFolder}/messages/` - JSON files for group actions (register, send, etc.)
  - `data/ipc/{groupFolder}/tasks/` - JSON files for task operations
  - `data/sessions/` - Claude Agent SDK session folders (one per group)
  - `data/accounts/` - WhatsApp credentials
  - `data/env/` - Environment variable files per group

**`store/`:**
- Purpose: WhatsApp authentication state (not checked in)
- Contains: `store/auth/` with multi-file auth state from Baileys SDK
- Sensitive: Never committed

**`dist/`:**
- Purpose: Compiled JavaScript (generated by `npm run build`)
- Generated: From `src/` via TypeScript compiler
- Not committed: Rebuilds on deploy

**`.planning/codebase/`:**
- Purpose: Architecture documentation (this directory)
- Contains: `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, `TESTING.md`, `CONCERNS.md`
- Usage: Read by `/gsd:plan-phase` to understand codebase patterns

## Key File Locations

**Entry Points:**
- `src/index.ts` - Orchestrator main, runs on host. Loads state, connects channels, starts loops
- `container/agent-runner/src/index.ts` - Container entrypoint, runs inside Docker
- `src/whatsapp-auth.ts` - WhatsApp authentication helper, run via `npm run auth`

**Configuration:**
- `src/config.ts` - Constants and env var parsing (NOT secrets; secrets stay in `.env`)
- `src/env.ts` - `.env` file reading helper
- `.env` - Environment variables (not checked in, created during setup)
- `tsconfig.json` - TypeScript compiler options
- `vitest.config.ts` - Test runner config

**Core Logic:**
- `src/index.ts` - Message polling loop, channel coordination, container invocation
- `src/router.ts` - Message formatting (to XML), output sanitization
- `src/container-runner.ts` - Container spawning, volume mounts, output parsing
- `src/group-queue.ts` - Concurrency management, per-group state machine
- `src/db.ts` - SQLite schema and CRUD operations
- `src/ipc.ts` - Container-to-host communication polling and handling

**Channel Implementations:**
- `src/channels/whatsapp.ts` - WhatsApp via Baileys
- `src/channels/telegram.ts` - Telegram via Grammy
- Both implement `Channel` interface from `src/types.ts`

**Database:**
- `src/db.ts` - All database operations
- Located: SQLite file at `data/nanoclaw.db` (created by `initDatabase()`)
- Tables: `messages`, `chats`, `registered_groups`, `sessions`, `scheduled_tasks`, `task_run_logs`, `router_state`

**Testing:**
- `src/*.test.ts` - Unit tests (co-located with source)
- `vitest.config.ts` - Vitest configuration
- Pattern: Test file names match source with `.test.ts` suffix

## Naming Conventions

**Files:**
- TypeScript: `kebab-case.ts` (e.g., `container-runner.ts`, `group-queue.ts`)
- Tests: `{name}.test.ts` (e.g., `db.test.ts`, `routing.test.ts`)
- Config: `{name}.config.ts` or just name (e.g., `tsconfig.json`, `vitest.config.ts`)

**Directories:**
- Lowercase with hyphens: `src/`, `container/`, `groups/`, `data/ipc/`
- Group folders: Alphanumeric start, then alphanumeric/underscore/hyphen (e.g., `main`, `project-a`, `group_1`)

**Functions:**
- camelCase: `runContainerAgent()`, `processGroupMessages()`, `formatMessages()`
- Prefixed underscores for internal/testing: `_setRegisteredGroups()`

**Types and Interfaces:**
- PascalCase: `RegisteredGroup`, `ContainerInput`, `NewMessage`, `Channel`, `GroupQueue`
- File: `src/types.ts` contains all shared types

**Constants:**
- UPPER_SNAKE_CASE: `MAIN_GROUP_FOLDER`, `POLL_INTERVAL`, `CONTAINER_TIMEOUT`
- Location: `src/config.ts`

**Environment Variables:**
- UPPER_SNAKE_CASE: `ASSISTANT_NAME`, `TELEGRAM_BOT_TOKEN`, `CONTAINER_IMAGE`
- File: Read from `.env`, exported from `src/config.ts`

## Where to Add New Code

**New Messaging Channel:**
- Implementation: `src/channels/{name}.ts`
- Must implement: `Channel` interface from `src/types.ts`
- Methods required: `connect()`, `disconnect()`, `sendMessage()`, `isConnected()`, `ownsJid()`
- Optional: `setTyping()`
- Registration: Add to `channels` array in `src/index.ts` main()
- Test: `src/channels/{name}.test.ts` (co-located)

**New Scheduled Task or Cron Job:**
- Implementation: Add trigger in `src/task-scheduler.ts` or create task via IPC
- Database: Tasks stored in `scheduled_tasks` table (created via `createTask()`)
- Execution: Container runs prompt, logs to `task_run_logs`
- Trigger: IPC file `data/ipc/{groupFolder}/tasks/create-task-{uuid}.json`

**New Group Agent Feature:**
- Memory storage: `groups/{folder}/{category}/{document}.md`
- IPC to orchestrator: Write JSON to `data/ipc/{groupFolder}/messages/{action}-{uuid}.json`
- Supported IPC actions: `message` (send to chat), `register-group` (new group), task creation
- Authorization: Main group can act on any group; non-main limited to own group

**New Database Table:**
- Location: Schema definition in `src/db.ts` function `createSchema()`
- CRUD: Add functions to `src/db.ts` (e.g., `createWidget()`, `getWidget()`, `updateWidget()`)
- Export: From `src/db.ts` for import in other modules

**New Utility Function:**
- Location: Create `src/{function-name}.ts` or add to existing module
- Naming: camelCase function names
- Exports: Use named exports for clarity (avoid default exports)

**New Test:**
- Location: Co-located with source as `{source}.test.ts`
- Pattern: Use Vitest syntax, structure with `describe()` and `it()`
- Running: `npm test` runs all tests

## Special Directories

**`dist/`:**
- Purpose: Compiled output from TypeScript
- Generated: By `npm run build`
- Not committed: Ignored in `.gitignore`
- Updated: On deployment via GitHub Actions (triggered by tag)

**`data/` subdirectories:**
- `data/ipc/` - Not committed, created at runtime
- `data/sessions/` - Not committed, contains Claude SDK session state
- Purpose: Runtime state that persists between restarts

**`groups/` structure (committed):**
- `groups/main/` - Committed to git, tracked with group memory
- `groups/{other}/` - Not committed by default (created on registration)
- `groups/global/` - Shared read-only context visible to all groups
- Note: Individual group folders can be added to `.gitignore` or committed depending on setup

**`store/auth/`:**
- WhatsApp credentials, never committed
- Created by `npm run auth` (WhatsApp QR code authentication)

**`.nanoclaw/` directory (outside project):**
- `~/.nanoclaw/` is not this project's root; likely reference in `/nanoclaw/base/`
- Parent folder structure for backup/configuration
