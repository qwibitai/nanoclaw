**2. Fresh session logic (CONC-02):**

Currently: `const sessionId = sessions[group.folder];` — passes the group's last session to every container.

New logic:
- `sendMessage()` returned true (piped to idle container) → that container ALREADY has a session (it was the one that created it). No change needed — idle reuse preserves session. This path doesn't go through `processGroupMessages` at all.
- `processGroupMessages` called (new container) → pass `sessionId: undefined` to `runContainerAgent`. This forces a fresh Claude session. The container will return a `newSessionId` which we save as the group default.

```typescript
// In runAgent:
// Fresh container = fresh session. Don't pass existing sessionId.
// The container will create a new session and return newSessionId.
// We save that as the group's default for future idle-reuse.
const sessionId = undefined; // CONC-02: fresh session per container
```

Wait — but idle-reuse containers already have their session. The session saving logic (`sessions[group.folder] = output.newSessionId`) should still work: the LAST container to produce a `newSessionId` becomes the group default. This is correct — if multiple containers run, the most recent session becomes the default. But since new containers always get fresh sessions anyway, the saved default only matters for idle-reuse (where the container already has the session internally).

Actually, reviewing the container-runner: `sessionId` is passed via stdin as part of `ContainerInput`. The agent-runner inside the container uses it for `--resume`. So:
- New container: pass `sessionId: undefined` → agent starts fresh session → returns `newSessionId`
- Idle-reuse: the container already has its session internally (from its initial run). When it picks up a piped message, it processes it in the same session. No `sessionId` needs to be passed again.

So the change is simple: in `runAgent`, don't pass `sessions[group.folder]` — pass `undefined`.

**But** we still need to save the returned `newSessionId` as the group default:
```typescript
if (output.newSessionId) {
  sessions[group.folder] = output.newSessionId;
  setSession(group.folder, output.newSessionId);
}
```
This stays — it updates the saved default. But since every new container gets a fresh session, this just means "last container's session is the default" which is fine.

**3. Idle timeout per-container (CONC-03):**

Currently the idle timer in `processGroupMessages` calls `queue.closeStdin(chatJid)`. This needs to target the SPECIFIC container:

idleTimer = setTimeout(() => {

The `onProcess` callback in `runContainerAgent` is called with `(proc, containerName)`. We need to also pass `containerId`.

Update the task scheduler to work with the refactored GroupQueue API.

**1. onProcess callback:**

Currently in the `SchedulerDeps` interface:
```typescript
onProcess: (groupJid: string, proc: ChildProcess, containerName: string, groupFolder: string) => void;
```

The task scheduler calls `queue.registerProcess` via `deps.onProcess`. It needs to pass `containerId`.

But wait — tasks are enqueued via `queue.enqueueTask(groupJid, taskId, fn)`. The `fn` callback runs inside `GroupQueue.runTask()`, which creates the `containerId` internally. The task's `fn` doesn't receive `containerId`.

**Solution:** The task scheduler's `fn` calls `runContainerAgent` with an `onProcess` callback. Inside `runTask()`, the GroupQueue creates the containerId and uses `pendingRegistrations` to map it. When `registerProcess` is called (from inside the task fn's `onProcess` callback), GroupQueue looks up the pending registration.

Actually, reviewing the GroupQueue Plan 01 design: `processMessagesFn` receives `containerId` as a parameter. But `enqueueTask`'s `fn` callback doesn't — it's `() => Promise<void>`.

Two approaches:
A) Change task fn signature to receive containerId: `(containerId: string) => Promise<void>`
B) Use the pendingRegistrations pattern for tasks too

**Choose A — it's explicit.** Update `QueuedTask`:
```typescript
interface QueuedTask {
  id: string;
  groupJid: string;
  fn: (containerId: string) => Promise<void>;
}
```

And `enqueueTask`'s parameter type:
```typescript
enqueueTask(groupJid: string, taskId: string, fn: (containerId: string) => Promise<void>): void
```

This means the task scheduler's callback receives `containerId`:

```typescript
queue.enqueueTask(task.chat_jid, task.id, async (containerId: string) => {
  // ... existing task execution code ...
  
  const output = await runContainerAgent(
    group,
    { prompt: task.prompt, sessionId, ... },
    (proc, containerName) =>
      deps.onProcess(task.chat_jid, proc, containerName, task.group_folder, containerId),
    async (streamedOutput) => {
      // ...
      if (streamedOutput.status === 'success') {
        deps.queue.notifyIdle(task.chat_jid, containerId);
      }
      // Close stdin with containerId
      scheduleClose(containerId);
    },
  );
});
```

**NOTE:** This requires updating the `QueuedTask` interface and `runTask` in group-queue.ts to pass `containerId` to `task.fn(containerId)`. If Plan 01 didn't do this, make the change in group-queue.ts here.

**2. Session for tasks:**

Currently: `task.context_mode === 'group' ? sessions[task.group_folder] : undefined`

For tasks, the session behaviour should stay the same — tasks in 'group' mode reuse the group session, 'isolated' mode gets fresh. This is fine because tasks are always new containers (never idle-reuse).

Actually for consistency with CONC-02, consider: should tasks also get fresh sessions? The requirement says "Each concurrent container gets a fresh Claude session." Tasks are containers. But `context_mode: 'group'` explicitly means "use group context/session." 

Keep existing task session logic — it's an intentional feature. CONC-02 applies to message containers. Tasks with `context_mode: 'group'` continue to resume the group session.

**3. closeStdin per-container:**

  // ...
