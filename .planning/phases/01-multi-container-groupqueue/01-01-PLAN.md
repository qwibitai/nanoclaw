Refactor the data model and all core methods in `group-queue.ts`:

**New types (define at top of file, not exported — internal to GroupQueue):**

```typescript
interface ContainerSlot {
  containerId: string;       // unique ID for this slot (use containerName once registered)
  type: 'message' | 'task';
  idleWaiting: boolean;
  process: ChildProcess | null;
  containerName: string | null;
  groupFolder: string | null;
  runningTaskId: string | null;
}

interface GroupState {
  containers: Map<string, ContainerSlot>;  // containerId → slot
  pendingMessages: boolean;
  pendingTasks: QueuedTask[];
  retryCount: number;
}
```

**Key design decisions:**
- `containerId` is a generated string (e.g. `${groupJid}-${Date.now()}-${counter}`) created when `runForGroup` or `runTask` starts. This is the slot key.
- `containerName` is set later via `registerProcess()` (Docker name assigned during spawn).
- A group is "active" when `containers.size > 0`. Remove the `active: boolean`.
- `activeCount` still tracks total containers across ALL groups (for global cap).

**Method-by-method refactor:**

1. `getGroup()` — Initialise with `containers: new Map()` instead of `active: boolean`.

2. `enqueueMessageCheck()` — NEW LOGIC:
   - If shutting down, return.
   - Get group state.
   - Find an idle non-task container: iterate `state.containers` for one with `idleWaiting === true && type === 'message'`. If found, set `pendingMessages = true` and return (the idle container will pick up via `sendMessage()` in the message loop — this matches current flow).
   - If at global cap (`activeCount >= MAX_CONCURRENT_CONTAINERS`), queue as before.
   - Otherwise, call `runForGroup()` to spawn a NEW container (even if others are running for this group).

3. `enqueueTask()` — Similar change:
   - Dedup check: scan ALL `ContainerSlot` entries for `runningTaskId === taskId`, plus `pendingTasks`.
   - If group has ANY containers running and an idle one exists, preempt the idle one (write `_close`) and queue the task for after it exits.
   - If group has containers but none idle, queue the task.
   - If no containers and under cap, run immediately.

4. `registerProcess(groupJid, proc, containerName, groupFolder, containerId)` — Add `containerId` parameter. Look up the specific `ContainerSlot` by `containerId` and set `process`, `containerName`, `groupFolder`. Also update the slot's key in the map to use `containerName` if desired (or keep using `containerId` — simpler).

5. `notifyIdle(groupJid, containerId)` — Add `containerId` parameter. Set `idleWaiting = true` on the SPECIFIC slot. If pending tasks exist, preempt THIS container.

6. `sendMessage(groupJid, text)` — Find ANY idle non-task container in the group. Write IPC file using that container's `groupFolder`. Set `idleWaiting = false` on that specific slot. Return true/false.

7. `closeStdin(groupJid, containerId?)` — If `containerId` provided, close that specific container. If not provided (backward compat for task-scheduler), close ALL idle containers for the group (or the first idle one).

8. `runForGroup()` — Create a new `ContainerSlot` with a generated `containerId`, add to `state.containers`, increment `activeCount`. In the `finally` block: remove the slot from `state.containers`, decrement `activeCount`, call `drainGroup()`.

9. `runTask()` — Same pattern: create slot, add to map, clean up in `finally`.

10. `drainGroup()` — Check pending tasks/messages as before. The key change: it now only runs new containers when there are pending items AND room under the cap. No longer blocked by "active" boolean.

11. `shutdown()` — Iterate all groups' `containers` maps to count detached containers.

**CRITICAL: Backward compatibility of public API.**
The callers (`index.ts`, `task-scheduler.ts`) currently call:
- `registerProcess(groupJid, proc, containerName, groupFolder)` — needs containerId added
- `notifyIdle(groupJid)` — needs containerId added  
- `closeStdin(groupJid)` — needs containerId added (optional for backward compat)
- `sendMessage(groupJid, text)` — NO change needed (finds idle container internally)
- `enqueueMessageCheck(groupJid)` — NO change needed
- `enqueueTask(groupJid, taskId, fn)` — NO change needed

For `registerProcess`, `notifyIdle`, and `closeStdin`: the `containerId` must flow from `runForGroup`/`runTask` through the `processMessagesFn` callback and back. The cleanest approach:

- `runForGroup()` generates `containerId` and passes it through `processMessagesFn` — BUT `processMessagesFn` currently has signature `(groupJid: string) => Promise<boolean>`. 
- CHANGE the callback signature to `(groupJid: string, containerId: string) => Promise<boolean>`.
- Similarly, `runTask` already has the `fn` callback — but the task's own fn doesn't need containerId. The task scheduler calls `queue.registerProcess` directly via `onProcess` — so `onProcess` needs `containerId`.

Actually, simpler approach: `runForGroup` and `runTask` create the containerId internally and pass it to `registerProcess` via a wrapper. The external callers don't need to know about containerId at all — it's an internal concern.

**Revised approach for containerId flow:**
- `runForGroup()` creates `containerId`, creates slot, calls `processMessagesFn(groupJid)`.
- `registerProcess()` is called by the external code (index.ts) with `(groupJid, proc, containerName, groupFolder)` — it needs to know WHICH slot to register against.
- Solution: `runForGroup()` stores `currentContainerId` on the GroupState before calling `processMessagesFn`. `registerProcess()` reads `state._pendingContainerId` to find the right slot. This avoids changing external signatures.

Better solution: Since NanoClaw is single-threaded (Node.js event loop), between `runForGroup()` starting and `registerProcess()` being called (within the same async chain), we can store the `containerId` on a per-group "pending registration" field.

Implement as:
```typescript
private pendingRegistrations = new Map<string, string>(); // groupJid → containerId

// In runForGroup, before calling processMessagesFn:
this.pendingRegistrations.set(groupJid, containerId);

// In registerProcess:
const containerId = this.pendingRegistrations.get(groupJid);
this.pendingRegistrations.delete(groupJid);
// Then assign proc/containerName to the right slot
```

This works because `processMessagesFn` is awaited, and `registerProcess` is called synchronously within it (via the `onProcess` callback in `runContainerAgent`).

For `notifyIdle` and `closeStdin` called from index.ts: these are called with just `groupJid`. For `notifyIdle`, we can look up the container by matching the groupFolder or by finding the most recently registered container. Actually, `notifyIdle` is called from the streaming callback context — at that point there's only ONE container being processed in that callback chain. Use the same `pendingRegistrations` pattern, or better: have `runForGroup` return the `containerId` context through the callback.

**Simplest correct approach:** Change the external callbacks to include `containerId`:
- `processMessagesFn: (groupJid: string, containerId: string) => Promise<boolean>`
- External callers (`index.ts`, `task-scheduler.ts`) receive `containerId` and pass it back to `registerProcess`, `notifyIdle`, `closeStdin`.

This is the cleanest — explicit data flow, no hidden state. Plan 02 will update the callers.

After implementing Task 1, verify these edge cases are handled correctly in the code:

1. **Race condition: two `enqueueMessageCheck` calls before either starts a container.** Since Node.js is single-threaded, `runForGroup` is called synchronously before returning. The second call sees `state.containers.size > 0` or the idle check. Verify the idle check only finds containers with `idleWaiting === true` (the just-spawned container is NOT idle).

2. **Global cap reached mid-group:** Group A has 3 containers, Group B has 2 (cap = 5). New message for Group A should queue, not spawn. Verify `activeCount` check happens BEFORE the idle container check — actually no, idle container reuse doesn't increment activeCount (container already counted). So: check for idle first, THEN check global cap.

3. **Container cleanup on error:** If `processMessagesFn` throws, the `finally` block must remove the slot from `state.containers` and decrement `activeCount`. Verify no orphan slots.

4. **Multiple idle containers:** If two containers for the same group are both idle, `sendMessage` should pick one (first found is fine). The other stays idle and eventually times out.

5. **Task dedup across containers:** `runningTaskId` is per-slot. The dedup check in `enqueueTask` must scan ALL slots for the group, not just check a single field.

6. **Drain with multiple containers:** When one container finishes, `drainGroup` should only start new work if there's pending items AND room under the cap. It should NOT interfere with other still-running containers.

Review the implementation from Task 1 and fix any issues found. Add inline comments for the non-obvious edge cases.
