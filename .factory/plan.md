1. Problem
NanoClaw currently relies on CLAUDE.md and session transcripts for memory, but it lacks a durable, queryable memory layer that can learn user facts and reusable procedures across turns, session resets, and task runs. The result is weak long-term recall and no reliable self-improvement loop.

2. Scope / Non-goals
Scope:
- Add a host-side MemoryService that owns ingestion, storage, retrieval, and reflection decisions.
- Add SQLite memory schema for facts, chunks, procedures, and audit events.
- Extend container IPC MCP tools with memory/procedure save/search/patch operations.
- Inject compact memory context into the existing agent prompt path.
- Add reflection triggers after successful turns, archived-session close (/new), and scheduled-task completion.
- Add tests for core memory behavior and integration seams.

Non-goals (v1):
- No reranker.
- No pgvector/Postgres backend.
- No auto-editing repo skills.
- No multi-database infrastructure.

3. Acceptance Criteria
- A dedicated memory SQLite database initializes with tables:
  - memory_items
  - memory_chunks
  - memory_procedures
  - memory_events
  plus an FTS5 index for lexical retrieval.
- Host MemoryService supports:
  - save/patch/search memory items
  - save/patch/search procedures
  - ingestion/chunking from group CLAUDE.md and archived conversations
  - retrieval fusion from lexical + embedding scores with deterministic ranking.
- Container MCP server exposes tools:
  - memory_search
  - memory_save
  - memory_patch
  - procedure_save
  - procedure_patch
  and they complete via host IPC with structured success/error responses.
- Agent-runner prompt path injects a compact memory block containing:
  - top user facts
  - top group/global facts
  - relevant procedures
  - recent-work recap when prompt indicates continuity intent.
- Reflection worker runs on:
  - successful conversational turns
  - /new transcript archive completion
  - scheduled-task completion
  and records accepted actions to memory while filtering temporary chatter/secrets.
- Scope enforcement:
  - group scope default
  - global writes blocked for non-main groups unless explicit/global-allowed context.
- Tests cover chunking, dedupe, scope policy, stale patch protection, fusion behavior, reflection extraction, IPC memory flow, and prompt injection.

4. Technical Approach
- Create src/memory-store.ts for schema and CRUD/query primitives on SQLite (MEMORY_SQLITE_PATH default to store/memory.db).
- Use FTS5 virtual table for lexical search over memory chunks and procedure text.
- Add embedding provider (OpenAI embeddings API; model from MEMORY_EMBED_MODEL default text-embedding-3-large).
- Add retrieval backend abstraction in src/memory-retrieval.ts that performs:
  - lexical recall
  - vector recall
  - reciprocal-rank fusion / normalized score merge
  - scope/kind/source filtering
- Add src/memory-service.ts as orchestration layer:
  - ingestion from group CLAUDE.md + conversations
  - save/patch APIs with confidence/source metadata
  - recent-work lookup from archived chunks
  - reflection extraction + apply pipeline
- Add IPC request/response memory channel:
  - container writes request files under /workspace/ipc/memory-requests
  - host watcher processes via MemoryService and writes /workspace/ipc/memory-responses/<requestId>.json
- Extend container/agent-runner/src/ipc-mcp-stdio.ts memory tools to perform synchronous host roundtrip over request/response files.
- Extend container/agent-runner/src/index.ts prompt assembly to read a per-group memory context snapshot generated by host run path before invocation.

5. Task Decomposition
- T1 Memory Schema + Store primitives
- T2 Retrieval fusion + embedding adapter + ingestion/chunking
- T3 IPC memory tools + host IPC handlers
- T4 Prompt memory injection + reflection triggers
- T5 Tests + verification artifacts

6. Risks
- sqlite-vec extension availability can differ per environment. Mitigation: optional vector-table path with JSON-vector fallback in v1 while preserving retrieval interface.
- Embedding API latency/failures can degrade retrieval. Mitigation: fail-open lexical-only search and event logging.
- Reflection over-writes noisy facts. Mitigation: confidence thresholds, stale patch checks, and excluded content rules.
- Prompt bloat from memory injection. Mitigation: strict top-k and compact formatting.
