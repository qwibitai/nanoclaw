# Tasks
## Summary
url_watch で自動作成した Discord スレッドの chats 未登録による FK 制約違反を解消し、初回エージェント処理を確実に起動できる状態にする。

## Tasks
### 1. URL Watch スレッド初期化の DB 整合性を保証する
- Why: URL 投稿時に thread 作成後の synthetic message 保存で FOREIGN KEY 制約違反が発生し、初回処理が停止する問題を解消するため。
- What: URL 由来で生成された thread JID について、messages 保存前に chats 側の前提データが成立するよう初期化フローを見直す。
- Done when: url_watch チャンネルで URL を投稿した際、thread 作成から synthetic message 保存、初回処理キュー投入までが中断なく実行される。

### 2. URL Watch スポーン処理の成功条件と失敗時一貫性を揃える
- Why: 処理途中の失敗で thread だけ作成される/起動が走らない等の不整合を防ぎ、再発時の切り分けを可能にするため。
- What: spawn 成功時のみ子 group 登録と初回処理起動が進むこと、失敗時は予約解放・通常保存・エラーログの振る舞いが一貫するよう整理する。
- Done when: createThread 失敗や初期化失敗の各ケースでデータ不整合が残らず、期待したフォールバック挙動とログが確認できる。

### 3. URL Watch の回帰検知範囲を強化する
- Why: 今回の不具合は既存テストで検出できなかったため、保存順序や前提欠落の再発を自動検知できるようにする必要があるため。
- What: URL thread 生成パスについて、メタデータ前提成立・保存シーケンス・初回起動までの期待動作を検証するケースを既存テストに反映する。
- Done when: url_watch の成功系/失敗系の主要シナリオで、保存と起動のシーケンスがテストで担保される。

### 4. URL Watch の前提条件を運用知識として同期する
- Why: 「受信イベントが来ない programmatic thread では chats 登録が別途必要」という前提が共有されないと、同種の欠陥を将来の機能追加で再発するため。
- What: URL Watch スレッド自動生成フローの前提と修正意図を、関連ドキュメントまたは変更履歴に反映する。
- Done when: URL Watch の運用・改修時に参照できる形で、今回の制約と期待挙動が明文化されている。
