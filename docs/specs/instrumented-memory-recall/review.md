## MUST-FIX Findings (Cycle 3)

### [M3-1] `daemonRequestApproval` references nonexistent `decided_at` column

- **Raised by:** Reviewer A (#1), Reviewer C (#1, #2)
- **Finding:** Design §3.4 step 2 specifies dedup via `decided_at IS NULL AND expires_at > now()`. Verified: `pending_approvals` table at `src/db/migrations/module-approvals-pending-approvals.ts:23-39` has NO `decided_at` column. Real schema uses `status` (default `'pending'`). Plus the dedup logic (check-then-insert) is not atomic — two daemon sweeps or restarts can both observe no pending row and enqueue duplicate DMs.
- **Effort to Fix:** Low (column rename + atomic INSERT)
- **Impact if Ignored:** High — wrapper as written doesn't compile; idempotency goal violated under concurrency.
- **Suggested resolution:**
  - Use `status = 'pending' AND expires_at > ?` instead of `decided_at IS NULL`.
  - Either reuse `request_id` as idempotency carrier OR add a new column `idempotency_key` via the new migration.
  - Add `UNIQUE(action, idempotency_key)` constraint with `INSERT ... ON CONFLICT DO NOTHING` to make dedup atomic. Wrapper returns `deduped: true` from conflict path or transactional read.
- **Status:** Open

### [M3-2] Cross-process write race on `memory-health.json` (introduced by M2-4 fix)

- **Raised by:** Reviewer A (NEW-1, finding #2)
- **Finding:** §1.2's M2-4 resolution has the host writing `ollamaCheckHost` field to `memory-health.json`. But `HealthRecorder.flush()` at `src/memory-daemon/health.ts:240-269` atomically rewrites the entire file via tmp+rename. Concurrent writes race; daemon flushes will overwrite host writes and vice versa. Same single-writer-per-file violation that C11 polices for SQLite — applied to JSON.
- **Effort to Fix:** Low (~10 LOC)
- **Impact if Ignored:** Medium-High — operator dashboard shows inconsistent or oscillating Ollama status; new C11-equivalent violation.
- **Suggested resolution:** Host writes a separate `data/.host-ollama-status.json` flag file (host-owned, daemon read-only). Daemon reads + merges into `HealthRecorder` payload on each sweep before its atomic flush. Single-writer-per-file invariant preserved on both files.
- **Fact-check:** Verified `HealthRecorder.flush()` does atomic-tmp-rename of entire file; verified host has no equivalent merge mechanism.
- **Status:** Open

### [M3-3] `daemonRequestApproval` cannot reuse `requestApproval` because that requires Session

- **Raised by:** Reviewer A (#3, NEW-3)
- **Finding:** §3.4 says "Call existing approval queue insertion logic but suppress notifyAgent failure-fallback." But `requestApproval` (primitive.ts:188) takes `session: Session` as required field — used for delivery routing, agent notification, AND `pending_approvals.session_id`. The wrapper must either:
  - (a) Factor `requestApproval`'s body into a session-agnostic core (cleanest, ~30 LOC change to primitive.ts).
  - (b) Reimplement `pickApprover` + `pickApprovalDelivery` + the pending-row insert inline (~40 LOC duplication).
  Design says "(b) implied" but doesn't enumerate the 4-5 functions that need to be called by the wrapper.
- **Effort to Fix:** Medium (factor-out OR explicit reimplementation list)
- **Impact if Ignored:** High — implementer guesses; either path produces a half-baked daemon approval surface.
- **Suggested resolution:** Choose path (a). Add a `requestApprovalCore(opts: { session?: Session, agentGroupId: string, ... })` to primitive.ts that:
  - If session present: existing behavior (delivery via session's channel).
  - If session absent: skip notifyAgent fallback; require `agentGroupId`; resolve delivery via `pickApprover(agentGroupId)` + `pickApprovalDelivery(approver)`.
  Then `daemonRequestApproval` calls `requestApprovalCore({ agentGroupId, ... })`. Existing `requestApproval` becomes a thin wrapper that passes through `session.agent_group_id`.
- **Status:** Open

### [M3-4] Queue cap doesn't actually cap the queue (semantic gap in S2-2)

- **Raised by:** Reviewer C (#3)
- **Finding:** §1.3 per-group queue cap says "if `count_pending > 10000`, daemon stops creating new judge work." But §1.2 says the **host** is the writer of pending rows (always). And §1.7 says even with `feedback_enabled=false`, "host still writes pending rows." So under sustained load with judge backend down, pending rows grow unbounded — the daemon-side cap doesn't actually cap the queue, only stops processing.
- **Effort to Fix:** Medium (move cap check to host insertion point)
- **Impact if Ignored:** High — DB grows unbounded under judge-backend outage; partial index becomes huge; coverage metrics distorted.
- **Suggested resolution:** Two equally-grounded paths — **requires user judgment** to pick (escalate as `no-grounding`):
  - (a) **Cap at host insertion point.** Host queries `count_pending(group)` before writing; if over threshold, writes terminal row instead: `judge_method='queue-cap-skipped', judged_at=now(), judge_score=NULL`. Adds DB read on host's recall hot path.
  - (b) **Keep host writing always, but add `eligible_after` column.** Pending rows have a "this row is queued for judging" flag set to false during cap-violation periods; daemon only processes flag=true rows. Doesn't prevent unbounded growth, just moves it to a column.
  Path (a) is more honest about "cap" semantics; (b) preserves all data for post-hoc analysis.
- **Status:** Open

### [M3-5] Daily judge limit can starve old groups (cross-group fairness)

- **Raised by:** Reviewer C (#4)
- **Finding:** §1.3 daily limit + per-sweep limit don't specify global scheduling order across groups. A noisy group can consume the daily limit early every UTC day; quiet groups never get judged. Breaks the per-group isolation goal.
- **Effort to Fix:** Medium (per-group budget allocation)
- **Impact if Ignored:** Medium-High — cross-group fairness violation; quiet groups silently starved.
- **Suggested resolution:** Two equally-grounded paths — **requires user judgment**:
  - (a) **Per-group daily budgets.** Allocate `MEMORY_RECALL_JUDGE_DAILY_LIMIT / num_active_groups` per group. Simple but may waste budget if some groups are inactive.
  - (b) **Weighted round-robin.** Daemon iterates groups round-robin within each sweep; oldest-first within each group. Fairness emerges from rotation; daily limit is global.
  Path (b) is more flexible; (a) is simpler. Both are defensible.
- **Status:** Open

### [M3-6] `feedback_enabled=false` semantics contradict pending-row processing

- **Raised by:** Reviewer C (#5)
- **Finding:** §1.7 says default `feedback_enabled=false` on first release; "host still writes pending rows; daemon skips processing." But §1.3 says pending rows are the daemon's work queue. With default false:
  - Every memory-enabled group writes pending rows from day 1.
  - When operator flips `feedback_enabled=true` 30 days later, daemon suddenly has 30 days of accumulated pending rows to process — backlog spike of thousands of judge calls all at once.
  - Defeats the "before baseline before turning on" intent.
- **Effort to Fix:** Low-Medium (define the semantics)
- **Impact if Ignored:** High — backlog spike on flip; operator surprise; potential cost spike.
- **Suggested resolution:** Three equally-grounded paths — **requires user judgment**:
  - (a) **Don't write pending rows when feedback_enabled=false.** Cleanest; loses any retroactive visibility but matches operator expectation.
  - (b) **Write pending rows terminally when disabled.** `judge_method='feedback-disabled', judged_at=now()`. Preserves audit trail; flip doesn't trigger backlog.
  - (c) **Persist `feedback_enabled_at` timestamp.** Daemon only processes rows created after that timestamp. Allows historical visibility but flip cleanly.
  Path (b) is closest to "data preserved, flip is clean."
- **Status:** Open
