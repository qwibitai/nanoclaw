### MUST-FIX (cycle 2)

#### [M1-c2] chokidar v4+ removed glob support — `data/v2-sessions/*/*/outbound.db` would silently watch nothing

- **Raised by:** Reviewer A (cycle-2 finding F1)
- **Finding:** chokidar v4 (Sep 2024) and v5 explicitly removed glob support per the official README changelog. The cycle-1 design's `chokidar.watch([..., 'data/v2-sessions/*/*/outbound.db'])` would be interpreted as a literal filename, never matching anything. The SSE feed would silently produce zero events.
- **Fact-check:** `npm install chokidar@5` and grep README.md confirmed: "v4 (Sep 2024): remove glob support and bundled fsevents." Migration path documented in README via `node:fs/promises` glob.
- **Resolution applied (cycle 2):** §5 rewritten to use a directory watch on `data/v2-sessions/` with an `ignored` filter that accepts only `inbound.db` and `outbound.db` filenames, plus a separate watcher for `data/v2.db`. New session directories are detected automatically by the recursive watcher; no `.add()` plumbing required. `^5.0.0` pin retained from cycle 1.
- **Status:** Addressed in design.md §5 + §Research Summary.

#### [M2-c2] INTERCEPT_COMMANDS gate runs per-agent inside `deliverToAgent` — fan-out produces N tokens per command

- **Raised by:** Reviewer A (cycle-2 finding F2)
- **Finding:** The cycle-1 §3a placed the new `INTERCEPT_COMMANDS` gate at the existing `gateCommand` location (`src/router.ts:666`), which is inside `deliverToAgent`, called in the fan-out loop at line 466. A DM wired to N agent groups would trigger the intercept handler N times — generating N distinct dashboard tokens. The handler signature also lacked the `replyMessagingGroup` context needed to send the DM reply.
- **Fact-check:** `grep -n` on `router.ts` confirmed `gateCommand` at 666, fan-out at 466 + 507, `deliverToAgent` declaration at 601.
- **Resolution applied (cycle 2):** §3a rewritten to introduce `preFanoutGate(content, userId)` running in `routeInbound` BEFORE fan-out. Check order: FILTERED → INTERCEPT (admin auth) → fan-out → ADMIN (per-agent, existing). Handler signature now takes `InterceptContext { userId, replyMessagingGroupId, command, args }`. Added `src/router.ts (modified) +30` to §1 file inventory. ASSERT: intercept fires AT MOST ONCE per inbound message regardless of fan-out count.
- **Status:** Addressed in design.md §3a + §1.

#### [M3-c2] Steer retry after partial write throws SQLITE_CONSTRAINT_PRIMARYKEY — design's "client retry will replay-resume" was unimplementable

- **Raised by:** Reviewer A (cycle-2 finding F3) + Reviewer C (cycle-2 finding C2)
- **Finding:** `insertMessage` at `src/db/session-db.ts:166-174` is a bare `INSERT` (no `OR IGNORE`); `messages_in` has `id TEXT PRIMARY KEY`. The cycle-1 design's recovery path (retry → reserve sees pending, returns same msgId, retry calls writeSessionMessage with same id) would throw SQLITE_CONSTRAINT_PRIMARYKEY when the prior attempt's row is still there. The "replay-resume" claim doesn't hold without explicit handling.
- **Fact-check:** Read `src/db/session-db.ts:166-174` directly. Confirmed plain INSERT, no IGNORE/UPSERT. Schema confirms `id TEXT PRIMARY KEY`.
- **Resolution applied (cycle 2):** §4 step 5 augmented with a pre-write existence check (`sessionInboundHasMessage(group, session, messageId)`) — if the row already exists, skip the INSERT and proceed to apply. Also tolerates SQLITE_CONSTRAINT_PRIMARYKEY as success in case of a same-process race between two retries.
- **Status:** Addressed in design.md §4.

#### [M4-c2] SSE feed didn't watch inbound.db — operator's own steer was invisible to their dashboard until agent responded

- **Raised by:** Reviewer A (cycle-2 finding F4) + Reviewer C (cycle-2 finding C3)
- **Finding:** Cycle-1 §5's chokidar list was `[data/v2.db, data/v2-sessions/*/*/outbound.db]`. The dashboard's steer writes to `inbound.db`. SSE didn't fire when the dashboard wrote a steer. §4a's ASSERT — "MUST appear in TaskDetail within one SSE tick of being written" — was unsatisfiable. Especially broken for headless tasks (C10) where TaskDetail's inline-thread is the operator's only feedback surface.
- **Fact-check:** Direct cross-reference of §5 watch list vs §4 write target.
- **Resolution applied (cycle 2):** §5 watcher's `ignored` filter now accepts BOTH `inbound.db` and `outbound.db`. Plus, a programmatic `emitDashboardEvent('inbound_message', ...)` call is added to §4 step 5 immediately after the inbound write commits — guarantees zero-latency feedback regardless of chokidar's debounce window. Two emit paths (chokidar + programmatic) are complementary; the 200ms `awaitWriteFinish` debounce coalesces same-tick events into one client refetch.
- **Status:** Addressed in design.md §5 + §4.

## Cycle 3 — All MUST-FIX addressed inline (3-cycle cap reached)

### MUST-FIX (cycle 3)

#### [M1-c3] Cross-DB transcript merge: lexicographic sort across mixed timestamp formats reorders cross-DB rows

- **Raised by:** Reviewer A (F1)
- **Finding:** §4a sorts `merged.sort((a, b) => a.timestamp < b.timestamp ? 1 : -1)`. Host writes inbound `messages_in.timestamp` from `new Date().toISOString()` → `2026-05-10T12:34:56.789Z`. Container writes outbound `messages_out.timestamp` from SQLite `datetime('now')` → `2026-05-10 12:34:56` (no T, no millis, no Z) — verified at `container/agent-runner/src/db/messages-out.ts:61`. Lexicographic compare puts container rows BEFORE inbound rows at the same wall-clock instant (space 0x20 < T 0x54). The §4a "true temporal order" guarantee was violated for every cross-DB pair.
- **Resolution applied:** §4a sort updated to use `Date.parse(a.timestamp) - Date.parse(b.timestamp)` so both formats normalize to numeric epoch milliseconds before compare. Direction tiebreak (inbound-before-outbound) and seq tiebreak preserved.
- **Status:** Addressed in design.md §4a.

#### [M2-c3] Central DB chokidar watcher silently missed task-state writes between WAL checkpoints

- **Raised by:** Reviewer A (F2) + Reviewer C (C3 cycle-3) — reinforced
- **Finding:** §5 watched `data/v2.db` directly. Central DB is opened with `journal_mode = WAL` (verified at `src/db/connection.ts:17`; live install has `data/v2.db` 1.1MB + `data/v2.db-wal` 3.9MB). In WAL mode, writes accumulate in the `-wal` sidecar; the main file's mtime doesn't change until checkpoint (default ~1000 pages). chokidar+inotify on the main file misses every task admit / status / progress write between checkpoints — could be hours on a low-write central DB.
- **Resolution applied:** §5 dropped the `centralDbWatcher` entirely. Replaced with programmatic `emitDashboardEvent('task_event', ...)` calls inserted at each of the 6 `tasks`-mutation callsites (`applySpawnTask`, `applySpawnComplete`, `applySpawnFailed`, `applySpawnCancel`, `applySpawnProgress`, watchdog reaper). §1 file inventory adds `src/modules/orchestrator-dispatch/dispatch.ts (modified) +12`. The dashboard event-bus filters by per-user scope before fanout. This also closes the macOS FSEvents/SQLite-WAL flakiness that cycle-2 W2-c2 had carried as a deferred concern — programmatic emit avoids filesystem observation entirely.
- **Status:** Addressed in design.md §5 + §1.

#### [M3-c3] Routing primitive lacked splat/prefix support — `/dashboard/static/*` couldn't dispatch

- **Raised by:** Reviewer C (C1 cycle-3)
- **Finding:** §2 pattern syntax was `:id` params only. Vite emits asset URLs like `/dashboard/static/assets/app-Bq7XYZ.js`. The router's `pathMatch` would never match these; the SPA would build but fail at runtime with missing JS/CSS.
- **Resolution applied:** §2 pattern syntax extended with terminal splat (`*tail` captures everything after the prefix). `pathMatch` updated to handle the splat segment. Static path becomes `/dashboard/static/*tail` — `tail` is the path arg passed to the static handler (which already does the `path.resolve` containment check from cycle-1 fix M36).
- **Status:** Addressed in design.md §2.

#### [M4-c3] SSE handler couldn't `req.on('close')` because `Handler` signature only exposed Web `Request`

- **Raised by:** Reviewer C (C2 cycle-3)
- **Finding:** §2 handler signature was `(req: Request, params, ctx) => Promise<Response>`. Web `Request` has no `.on('close')` method. §5's SSE cleanup logic (`req.on('close', () => { conns.delete(conn); })`) was unimplementable as written — disconnections would only be detected when the next keepalive write threw, by which point connections leak indefinitely.
- **Resolution applied:** §2 `RequestContext` now includes `rawNodeReq: http.IncomingMessage`. `Handler` return type formalized as `Promise<Response | null>`. `dispatch` signature takes both `nodeReq` and `nodeRes`. SSE handler subscribes to `ctx.rawNodeReq.on('close', ...)` for prompt cleanup.
- **Status:** Addressed in design.md §2.
