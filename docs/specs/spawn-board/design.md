# Design: Spawn Board — Multi-Task Steer Dashboard for Self-Orchestration

> Generated by `/team-design` — 2026-05-10
> Saved to: `docs/specs/spawn-board/design.md`

## Brief Reference

Approved brief: `docs/specs/spawn-board/brief.md`. Phase 2 of the orchestrator-spawn feature line; sibling to `docs/specs/orchestrator-dispatch/` (Phase 1, shipped at merge `8fe1d86`). The board is a single-page operator UI mounted on the existing Node host that reads `tasks` + per-session inbound/outbound DBs to render a kanban view (lanes by status), a task-detail view (lifecycle + last-200-message inline thread), and a session-list view (live sessions across the viewer's auth scope). Operator can post a steering chat into any child session from the task-detail view; for `native_thread` children the steer is echoed `[via dashboard] <text> — <operator>` to the platform thread via the channel adapter. Auth is owner + scoped admins via existing `canAccessAgentGroup`. Live updates via SSE driven by chokidar. 12 HARD/SOFT constraints + 12 rejected alternatives in `decisions.yaml`.

### Flagged Constraints

- **C5 (memory budget) is provisional — no pre-build benchmark exists.** The 150MB/60min/5-tasks number is from Phase 1 design §7 and is unmeasured for the dashboard specifically. Realistic incremental cost breakdown at full load (5 active tasks, 20 SSE connections):
  - chokidar with full glob scope: ~5-10MB
  - SSE connection state: ~5KB × 20 = ~100KB
  - Routing primitive code: ~1MB
  - Per-request better-sqlite3 connection lifecycle: 0 incremental at idle (opened/closed per request)
  - Static asset filesystem-stream serving: 0 (no in-memory cache)
  - Token + idempotency table state: trivial; SQLite memory is on-demand
  - Headroom for V8 GC variance + unexpected growth: comfortable

  Estimated incremental at load: ~7-15MB. 150MB ceiling provides 10× headroom — generous on purpose. Reconciliation gate at `/team-ship`: spawn 5 tasks, open 5 dashboard tabs (5 SSE connections), hold 60min, snapshot host RSS via `ps -o rss= -p $NODE_HOST_PID` every 5min, fail ship if median post-warm RSS exceeds the no-dashboard baseline by >150MB. The budget is loose enough that hitting it implies a real bug (memory leak, runaway listener, in-process cache that shouldn't exist), not just incidental growth — the gate's job is to catch regressions, not to enforce parsimony for its own sake.

No other constraints flagged. Brief's HARD/SOFT classifications hold up under scrutiny.

## Research Summary

### Project patterns

- **CLAUDE.md (host module system):** "Host (`src/`) is ESM. Never use `require()`." Dashboard handlers use ESM imports.
- **CLAUDE.md (two-DB session split):** "Each session has two SQLite files... `inbound.db` — host writes, container reads. `outbound.db` — container writes, host reads." Echo write to inbound.db is host-side; reading outbound.db for TaskDetail history is host-side read-only.
- **CLAUDE.md (host-side ad-hoc queries):** "Use the in-tree wrapper rather than the `sqlite3` CLI: `pnpm exec tsx scripts/q.ts <db> '<sql>'`." Not used by dashboard at runtime — `better-sqlite3` is the production access path. Cited only because the dashboard's design avoids spawning external CLIs.
- **CLAUDE.md (supply chain):** "`minimumReleaseAge: 4320` (3 days) preserved; no `minimumReleaseAgeExclude` additions without operator approval." All new dashboard deps must be ≥3 days old at install time.
- **Phase 1 design §7 (referenced explicitly):** establishes the routing layer scope (~200 LOC), the DM-token auth flow, the chokidar-debounce-200ms feed, and the memory budget. The Phase 2 design here is consistent with that scaffold with one substantive deviation: Phase 1 §7 was read-only; Phase 2 adds the steer write path, which requires a new write-auth check, the echo-to-adapter call, and rate-limit + idempotency machinery.

### Library docs

- **chokidar 5.0.0** (released 2025-11-25 per `npm view chokidar@5.0.0 time` — past the 3-day `minimumReleaseAge: 4320` window): ESM-only (v4 was dual ESM/CJS; v5 dropped CJS), Node 20+, shutdown via `.close(): Promise<void>`, `awaitWriteFinish` debounce option preserved from v4. **Globs are NOT supported** in v4+ (removed Sep 2024 per the official README changelog) — the watch list takes literal paths only. Pin to `^5.0.0`. (Cycle-1 fix M1: pin corrected from ^4.0.3. Cycle-2 fix F12: AbortController wording removed; close-API documented correctly.)
- **react / react-dom 19.2.6**: latest stable. v19 introduces `use()` hook and Server Components, neither of which apply to this SPA. Pin to `19.x` (any patch) — we ship a client-only Vite SPA, no SSR.
- **vite 8.0.11**: latest stable. New build defaults are aligned with our use case (ESM, code-splitting).
- **swr 2.4.1**: latest stable. Used for `GET /dashboard/api/tasks` + detail; pairs with EventSource for invalidation on SSE events.
- **@vitejs/plugin-react 6.0.1**: required for Vite + React 19. Pin to ^6.0.0.
- **better-sqlite3** is the canonical SQLite client used host-side; already a dependency. Dashboard reuses `getDb()` and `openOutboundDb()` from `src/db/`.
- **Native Node http.ServerResponse for SSE:** No library required. Set `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`. Write `event: <type>\ndata: <json>\n\n` payloads. Periodic `:keepalive\n\n` comments every 25s. **Critical:** must bypass `fromWebResponse` (`webhook-server.ts:52-67`) — that helper reads the Web Response body to completion before calling `nodeRes.end()`, which is fatal for an infinite SSE stream. The dashboard SSE handler writes directly to the Node `ServerResponse`, never going through the Web-Response adapter.

### Existing code

- `src/webhook-server.ts:88-128` — current ensureServer matches only `/webhook/{adapter}` URLs and 404s everything else. The refactor introduces a multi-pattern router.
- `src/webhook-server.ts:52-67` — `fromWebResponse` reads `webRes.body` to completion. Confirmed: incompatible with SSE.
- `src/modules/permissions/access.ts:21-28` — `canAccessAgentGroup(userId, agentGroupId)` returns `AccessDecision` with reasons including `owner | global_admin | admin_of_group | member | not_member | unknown_user`. Dashboard auth treats `owner | global_admin | admin_of_group` as authorized for steer; `member`-only is read access.
- `src/channels/adapter.ts:149` — `deliver(platformId, threadId, message)` posts INTO an existing thread (with `threadId` non-null). This is the right echo target — not `postParent` (which posts at root, no thread). The dashboard echo calls `adapter.deliver(child_platform_id, child_platform_thread_id, {kind: 'chat', content: {text: '[via dashboard] ... — Dave'}})`.
- `src/db/session-db.ts:128-167` — `nextEvenSeq()` and `insertMessage()` are host-side primitives that already enforce the even-seq invariant. The steer write reuses `writeSessionMessage()` from `src/session-manager.ts`, which already calls these.
- `src/modules/permissions/db/users.ts` — `getUser(userId)` returns the `users` row; gives us `display_name` for the echo suffix.

## Options

The architecturally consequential decision is **how the new dashboard routes coexist with the existing webhook adapter routes inside `webhook-server.ts`**. Three distinct strategies; sub-decisions (SSE feed implementation, SPA build, steer write path) are nearly identical across them.

### Option A: Inline expansion of webhook-server.ts

**Approach:** Keep `webhook-server.ts` as the single HTTP entry point, but expand the existing `ensureServer()` request handler with a chain of `if (url.startsWith(...))` branches: `/webhook/...` → existing logic; `/dashboard/static/...` → static file handler; `/dashboard/api/...` → dashboard API handler module; `/dashboard` → SPA index.html. Auth middleware is a function called inline at the top of every dashboard branch.

**Pros:**
- Minimal structural change to one file. ~150 LOC of conditional dispatch added to the existing 143 LOC.
- No new module boundaries to maintain. Refactor is contained.
- Easiest to reason about for a small surface area.

**Cons:**
- `webhook-server.ts` becomes ~300 LOC with three concerns mixed: webhook adapter dispatch, dashboard API, static asset serving. Cohesion suffers.
- Auth check inlined per-branch — easy to miss on a new endpoint. No central enforcement point.
- Future routes (Phase 2B "steer to spawn", Phase 3 owner-cancel) keep growing the same file.

**Pattern alignment:** Doesn't match any existing project pattern — webhook-server is the only HTTP entry currently. Closest analogue is `src/router.ts` (inbound message router with conditional branches), which is itself ~700 LOC and frequently flagged as a maintenance pain point.

**Confidence:** High — mechanically simple; the Cons are real but bounded.

**Assumptions:**
- The total dashboard route count stays small (<12 distinct paths). Validated against the brief's 8 routes; one path of growth is realistic.

---

### Option B: Internal route-table primitive in webhook-server.ts

**Approach:** Introduce a small route table (~30 LOC) inside `webhook-server.ts`: array of `{ method, pathPattern, handler }` entries with first-match dispatch. Webhook adapter routes are converted to use the table (`POST /webhook/:name`); dashboard routes register their own entries. Auth becomes a typed `requireAuth(handler)` higher-order function applied at registration time. Handlers themselves live in dedicated files (`src/dashboard/api/tasks.ts`, etc.) imported by the registration site.

**Pros:**
- Auth enforcement is structural — `requireAuth` wraps every dashboard handler at registration; the handler signature won't compile if you forget. Hard to add an unauthenticated dashboard endpoint by accident.
- Webhook + dashboard routes share the same dispatch mechanism, eliminating the "two parallel code paths" smell.
- Handler files are small and testable in isolation. Adding Phase 2B steer-into-spawn becomes "add a file, register a route" not "modify the central server."
- Pattern transfers to other infra: scheduled HTTP triggers, future webhook receivers, future API routes — all use the same primitive.

**Cons:**
- New abstraction (~80 LOC of routing primitive: pattern-match, param extraction, method dispatch, requireAuth wrapper).
- One more module boundary to maintain. Refactor touches more files than Option A (~12 vs ~3).
- Slight conceptual overhead for someone reading webhook-server.ts cold — they have to understand the table-driven dispatch before tracing a request.

**Pattern alignment:** Matches the agentic-systems convention of "registry primitives that owns enforcement, handlers register themselves" — same shape as `registerDeliveryAction` (`src/delivery.ts`), `registerChannelAdapter` (`src/channels/channel-registry.ts`), `registerTools` (`container/agent-runner/src/mcp-tools/server.ts`). Multiple precedents in the codebase for "small registry, distributed handlers."

**Confidence:** High — registry pattern is well-established here; Express-shaped routing is well-understood; both pieces compose without surprises.

**Assumptions:**
- Path-pattern matching can stay simple (no nested params, no query-string routing, no content-negotiation). Validated against the brief's 8 routes.
- The webhook adapter migration to the new table doesn't break the adapters themselves. Validated by reading `webhook-server.ts:81-86` — `registerWebhookAdapter` already encapsulates registration; we just point it at the new table.

---

### Option C: Split webhook-server into a thin shell + two route modules

**Approach:** Keep `webhook-server.ts` as a thin Node `http.createServer` shell that delegates to `webhook-routes.ts` (existing adapter dispatch) and `dashboard-routes.ts` (new). Each module exports a `match(req): handler | null` function; the shell tries them in order. `dashboard-routes.ts` further splits into `auth.ts`, `api.ts`, `static.ts`, `sse.ts`.

**Pros:**
- Clearest separation: webhook concerns and dashboard concerns never share a file.
- Each module is small and easy to test.

**Cons:**
- Two routing systems coexist (one per module). The "two parallel paths" smell that Option B eliminates is exactly what this option entrenches.
- Auth has to be reimplemented or re-imported in each module (or extracted to a shared helper anyway, at which point Option B's primitive is doing the same job with less ceremony).
- Module boundaries proliferate without clear lifecycle ownership. Who closes the chokidar watcher on shutdown? Who tears down the SSE connections? `webhook-server.ts` becomes a coordinator without authority.
- Largest refactor footprint — ~5 new files for what is fundamentally a routing decision.

**Pattern alignment:** Doesn't match any existing project pattern. The closest is the `channels/` directory (one file per adapter), but those are independent-purpose modules, not co-equal routing concerns.

**Confidence:** Medium — the split feels clean in the abstract but creates more surface than the actual problem warrants; risks introducing cohesion debt elsewhere.

**Assumptions:**
- Future Phase 2B/Phase 3 routes don't blur the webhook/dashboard split. **High risk** — owner-cancel from the dashboard requires posting outbound message back, which crosses concern boundaries.

---

## Recommendation

**Selected:** Option B — internal route-table primitive in webhook-server.ts.

**Justification:**
- **Auth structural enforcement.** The single biggest risk for a write-capable dashboard is forgetting to authenticate a route. Option B's `requireAuth(handler)` makes that a compile-time question; Options A and C require per-route discipline, which decays as the surface grows.
- **Pattern alignment with existing registries.** `registerDeliveryAction`, `registerChannelAdapter`, `registerTools` — every meaningful host primitive in this codebase is a tiny registry with distributed registration. Option B is "do that for HTTP routes." Option C invents new module boundaries the codebase doesn't otherwise use; Option A foregoes the abstraction entirely and pays for it as the file grows.
- **Right-sized scope.** Phase 2 needs 8 routes; Phase 2B will add 1-3 (steer-related extensions); Phase 3 might add 1-2 (owner-cancel). That's 12-15 routes in the steady state. Option A is fine for 8, painful at 15. Option B handles both gracefully and is the same effort either way. Option C is over-engineered for any of these scales.
- **Refactor risk is bounded.** Option B's ~80-LOC primitive is contained and unit-testable; the webhook adapter migration is a structural rewrite of `ensureServer()` that we get to verify against the existing test surface. Option C's multi-file split has more coupling exposure across module boundaries.

**Confidence note:** High. No spike needed. The route-table primitive pattern is well-trodden (Express, Hono, raw Node — all use variations of this); the registry-with-distributed-registration pattern is the project's house style; the refactor scope is small and bounded.

### 2. Routing primitive (`src/dashboard/router.ts`)

```ts
type Method = 'GET' | 'POST' | 'PUT' | 'DELETE';
type Handler = (req: Request, params: Record<string, string>, ctx: RequestContext) => Promise<Response | null>;
type AuthHandler = (req: Request, params: Record<string, string>, ctx: AuthedRequestContext) => Promise<Response | null>;

// Cycle-3 fix M4-c3: include the raw Node IncomingMessage so SSE handlers can
// register `req.on('close')` for client-disconnect cleanup. The Web `Request` shape
// has no `.on('close')`, so without raw access SSE connections would leak when
// the client tab closed (until the keepalive write next failed).
interface RequestContext {
  rawNodeReq: http.IncomingMessage;
  rawNodeRes?: http.ServerResponse;   // for SSE — handler writes directly
}
interface AuthedRequestContext extends RequestContext { user: User; scopes: GroupScope[] }

interface RouteEntry { method: Method; pattern: string; handler: Handler; }

const routes: RouteEntry[] = [];

export function register(method: Method, pattern: string, handler: Handler): void { ... }
export function requireAuth(handler: AuthHandler): Handler { ... }   // wraps with cookie verify
// dispatch returns Promise<Response | null>:
//   Response → caller passes to fromWebResponse to populate nodeRes
//   null     → handler wrote directly to ctx.rawNodeRes; do not call fromWebResponse
export function dispatch(
  req: Request,
  nodeReq: http.IncomingMessage,
  nodeRes: http.ServerResponse,
): Promise<Response | null> { ... }
```

Pattern syntax: `/dashboard/api/tasks/:id` → params `{ id }`. **`/dashboard/static/*tail`** (cycle-3 fix M3-c3) → params `{ tail: 'assets/app.js' }` for Vite-emitted asset paths and any nested static asset. The single trailing splat captures everything after the prefix; one splat per pattern, terminal position only. No regex, no query-routing. First-match wins.

```ts
function pathMatch(pattern: string, urlPath: string): Record<string, string> | null {
  const pSegs = pattern.split('/');
  const uSegs = urlPath.split('/');
  const params: Record<string, string> = {};
  for (let i = 0; i < pSegs.length; i++) {
    const p = pSegs[i];
    if (p.startsWith('*')) {
      // Splat: must be the last segment in the pattern; captures everything after.
      params[p.slice(1)] = uSegs.slice(i).join('/');
      return params;
    }
    if (i >= uSegs.length) return null;
    if (p.startsWith(':')) { params[p.slice(1)] = uSegs[i]; continue; }
    if (p !== uSegs[i]) return null;
  }
  return uSegs.length === pSegs.length ? params : null;
}
```

`requireAuth` reads the cookie, looks up the user, populates `ctx.user` and `ctx.scopes`, and calls the handler. On missing/invalid cookie returns `401 { error: 'unauthenticated' }`. `ctx.scopes` is the array of `{ agent_group_id, role }` rows from `user_roles` plus implicit `owner` / `global_admin` scopes.

**Handler-vs-dispatch contract (cycle-1 fix S10):** `register`'s `Handler` type returns `Promise<Response | null>`. Returning a `Response` is the normal path — `dispatch` passes it to `fromWebResponse` to populate `nodeRes`. Returning `null` signals "I wrote directly to `ctx.rawNodeRes`; do not call `fromWebResponse`." Only the SSE handler (and any future raw-streaming handler) returns `null`. `dispatch` interprets `null` as "handled raw."

**CSRF Origin check (cycle-1 fix S9):** For methods `POST | PUT | DELETE`, `requireAuth` checks the request `Origin` header. If `Origin` is set and does not match the request `Host`, returns `403 { error: 'origin_mismatch' }`. Localhost origins (`localhost`, `127.0.0.1`, `::1`) are accepted regardless. Defense-in-depth on top of `SameSite=Strict`.

### 2a. Resource-level authorization (cycle-1 fix M29)

`requireAuth` proves the user is authenticated. Authorization for resource access happens at the query level — every dashboard handler that reads tasks / sessions / messages MUST inject a server-side scope predicate:

| Endpoint | Owner / global_admin | Scoped admin (admin_of_group) | Member |
|----------|---------------------|------------------------------|--------|
| `GET /dashboard/api/tasks` | no filter | `WHERE parent_agent_group_id IN (ctx.scopes.allowed_group_ids)` | (empty result; `member` cannot read or steer) |
| `GET /dashboard/api/tasks/:id` | direct lookup | `WHERE task_id = ? AND parent_agent_group_id IN (allowed)` (returns 404 on out-of-scope) | 403 |
| `GET /dashboard/api/sessions` | no filter | `WHERE agent_group_id IN (allowed)` | empty |
| `POST /dashboard/api/tasks/:id/message` | steer any | steer if `task.parent_agent_group_id IN (allowed)` | 403 (`member_role_cannot_steer`) |

### 3. Auth flow (cycle-1 fix M4 + M33 + M34)

**Token issuance** is via the existing chat surface, host-side intercept (NOT a dashboard endpoint, NOT container-side).

1. Operator DMs any agent with `/dashboard-token`.
2. Host-side `command-gate` (see §3a below) intercepts the message via the new `INTERCEPT_COMMANDS` map and routes to the handler `dashboard_token_issue`. The container never sees the message.
3. Handler `applyDashboardTokenIssue`:
   - Generates a 24h-TTL hex token: `token_raw = crypto.randomBytes(32).toString('hex')` (256 bits).
   - Computes `token_hmac = HMAC-SHA256(token_raw, server_key)` where `server_key` is read from `NANOCLAW_DASHBOARD_COOKIE_SECRET`. The same key signs cookies; one rotation invalidates everything together (acceptable scope per W3 / W9).
   - INSERTs into `dashboard_tokens (user_id, token_hmac, issued_at, expires_at, used_at=NULL)`. **The raw token is never persisted** (cycle-1 fix M34).
   - Writes a chat reply to the user's DM with the raw token + dashboard URL: `Open https://<host>:3000/dashboard/ and paste this token (single-use, 24h TTL): <token_raw>`.
   - Sample DM payload uses the existing outbound delivery system (same path adapters use to reply to users).
4. Operator opens `https://<host>:3000/dashboard/` (or `http://localhost:3000/dashboard/` in localhost-default deploys). The dashboard's AuthGate component shows a single text input and a Submit button. **The URL `?token=` query-string variant is removed** (cycle-1 fix M33) — tokens never appear in URLs (browser history, server access logs, reverse-proxy logs).
5. `POST /dashboard/api/auth/exchange { token }` (token in JSON body, not URL):
   - Server computes `HMAC-SHA256(token, server_key)` and looks up `dashboard_tokens WHERE token_hmac = ? AND used_at IS NULL AND expires_at > now`.
   - On match: marks `used_at = now` atomically, sets cookie:

     ```
     Set-Cookie: spawn_board=<signed payload>; HttpOnly; Secure; SameSite=Strict; Max-Age=43200; Path=/dashboard
     ```

     Cookie payload: `{ user_id, expires_at: now + 12h }`, signed with HMAC-SHA256 using the same `server_key`.

   **Server key resolution (cycle-2 fix F9):** at host bootstrap, the server key is resolved by precedence:
     1. If `process.env.NANOCLAW_DASHBOARD_COOKIE_SECRET` is set and ≥64 hex chars (256 bits), use it as the value.
     2. Else if `~/.nanoclaw/cookie-secret` exists, read it (mode-0600 file containing 64 hex chars).
     3. Else generate `crypto.randomBytes(32).toString('hex')` (256 bits), write to `~/.nanoclaw/cookie-secret` with mode 0600, and use the value.
     The resolved key is held in process memory only — never re-exported to `process.env` (avoids `ps`-listing exposure on Linux). Rotating the key invalidates all outstanding cookies AND tokens at once; that's intentional (W3, W9). The `~/.nanoclaw/` location is preserved by `migrate-v2.sh`'s state-preservation list — see W3.
   - On no match (token not found, already used, expired): returns `400 { error: 'invalid_token' }` with no detail to prevent enumeration.
6. All subsequent `/dashboard/api/*` requests verify the cookie via `requireAuth`. `GET /dashboard/api/auth/me` is the authenticated probe — returns `{ user_id, scopes }` so the UI knows what to render.

**Token reuse safety:** tokens are single-use; after exchange, `used_at` is non-null and a re-exchange returns 400. Compromised cookies are bounded by the 12h TTL; rotating `NANOCLAW_DASHBOARD_COOKIE_SECRET` invalidates all outstanding cookies and unused tokens at once. `/dashboard-revoke` is deferred to Phase 2B (W9).

**TLS / `Secure` cookie note (cycle-1 partial fix W12):** Browsers permit `Secure` cookies on `http://localhost`, `http://127.0.0.1`, and `http://[::1]` — these are "potentially trustworthy" origins per W3C secure-contexts spec. So localhost-default NanoClaw deploys work without TLS. Non-localhost deploys (e.g. accessing the dashboard from a different host on the LAN) require TLS termination either in the Node host or upstream of it. This is a deployment concern, not a design concern — `docs/setup-wiring.md` documents the localhost-vs-LAN distinction.

### 3a. Host-side slash-command intercept primitive (cycle-1 fix M4 + cycle-2 fix M2)

`/dashboard-token` requires a new third command class beyond `FILTERED_COMMANDS` (silent drop) and `ADMIN_COMMANDS` (forwarded to container). The new class is `INTERCEPT_COMMANDS`: messages are NOT forwarded to the container; instead, a host-side handler runs and emits an outbound DM back to the user.

**Gate placement (cycle-2 fix M2 — critical):** the existing `gateCommand` invocation lives at `src/router.ts:666` inside `deliverToAgent`, which is called in the fan-out loop at `src/router.ts:466`. Running `INTERCEPT_COMMANDS` at that location would issue N tokens per command if the user's DM is wired to N agent groups. The intercept gate MUST run BEFORE fan-out.

Gate hierarchy (cycle-2 specification):

| Stage | Where | Classes handled | Action |
|-------|-------|----------------|--------|
| Pre-fanout (NEW) | `routeInbound`, before line ~440 fan-out | `FILTERED`, `INTERCEPT` | filter → drop entirely; intercept → run handler once, do not enter fan-out |
| Per-agent (existing) | `deliverToAgent`, line 666 | `ADMIN`, default pass | admin → auth-check then forward; pass → forward |

```ts
// src/command-gate.ts (cycle-2 additions)

type InterceptHandler = (ctx: InterceptContext) => Promise<void>;

interface InterceptContext {
  userId: string;          // already resolved by senderResolver upstream of the gate
  // Where to reply: the messaging_group the user DMed (NOT a per-agent reply).
  // Already resolved at router.ts:297-334 as `mg`; pass `mg.id` here.
  replyMessagingGroupId: string;
  command: string;         // e.g. '/dashboard-token'
  args: string;            // text after the command
}

const INTERCEPT_COMMANDS = new Map<string, {
  handlerName: string;
  requiresAuth: 'admin' | 'any';
}>([
  ['/dashboard-token', { handlerName: 'dashboard_token_issue', requiresAuth: 'admin' }],
  // future host-side intercepts (e.g. /dashboard-revoke) register here
]);

const handlers = new Map<string, InterceptHandler>();

export function registerInterceptHandler(name: string, h: InterceptHandler): void {
  handlers.set(name, h);
}

export function preFanoutGate(content: string, userId: string): GateResult {
  let text: string;
  try { text = (JSON.parse(content).text || '').trim(); } catch { text = content.trim(); }
  if (!text.startsWith('/')) return { action: 'pass' };
  const command = text.split(/\s/)[0].toLowerCase();
  if (FILTERED_COMMANDS.has(command)) return { action: 'filter' };
  const intercept = INTERCEPT_COMMANDS.get(command);
  if (intercept) {
    if (intercept.requiresAuth === 'admin' && !isAnyAdmin(userId)) {
      // Cycle-3 fix S4-c3: `isAnyAdmin` covers owner / global_admin / scoped admin.
      // The earlier draft said `isAdminGlobal` (or `isGlobalAdmin`), which is too narrow
      // and would silently lock out scoped admins from issuing dashboard tokens.
      return { action: 'deny', command };
    }
    return { action: 'intercept', handlerName: intercept.handlerName, command, args: text.slice(command.length).trim() };
  }
  return { action: 'pass' };
}

// gateCommand (existing per-agent gate, unchanged) handles ADMIN_COMMANDS only at fan-out time.

export type GateResult =
  | { action: 'pass' }
  | { action: 'filter' }
  | { action: 'deny'; command: string }
  | { action: 'intercept'; handlerName: string; command: string; args: string };  // NEW
```

**Router integration (`src/router.ts` modification):** before the fan-out loop, call `preFanoutGate(content, userId)`. If `action === 'intercept'`, resolve the user's reply messaging-group, build `InterceptContext`, look up the handler, await it (with a 5s timeout to bound router latency), then RETURN from `routeInbound` — the message NEVER enters fan-out, no agent receives it. If `action === 'filter'` or `'deny'`, follow existing semantics. Else continue to the fan-out loop unchanged.

**Auth model:** `requiresAuth: 'admin'` checks for ANY admin role (owner / global_admin / any agent_group_id admin) — not scoped per-group, because the intercept runs before we know which agent group the user is even targeting. `/dashboard-token` is admin-only because dashboard auth carries write access (steer); allowing non-admins to issue tokens would let them sign in and then 403 on every write. Better to deny issuance.

**Reply path:** the handler writes its DM reply to the user's resolved messaging_group via the standard outbound delivery system (`writeOutboundForMessagingGroup` or equivalent — same path Slack/Discord adapters use to reply). For the host-side `/dashboard-token` flow, this means the message comes from "the host" rather than a specific agent — operator sees it as a system DM, which matches the chat-side bot conventions.

**ASSERT:** `INTERCEPT_COMMANDS` handlers MUST NOT block the router for >5 seconds. They emit and return; outbound delivery is asynchronous through the existing `messages_out` queue. A handler exceeding 5s is killed (timeout); router emits a warning log.

**ASSERT (cycle-2 fix M2):** for any inbound message resolving to multiple wired agents, the intercept gate fires AT MOST ONCE per inbound message. Reviewers verify by triggering `/dashboard-token` in a DM wired to ≥2 agent groups and confirming exactly 1 row in `dashboard_tokens` results.

**Effort estimate:** ~50 LOC in `command-gate.ts` (the map + new `preFanoutGate` function) + ~60 LOC in `dashboard/auth/dashboard-token-issue.ts` (the handler) + ~30 LOC in `router.ts` (pre-fanout gate dispatch + reply-mg resolution). Total: ~140 LOC across 3 files.

### 4. Steer write path (`src/dashboard/steer.ts`) — revised cycle 1 (M3 + S6 + S12)

Endpoint: `POST /dashboard/api/tasks/:id/message`.

Body: `{ idempotency_key: UUIDv4, text: string }`.

**Critical correctness change in cycle 1:** the original draft recorded idempotency AFTER side-effects, which broke the safety property under host crash (M3 / Reviewer A #3 / Reviewer C #3). The revised flow reserves the idempotency row in a transaction BEFORE the inbound write, then advances to `applied` after the write commits. A retry with the same key while the row is `pending` returns the original `message_id` (not a fresh write). A retry after `applied` returns the cached response.

**Step ordering change in cycle 1:** echo runs AFTER wakeContainer (was before). Slow Slack adapter no longer blocks the 202 (S12).

```ts
async function applySteer(taskId, text, idempotencyKey, ctx) {
  // 1. Validate
  if (!text.trim()) throw 400('empty_message');
  if (text.length > 4000) throw 400('message_too_long');

  // 2. Authorize via §2a — task must be in viewer's allowed groups + role allows steer
  const task = getTaskById(taskId);
  if (!task) throw 404('task_not_found');
  if (!ctx.scopes.no_filter
      && !ctx.scopes.allowed_group_ids.includes(task.parent_agent_group_id)) {
    throw 404('task_not_found');           // disclose-as-not-found per §2a
  }
  if (ctx.scopes.role === 'member') throw 403('member_role_cannot_steer');
  if (!task.child_session_id) throw 409('task_has_no_child_session');

  // 3. Rate-limit (fixed-window: per-(user_id, child_session_id), 30/min)
  //    In-memory Map<key, {count, windowStart}>. Bucket clears on host restart (S3).
  if (!consumeRateBudget(ctx.user.id, task.child_session_id)) {
    throw 429({ retry_after: rateLimitRetryAfter(ctx.user.id, task.child_session_id) });
  }

  // 4. Reserve idempotency row in a transaction with deterministic message_id (M3).
  //    Cycle-2 fix F8: reservation binds (user_id, idempotency_key, task_id) — a key
  //    reused across different task_ids returns 422 (mismatched payload) per IETF
  //    idempotency-key draft §5 guidance.
  //    Cycle-3 fix S5-c3: reservation also stores sha256(normalized_text) — a key
  //    replayed with different text returns 422 too (per same IETF draft §2.7).
  //    Recovery from `pending` uses the stored text, NOT the retry body.
  const messageId = randomUUID();
  const requestHash = sha256(text.trim());
  let reserved: { messageId: string; status: 'pending' | 'applied'; cached?: SteerResponse;
                  storedText?: string; echoAttempted: boolean; idempotencyRowId: number };
  try {
    reserved = reserveIdempotency(ctx.user.id, idempotencyKey, taskId, messageId, text, requestHash);
    // reserveIdempotency throws 422 if the existing row binds a DIFFERENT task_id OR
    // a DIFFERENT request_hash. Recovery uses reserved.storedText for the actual write.
  } catch (err) {
    refundRateBudget(ctx.user.id, task.child_session_id);
    throw err;
  }
  // Cycle-3 fix S5-c3: pending recovery uses the stored text, not the retry body.
  // This prevents the case where a retry with mutated body produces a different
  // inbound write than the original.
  const writeText = reserved.storedText ?? text;
  if (reserved.status === 'applied' && reserved.cached) {
    refundRateBudget(ctx.user.id, task.child_session_id);  // didn't actually consume
    return 202(reserved.cached);                            // replay
  }
  // Either fresh reservation (status=pending) OR an in-flight retry of an earlier crash.

  // 5. Write to child's inbound.db.
  //    Cycle-2 fix M3: this is the partial-write recovery path. If status=pending and
  //    messages_in already has a row with id=reserved.messageId, the inbound write from
  //    a prior attempt landed before the crash. Skip the write and proceed to apply.
  //    Otherwise issue the INSERT.
  const inboundExists = await sessionInboundHasMessage(
    task.parent_agent_group_id,
    task.child_session_id,
    reserved.messageId,
  );
  if (!inboundExists) {
    try {
      await writeSessionMessage(task.parent_agent_group_id, task.child_session_id, {
        id: reserved.messageId,
        kind: 'chat',
        timestamp: new Date().toISOString(),
        content: JSON.stringify({
          text: writeText,                    // cycle-3 fix S5-c3: use reserved/stored text
          sender: ctx.user.display_name ?? ctx.user.id,
          senderId: ctx.user.id,
          _via: 'dashboard',                  // metadata for logs, not for agent prompt
        }),
      });
    } catch (err) {
      if (isSqliteBusy(err)) {
        // S6: refund rate-limit, leave idempotency 'pending' so retry can resume cleanly.
        refundRateBudget(ctx.user.id, task.child_session_id);
        throw 503({ retry_after: 2 });
      }
      if (isSqliteConstraintPrimaryKey(err)) {
        // Race: a concurrent retry reserved the same key, both passed the existence
        // check, both raced to insert. Treat as success — the message DID land.
        log.debug('steer write hit primary-key race, accepting as idempotent', {
          taskId, messageId: reserved.messageId,
        });
      } else {
        // Other write error: leave idempotency 'pending'; client retry will replay-resume.
        refundRateBudget(ctx.user.id, task.child_session_id);
        throw err;
      }
    }
  }
  // M4 (cycle 2): emit programmatic SSE invalidation immediately so operator's view
  // updates without waiting for chokidar's 200ms debounce.
  emitDashboardEvent('inbound_message', {
    task_id: task.task_id,
    child_session_id: task.child_session_id,
    parent_agent_group_id: task.parent_agent_group_id,
    message_id: reserved.messageId,
  });

  // 6. Wake container so child sees the message on next turn (was step 7).
  // Cycle-3 fix S3-c3: wakeContainer takes a full Session, not a partial. Resolve
  // via getSession(id) — same pattern as src/modules/approvals/primitive.ts:139-142.
  const childSession = getSession(task.child_session_id);
  if (childSession) await wakeContainer(childSession);
  // (If childSession is missing, the message is already in inbound.db; container
  // will pick it up on its next start. No 409 — the steer landed.)

  // 7. Echo to Slack/Discord IF native_thread + adapter capability — fire-and-forget,
  //    NOT awaited (S12, C3). Echo status is reported via SSE as the next task event.
  //    Cycle-3 fix S6-c3: echo fires ONLY on a fresh inbound write OR a pending-row
  //    recovery whose echo_attempted is false. After echo fires, set echo_attempted=1
  //    on the idempotency row so concurrent retries don't double-echo.
  if (!inboundExists) {
    fireEchoAsync(task, text, ctx, reserved.idempotencyRowId);
  } else if (!reserved.echoAttempted) {
    fireEchoAsync(task, text, ctx, reserved.idempotencyRowId);
  }
  // fireEchoAsync internally marks echo_attempted=1 before emitting and never throws

  // 8. Advance idempotency row from 'pending' to 'applied' with cached response.
  const response = { task_id: taskId, message_id: reserved.messageId, echo_status: 'pending' };
  applyIdempotency(ctx.user.id, idempotencyKey, response);

  return 202(response);
}

// Fire-and-forget echo. Never blocks the steer response. Echo status is reported through
// the next SSE event on the task (echo_succeeded / echo_failed / adapter_unavailable).
// Cycle-2 fix F7: outer try/catch wraps the entire body so synchronous throws from
// getMessagingGroup / getChannelAdapter don't escape as unhandled promise rejections.
function fireEchoAsync(task, text, ctx): void {
  if (task.surface_mode !== 'native_thread') return;                       // C4 / C10
  if (!task.child_platform_thread_id || !task.child_messaging_group_id) return;
  setImmediate(async () => {
    try {
      const mg = getMessagingGroup(task.child_messaging_group_id);
      const adapter = mg ? getChannelAdapter(mg.channel_type) : undefined;
      if (!adapter || typeof adapter.deliver !== 'function') {
        emitTaskEvent(task.task_id, { echo_status: 'adapter_unavailable' });
        return;
      }
      try {
        await adapter.deliver(mg.platform_id, task.child_platform_thread_id, {
          kind: 'chat',
          content: { text: `[via dashboard] ${text} — ${ctx.user.display_name ?? ctx.user.id}` },
        });
        emitTaskEvent(task.task_id, { echo_status: 'echoed' });
      } catch (err) {
        log.warn('steer echo failed (deliver threw)', { taskId: task.task_id, err });
        emitTaskEvent(task.task_id, { echo_status: 'echo_failed' });
      }
    } catch (err) {
      // Catches synchronous throws from getMessagingGroup / getChannelAdapter.
      log.error('steer echo aborted (sync throw before deliver)', { taskId: task.task_id, err });
      try { emitTaskEvent(task.task_id, { echo_status: 'echo_failed' }); } catch { /* swallow */ }
    }
  });
}
```

**ASSERT (M3):** the `steer_idempotency` row is reserved in a transaction BEFORE the inbound.db write. A retry with the same `(user_id, idempotency_key)` while status=`pending` reuses the same `message_id` and resumes the write rather than creating a duplicate. After status=`applied`, the cached response is returned without any side effect.

**ASSERT (S12):** echo runs in `setImmediate`, never awaited by the request handler. The 202 returns within ~10ms of the inbound write committing, regardless of Slack adapter latency. Echo outcome is delivered via the next SSE `task_event` for the task.

**ASSERT (S6):** SQLITE_BUSY thrown by `writeSessionMessage` returns 503 `{retry_after: 2}` and refunds the rate-limit budget. Other write errors leave idempotency `pending` so a client retry replays the same `message_id`.

**ASSERT:** echo failure does NOT fail the steer request (echo is best-effort transparency for the Slack channel; failure is reported as `echo_status` on the next SSE event).

**ASSERT:** member-role users get 403, not silent acceptance — explicit role enforcement (owner / global_admin / admin_of_group required for steer).

**ASSERT:** the agent receives the steer message indistinguishably from a Slack chat message in its prompt — same `kind='chat'`, same `content.text`, same `content.sender` field. The `_via: 'dashboard'` is metadata for log inspection, NOT for the agent prompt (the agent receives `content.text` as authored by `content.sender`).

### 4a. Unified transcript query (cycle-1 fix M30 + cycle-2 fix C6)

`GET /dashboard/api/tasks/:id` returns task metadata + the **last 200 messages combined** from the child session's `inbound.db` and `outbound.db`, merged by **timestamp** (primary) and **seq within direction** (tiebreaker), and labeled with direction.

**Why timestamp over pure seq (cycle-2 fix C6):** the host's `nextEvenSeq` reads only `MAX(messages_in.seq)`, while the container's `nextOddSeq` reads `max(MAX(messages_in.seq), MAX(messages_out.seq))` (verified at `container/agent-runner/src/db/messages-out.ts:51-54`). So container observes both sides; host observes only its own. That's enough to keep parity (host writes even, container writes odd) but does NOT give a strict global temporal ordering — the host can write `seq=4` AFTER the container wrote `seq=5`, because the host doesn't read the outbound max. Pure seq sort can therefore put a host-late message before a container-early message. Timestamp sort gives true temporal order; seq tiebreaks for same-millisecond writes.

```ts
function loadUnifiedTranscript(agentGroupId: string, sessionId: string, limit = 200) {
  const inbound  = openInboundDb(inboundPath(agentGroupId, sessionId));
  const outbound = openOutboundDb(outboundPath(agentGroupId, sessionId));
  try {
    const inRows = inbound.prepare(
      `SELECT id, seq, kind, timestamp, content
       FROM messages_in
       ORDER BY timestamp DESC, seq DESC LIMIT ?`,
    ).all(limit);
    const outRows = outbound.prepare(
      `SELECT id, seq, kind, timestamp, content
       FROM messages_out
       ORDER BY timestamp DESC, seq DESC LIMIT ?`,
    ).all(limit);
    const merged = [
      ...inRows.map((r)  => ({ ...r, direction: 'inbound' as const })),
      ...outRows.map((r) => ({ ...r, direction: 'outbound' as const })),
    ]
      // Sort by timestamp DESC; tiebreak by direction (inbound first; user input
      // happens before agent reply at the same millisecond), then by seq DESC.
      .sort((a, b) => {
        // Cycle-3 fix M1-c3: normalize the two timestamp formats before lexicographic
        // comparison. Host writes ISO-8601 with T+millis+Z (`new Date().toISOString()`);
        // container writes SQLite `datetime('now')` → `'YYYY-MM-DD HH:MM:SS'` (no T,
        // no millis, no Z). Direct string compare puts container rows BEFORE inbound at
        // the same instant because space (0x20) < T (0x54). Use Date.parse for both.
        const ta = Date.parse(a.timestamp);
        const tb = Date.parse(b.timestamp);
        if (ta !== tb) return tb - ta;                              // newer first
        if (a.direction !== b.direction) return a.direction === 'inbound' ? -1 : 1;
        return b.seq - a.seq;
      })
      .slice(0, limit)
      .reverse();   // oldest first for display
    return merged.map(decorateSource);
  } finally {
    inbound.close();
    outbound.close();
  }
}
```

**ASSERT (M30):** the transcript MUST read both `inbound.db` and `outbound.db`. A dashboard steer (which writes to inbound.db) MUST appear in TaskDetail within one SSE tick of being written. Tested with a headless task where Slack echo is NOT available — the inline-thread panel is the operator's only feedback surface.

**ASSERT (C6):** transcript ordering is by timestamp first, with direction (inbound-before-outbound at same timestamp) and seq as tiebreakers. Pure seq sort is forbidden because the host's nextEvenSeq doesn't observe outbound maxes — this can produce host-late messages with numerically lower seqs than container-earlier messages.

**`source` decoration heuristic** (defensive — falls back gracefully on unparseable rows):
- `direction='inbound'` and `content._via='dashboard'` → `'dashboard'`
- `direction='inbound'` and `content.platformId` set → `'chat'`
- `direction='outbound'` and `content.kind in ['status','progress']` → `'system'`
- Unparseable JSON content → `'agent'` (default — log warn but do not throw)
- else → `'agent'`

### 5. SSE feed (`src/dashboard/api/events.ts`) — revised cycle 1 (M22 + S11) + cycle 2 (M1 + M4 + B.DRIFT1)

Single global chokidar watcher, started at host bootstrap (eager — D6 lazy-chokidar was withdrawn per `decisions.yaml#withdrawn_decisions`).

**Watch path strategy (cycle-2 fix M1):** chokidar v4+ removed glob support. Use a directory watch with an `ignored` filter to capture per-session DB writes, plus a separate watch for the central DB:

```ts
// The central DB is opened with journal_mode = WAL (verified src/db/connection.ts:17).
// In WAL mode, writes accumulate in `v2.db-wal` and the main `v2.db` file's mtime does
// not change between checkpoints (which can be hours for a low-write central DB).
// chokidar+inotify on the main file would silently miss task admit / status / progress
// events on Linux production hosts.
//
// Replacement strategy: programmatic emission from every `tasks` mutation site.
// `applySpawnTask`, `applySpawnComplete`, `applySpawnFailed`, `applySpawnCancel`,
// `applySpawnProgress`, the watchdog reaper, and the orphan-task reconciler all call
// `emitDashboardEvent('task_event', { task_id, kind, ... })` immediately after their
// transaction commits. This is a small surface (~6 callsites), each adding a single
// line. The dashboard event-bus filters by per-user scope (per §2a) before fanout.
//
// Note for macOS dev mode (carry-over from cycle-2 W2-c2): the same approach also
// avoids the macOS FSEvents/SQLite-WAL flakiness documented in anthropics/claude-code
// #16523. By emitting programmatically, the dashboard never depends on filesystem
// observation of WAL writes regardless of OS.
```

**Why directory watch over the v3-style glob (cycle-2 fix M1):** the previous draft used `data/v2-sessions/*/*/outbound.db`, which v4+ silently watches as a literal filename. Directory watch + `ignored` filter is the chokidar-recommended migration path for v4+. New session directories are detected automatically by the recursive watcher; no `.add()` plumbing required.

**Cycle-2 fix M4 — inbound.db is now watched.** The previous draft only watched outbound.db; that meant the operator's own steer (which writes to inbound.db) didn't trigger an SSE invalidation. The new ignored filter accepts both inbound.db AND outbound.db.

**Path parsing:** on `change` for the session-DB watcher, parse `<SESSIONS_ROOT>/<agent_group_id>/<session_id>/(inbound|outbound).db` to derive `(agent_group_id, session_id, side)`. The path layout is fixed — no glob ambiguity.

**Central DB events (cycle-3 fix M2-c3):** `task_admit | task_status_change | task_progress | task_complete | task_failed | task_cancel` events are emitted **programmatically** from each `tasks`-mutation callsite, NOT via filesystem observation. This avoids the central DB's WAL-mode invisibility on chokidar-based watchers. Callsites (~6): `applySpawnTask`, `applySpawnComplete`, `applySpawnFailed`, `applySpawnCancel`, `applySpawnProgress`, `applySpawnTaskWatchdogFail`. Each adds one line: `emitDashboardEvent('task_event', { task_id, kind, ...payload });` after the transaction commits.

**Programmatic SSE emission (cycle-2 fix M4):** the steer write path emits an SSE invalidation directly after the inbound.db write commits, NOT via chokidar. This guarantees zero-latency feedback for the operator's own steer regardless of chokidar's debounce window:

```ts
// In applySteer, after step 5 commits:
emitDashboardEvent('inbound_message', {
  task_id: task.task_id,
  child_session_id: task.child_session_id,
  parent_agent_group_id: task.parent_agent_group_id,
  message_id: reserved.messageId,
});
```

The chokidar-driven path covers all other writes (Slack/Discord adapter writes to inbound.db, container writes to outbound.db, host writes to v2.db). The two paths are complementary — no double-emission concern because chokidar's `awaitWriteFinish` 200ms debounce coalesces a same-tick programmatic emit + filesystem event into one client-side refetch.

**Central DB journal mode (cycle-2 fix B.DRIFT1):** `data/v2.db` is opened with `journal_mode=WAL` per the project's central-DB convention (see `src/db/connection.ts`). chokidar-on-WAL is **reliable on Linux** (where the production host runs — `Linux 6.17.0-1009-oracle` per environment) but the upstream chokidar issue tracker documents flaky WAL detection on macOS FSEvents (anthropics/claude-code #16523). If macOS dev mode is needed (it currently is not — Dave's host is Linux), the recommended mitigation is `usePolling: true` on `centralDbWatcher` only, OR a touch-marker file written after each task-state mutation, OR shifting `data/v2.db` to `journal_mode=DELETE` for dev. Production deploys on Linux are unaffected. A4's validation method explicitly tests on Linux; macOS coverage is non-blocking.

On `change`: parse path → derive `(agent_group_id, session_id)` → look up the relevant task(s) by `child_session_id` → emit summary projection.

Subscribers are tracked in a `Map<userId, Set<SSEConnection>>`. On dispatch, emit only to subscribers whose `scopes` cover the affected task's `parent_agent_group_id` (per §2a). Owner / global_admin get all events; scoped admins get only their groups' events.

**Per-user connection limit:** max 20 concurrent SSE connections per `user_id`. 21st returns 429 with `Retry-After: 1`. Generous so an operator can have multiple tabs / browsers without hitting the cap.

**Aggregate connection limit (cycle-1 fix S11):** max 200 SSE connections per host process across ALL users. 201st returns 429 regardless of per-user count. Prevents multi-account abuse on shared hosts and protects the C5 memory budget.

**Keepalive — single global timer (cycle-1 fix M22):**

```ts
// EXACTLY ONE setInterval for the lifetime of the SSE module.
// NOT one per connection — that pattern is the documented #1 production
// SSE memory-leak (cf. arlabs 10k/min ingestor; OpenCode issue #15645).
let keepaliveTimer: NodeJS.Timeout | null = null;

function ensureKeepaliveTimer(): void {
  if (keepaliveTimer) return;
  keepaliveTimer = setInterval(() => {
    for (const conns of connectionsByUser.values()) {
      for (const conn of conns) {
        try {
          conn.res.write(':keepalive\n\n');
        } catch {
          conns.delete(conn);
        }
      }
    }
  }, 25_000);
  keepaliveTimer.unref();   // don't keep the process alive on its own
}
```

**ASSERT (M22):** the SSE module has EXACTLY ONE `setInterval`/`setTimeout` instance for keepalive — the global timer that iterates the connection Set. Per-connection timers are explicitly forbidden; reviewers will check this at /team-qa. Connection close triggers `conns.delete(conn)`; the timer is never per-connection.

**Connection close:** when the client disconnects (`req.on('close')`) or `conn.res.write` throws, the connection is removed from `connectionsByUser`. If the Map becomes empty across all users, the watcher and timer remain (they're cheap; lazy teardown was withdrawn).

**Shutdown:** `stopWebhookServer()` clears the keepalive timer, closes the chokidar watcher, and flushes/ends all SSE responses.

### 6. SPA build + serving — revised cycle 1 (M36 + S8)

- **Build:** `pnpm run build:dashboard` → `vite build` from `dashboard/` → emits `dist/dashboard/index.html` + `dist/dashboard/assets/*.js|css`.
- **Serving `/dashboard/`:** reads `dist/dashboard/index.html` and pipes via `fs.createReadStream`. Content-Type `text/html`. No cache.
- **Serving `/dashboard/static/*`:** path-traversal-safe (cycle-1 fix M36) with explicit MIME table (cycle-1 fix S8):

  ```ts
  const STATIC_ROOT = path.resolve(process.cwd(), 'dist/dashboard');
  const MIME: Record<string, string> = {
    '.js':   'application/javascript',
    '.mjs':  'application/javascript',
    '.css':  'text/css',
    '.html': 'text/html',
    '.json': 'application/json',
    '.svg':  'image/svg+xml',
    '.png':  'image/png',
    '.webp': 'image/webp',
    '.woff2':'font/woff2',
    '.ico':  'image/x-icon',
  };
  function serveStatic(req, res): void {
    const requested = decodeURIComponent(new URL(req.url, 'http://x').pathname);
    if (!requested.startsWith('/dashboard/static/')) return notFound(res);
    if (requested.includes('\0')) return notFound(res);                  // null-byte
    const tail = requested.slice('/dashboard/static/'.length);
    if (!tail || tail.endsWith('/')) return notFound(res);               // F10: dir or empty
    const resolved = path.resolve(STATIC_ROOT, tail);
    if (!resolved.startsWith(STATIC_ROOT + path.sep)) {
      return notFound(res);                                              // escape attempt
    }
    let stat: fs.Stats;
    try { stat = fs.statSync(resolved); } catch { return notFound(res); }
    if (!stat.isFile()) return notFound(res);                            // F10: catches dir
    const ext = path.extname(resolved).toLowerCase();
    const contentType = MIME[ext] ?? 'application/octet-stream';
    // ETag from mtime, honor If-None-Match, stream with fs.createReadStream
    // ...
  }
  ```

**ASSERT (M36):** the static handler MUST resolve every request path through `path.resolve(STATIC_ROOT, ...)` and reject any resolved path that does not start with `STATIC_ROOT + path.sep`. URL-decoded `..` sequences, encoded sequences (`%2e%2e`, `%2f`), and null bytes are all defeated by `path.resolve` + the prefix check + the explicit null-byte check. Reviewers verify by manually requesting `/dashboard/static/../../etc/passwd` and `/dashboard/static/%2e%2e%2f%2e%2e%2fetc%2fpasswd` — both must return 404.

**ASSERT (S8):** unknown extensions return `application/octet-stream`, NOT `text/plain`. Browsers refuse to execute `text/plain` JS modules, so a missing entry would silently break the SPA — a stricter default surfaces the configuration gap immediately.

- **Auth:** `GET /dashboard/` and `GET /dashboard/static/*` are NOT auth-gated — the SPA itself doesn't contain secrets, and gating bundle assets is hostile to browser caching. The SPA's first action is `GET /dashboard/api/auth/me`, which fails with 401 if no cookie, prompting the AuthGate component.

- **Token-in-URL safety net (cycle-1 fix M33):** if the index.html JS detects a `?token=` query param at load, it MUST `history.replaceState(null, '', location.pathname)` to remove it from history before any other action. The exchange endpoint accepts tokens only via POST body; URL tokens are an accidental-leak path, never a primary path.

### 7. Schema migration 028 — revised cycle 1 (M34 + M3 + S2)

```sql
-- dashboard_tokens stores HMAC-of-token, NOT raw bearer (cycle-1 fix M34).
-- Raw token is shown to the user via DM and never re-readable from the server.
CREATE TABLE dashboard_tokens (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id     TEXT NOT NULL REFERENCES users(id),
  token_hmac  TEXT NOT NULL UNIQUE,           -- hex of HMAC-SHA256(token_raw, server_key)
  issued_at   TEXT NOT NULL,
  expires_at  TEXT NOT NULL,
  used_at     TEXT
);
CREATE INDEX idx_dashboard_tokens_unused
  ON dashboard_tokens(user_id, expires_at)
  WHERE used_at IS NULL;

-- steer_idempotency uses reserve-before-write pattern with status (cycle-1 fix M3).
-- Row is INSERTed with status='pending' before the inbound.db write, then UPDATEd to
-- status='applied' with the cached response after the write commits. A retry with the
-- same key while status='pending' replays the same message_id to avoid duplicate writes
-- under host crash recovery; after status='applied', the cached response is returned.
CREATE TABLE steer_idempotency (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id         TEXT NOT NULL REFERENCES users(id),
  idempotency_key TEXT NOT NULL,
  task_id         TEXT NOT NULL,
  message_id      TEXT NOT NULL,                -- deterministic, set at reservation time
  status          TEXT NOT NULL DEFAULT 'pending', -- 'pending' | 'applied'
  reserved_at     TEXT NOT NULL,
  applied_at      TEXT,
  cached_response TEXT,                         -- JSON of the 202 body, populated when applied
  echo_attempted  INTEGER NOT NULL DEFAULT 0,   -- cycle-3 fix S6-c3: prevent double-echo on retry
  request_hash    TEXT NOT NULL,                -- cycle-3 fix S5-c3: sha256 of normalized text;
                                                -- replay with same key+different text → 422
  UNIQUE(user_id, idempotency_key)
);
CREATE INDEX idx_steer_idempotency_age
  ON steer_idempotency(reserved_at);

-- Pruning (cycle-1 fix S2): host-sweep tick (60s) runs:
--   DELETE FROM steer_idempotency
--   WHERE (status = 'applied' AND applied_at < datetime('now', '-60 seconds'))
--      OR (status = 'pending' AND reserved_at < datetime('now', '-300 seconds'));
-- 'applied' rows: 60s dedup window per C12.
-- 'pending' rows: 5min crash-recovery window. After 5min, a stuck pending row implies a
-- crash whose retry never came; safe to drop because the inbound write either landed
-- (agent already saw it) or was rolled back (no harm in retrying with a new key).
```

### 8. Memory budget reconciliation gate (C5)

Runs at `/team-ship` time. Procedure:

1. Cold-start the host (no dashboard) and snapshot RSS via `ps -o rss= -p $NODE_HOST_PID` after 60s warm-up. Call this `baseline_rss_kb`.
2. Restart the host with the dashboard, spawn 5 spawn tasks via the orchestrator (real `spawn_task` calls, native_thread surface where possible).
3. Open 5 dashboard tabs (5 SSE connections) and leave them open for 60min.
4. Snapshot RSS every 5min. Take the median of post-warm samples. Call this `dashboard_rss_kb`.
5. Compute `incremental_mb = (dashboard_rss_kb - baseline_rss_kb) / 1024`.
6. **Gate:** `incremental_mb <= 150`. Pass → ship. Fail → revisit chokidar scope, projection size, or framework choice.

This is a `/team-ship` gate, not a `/team-build` gate. Build can ship with the gate unmet but the unmet gate blocks final ship.

### 8a. /team-build smoke gate (cycle-1 fix S7)

A lighter sanity check that runs as part of `/team-build` so gross regressions surface before the heavy /team-ship reconciliation:

1. After /team-build's test suite passes, spin up the dashboard with 1 spawn task and 1 SSE connection.
2. Hold for 5 minutes. Snapshot host RSS at start and end.
3. Gate: `incremental_mb <= 30` over 5min.

This is a small fraction of the full C5 budget (150MB at 5 tasks / 60min) but catches the cases the C5 gate would have caught — chiefly the M22 per-connection-timer leak. Acceptance criterion is asymmetric — pass at this gate does not imply pass at /team-ship; fail at this gate is a /team-build failure.
