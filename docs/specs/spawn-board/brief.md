# Brief: Spawn Board — Multi-Task Steer Dashboard for Self-Orchestration

> Generated by `/team-brief` — 2026-05-10
> Saved to: `docs/specs/spawn-board/brief.md`
> Sibling to: `docs/specs/orchestrator-dispatch/` (Phase 1 spawn primitive). This is its Phase 2.

## Why this exists

Phase 1 of the spawn primitive (PR #80 + rework PR #81, merge `8fe1d86`) gives an orchestrator agent the ability to spawn N parallel sessions of itself within a single agent group, each in its own thread, all sharing workspace / memory / CLAUDE.md / channels. Today, the operator drives that fanout entirely via Slack/Discord — opening N threads, switching between them, losing the bird's-eye view of what's running. The spawn-board is the bird's-eye view: a single page that shows every spawned task at a glance, lets the operator drill into any one's history, and **lets the operator post a steering message into a child without leaving the board** (the message echoes into the child's Slack thread when there is one, so the channel stays in sync).

The dashboard is multi-task in the sense that operating on multiple parallel children is the explicit primary use case — not an "advanced" affordance. A single agent fanning out to 5 backlog items needs 5 active monitoring surfaces; the board collapses them into one.

## Requirements

### Read-only views

- **Kanban view** — every spawn task in the host's `tasks` table grouped into lanes by status: `pending`, `running`, `completed`, `failed`, `cancelled`. Each card shows: task ID, parent session ID (truncated), `task_content` preview (~80 chars), `admitted_at` relative time, `last_progress_message` if running, `fail_reason` if failed.
- **Task detail view** — drilling into a card reveals: full `task_content`, lifecycle timestamps (`admitted_at` / `started_at` / `completed_at` / `failed_at` / `cancelled_at`), `result_summary`, `surface_mode`, `child_session_id`, `child_platform_thread_id` if any, plus the **last 200 messages** of the child session (combined inbound + outbound, ordered by `seq`). For headless tasks, this same panel becomes the only conversation surface (no Slack thread exists).
- **Session list view** — every live session across the agent groups the viewer is authorized to see (see Auth below). Columns: agent group, session ID, last_active, container_status, messaging_group_id (if any), thread_id (if any). Useful for triaging "is the agent running" outside the spawn-task lens.

### Steer (write)

- **Operator can post a chat message into any child session from TaskDetail.** The message lands in the child's `inbound.db` as `kind='chat'` with the operator's `user_id` as sender. The agent processes it on its next turn, indistinguishable from a Slack/Discord chat message routed by the normal channel adapter pipeline.
- **When the child has a Slack/Discord thread** (`surface_mode='native_thread'` and `child_platform_thread_id IS NOT NULL`): the steer message is **echoed into that thread** via the channel adapter, prefixed `[via dashboard]` and suffixed with the operator's display name. Slack/Discord stays the source-of-truth audit log; lurkers in the thread see the steer with provenance.
- **When the child is headless** (`surface_mode='headless'`): the steer goes only into the child's `inbound.db`. The TaskDetail inline-thread panel is the operator's view of where the message went and what the agent does with it.
- **Concurrency model:** steers and Slack messages can both arrive while the child is running — both are written to `inbound.db` in arrival order (with the host-owned even-`seq` invariant preserved). The agent's normal batch-processing loop on the next turn handles both inputs together. No server-side queueing or rejection.
- **Rate-limit:** 30 steers per minute per `(user_id, child_session_id)`. Server returns `429` with `Retry-After` on overflow.
- **Idempotency:** client generates a UUIDv4 `idempotency_key` per submit; server stores `(user_id, idempotency_key)` for a 60s dedup window. Re-submit on flaky network is safe.

### Authentication

- Operator pastes a `/dashboard-token` from a DM with the orchestrator agent (or `nanoclaw` core); host generates a single-use 24h-TTL hex token, DMs it back. Operator pastes it into the dashboard's auth box (or via `?token=<>` query). Server exchanges for an `httpOnly` `Secure` `SameSite=Strict` cookie bound to `user_id`, expires in 12h.
- All `/dashboard/api/*` routes require the cookie. Owner sees all tasks/sessions; scoped admin sees only tasks/sessions for groups where they have an `admin` row in `user_roles` (uses existing `canAccessAgentGroup` from `src/modules/permissions/access.ts`).
- Steer authorization is the same scope: owner can steer any child; scoped admin can steer children whose parent agent group is in their admin scope.

### Live updates

- SSE (`text/event-stream`) feed at `GET /dashboard/api/events`. Server uses chokidar to watch `data/v2.db` and `data/v2-sessions/*/outbound.db` for changes; emits `event: <type>\ndata: <json>\n\n` payloads on debounced (200ms) write events.
- Event types: `task_admit`, `task_status_change`, `task_progress`, `child_message` (new outbound from a child session). Payloads are summary projections (task_id, fields-that-changed, ts), not full rows — client refetches detail on demand.
- Periodic `:keepalive\n\n` comments every 25s.

### Hosting

- New routes mounted on the existing Node host's webhook-server (`src/webhook-server.ts`), not a separate process. No new port. Memory budget reconciliation gate at `/team-ship`: with 5 active live tasks running for 60 minutes, dashboard incremental RSS over a no-dashboard Node baseline must stay under 150MB.
- The webhook-server gets a bounded multi-route refactor (~200 LOC of routing logic) to support three endpoint families: existing `/webhook/<adapter>` semantics, new `/dashboard/static/*` (Vite-built bundle from `dist/dashboard/` with extension-based content-type + ETag), new `/dashboard/api/*` (handlers write to Node response streams directly, bypassing the existing `fromWebResponse` Web-Response adapter — fatal for SSE).

### Frontend

- Vite + React SPA. Three views (KanbanBoard, TaskDetail, SessionList) plus an auth-gate component. URL hash anchors for view selection (`#/board`, `#/task/<id>`, `#/sessions`); no client-side router.
- State: SWR for fetching, `EventSource` for SSE.
- Visual quality: `bootstrap-domain:jony-ive` and `impeccable:impeccable` audits gate `/team-ship` per the existing render-check policy.

## Constraints

- **[HARD] Single Node process.** No separate dashboard service or new port. Mounted on the existing webhook-server. Process-level constraint C2 from Phase 1's design.
- **[HARD] Auth via existing `user_roles`.** Owner + scoped admins per agent group, using `canAccessAgentGroup` from `src/modules/permissions/access.ts`. No new auth subsystem; no API keys; no localhost-only bypass.
- **[HARD] Steer is fire-and-forget on the next-turn boundary.** No synchronous round-trip. The dashboard returns `202 Accepted` once `inbound.db` write + Slack echo are committed; the agent processes on its next turn. UI shows pending state via SSE.
- **[HARD] Echo to Slack only when `surface_mode='native_thread'` AND `child_platform_thread_id IS NOT NULL`.** Adapter capability check matches the Phase-1 `adapter_unavailable` invariant — if the adapter loses `postParent` capability, dashboard skips echo and surfaces a warning rather than failing the steer.
- **[HARD] Memory budget.** Incremental RSS under 150MB at 5 active live tasks over 60 minutes. Dashboard should be lean by default (filesystem-stream static assets, no in-process caches the SPA could compute on the client, summary projections rather than full rows over SSE) but the budget is loose enough to absorb incidental growth — the gate's job is to catch regressions like memory leaks or runaway listeners, not to enforce parsimony for its own sake. Reconciliation gate at `/team-ship` enforces this.
- **[HARD] Rate-limit on steer.** 30 per minute per `(user_id, child_session_id)`. Backstop against runaway clients or abuse.
- **[HARD] Server-side input validation.** Steer text capped at 4,000 chars. Reject empty / whitespace-only. Sanitize for SQLite write only — agent receives raw text in prompt.
- **[HARD] No client-side state persistence beyond URL hash.** Filter selections, scroll positions, etc. live only in the URL or component state. No localStorage. No IndexedDB.
- **[SOFT] Mobile responsive.** Desktop-primary. Mobile is "doesn't break" not "first-class": one-column collapse, touch-friendly tap targets on cards. No mobile-specific affordances.
- **[SOFT] Light/dark theme.** Defer to `jony-ive` audit; if it produces a single-theme system that's fine.

## Non-Goals (Explicitly Out of Scope)

- **Spawn from the dashboard.** Operator cannot create new tasks via the board. Spawning happens via the orchestrator agent's `spawn_task` MCP tool only.
- **Cancel from the dashboard.** Phase-1 cycle-2 review S15 explicitly deferred owner-cancel from the board to a follow-up. Not changing that here.
- **Multi-tenant beyond owner + scoped admins.** No invite system, no per-task ACLs, no public boards.
- **Cross-host federation.** This is a single-host dashboard. No remote-NanoClaw-instance discovery.
- **Historical analytics.** No charts, no time-series, no export. The board is for live operations, not retrospection.
- **Undo for steers.** Sent is sent. Slack/Discord don't let you unsend; the dashboard inherits that. (Edit/delete may be reconsidered in a follow-up if there's a real use case.)
- **Mobile-first redesign.** Per [SOFT] above, mobile is graceful-degrade only.
- **Custom kanban-lane discriminator.** Lanes are status-only. No "by parent_session" or "by tag" mode in 2A.
- **Real-time message streaming inside TaskDetail.** TaskDetail's last-200-messages window refreshes on the SSE `child_message` event; no progressive streaming of partial agent output. The agent already batches turn output before writing to `outbound.db` — there's no sub-turn signal to stream.
- **Push notifications.** Browser-tab visibility only. No service-worker or system-tray integration.

## Key Concepts

- **Spawn task** — a row in the central `tasks` table created by `applySpawnTask`. Has a parent session, no target group (self-orchestration; child runs in parent's group), idempotency key, surface_mode, and a lifecycle (`pending` → `running` → terminal).
- **Steer** — dashboard-originated chat into a child's `inbound.db`. Functionally indistinguishable to the agent from a Slack/Discord chat message; UX-distinguished by the `[via dashboard]` Slack-echo prefix.
- **Surface mode** — `native_thread` (child has a Slack/Discord thread to converse in) or `headless` (no platform thread; conversation lives only in `inbound.db` / `outbound.db`). Set at admit time, immutable afterwards.
- **Lane** — kanban column. Lane membership is determined exclusively by `tasks.status`. A task moves between lanes when its status transitions.
- **Inline thread panel** — the message-history component inside TaskDetail. Renders the last 200 messages of the child session (combined inbound + outbound, ordered by `seq`), with the steer input below.
- **Echo** — the dashboard's act of mirroring a steer into the child's Slack/Discord thread via the channel adapter's `postParent`. Format: `[via dashboard] <text> — <operator-display-name>`.

## Open Questions (defaulted)

- **Echo prefix exact wording.** Defaulting to `[via dashboard] <text> — <operator-display-name>`. Flag to user if this should be different.
- **Authorization for read-only session list.** Defaulting to "scoped to viewer's auth scope" (owner sees all; scoped admin sees their groups only). Same shape as task list.
- **TaskDetail history depth.** Defaulting to last 200 messages combined. Configurable later if real use surfaces a different need.
- **Connection pool / SSE concurrency.** Defaulting to 1 SSE connection per browser tab (more is wasted bandwidth — chokidar emits to all subscribers). Server limits: max 20 concurrent SSE connections per `user_id`.
- **What happens when the dashboard is open and the operator has no admin/owner access to ANY group?** Defaulting to a "No groups visible to your account" empty state. No 403 — the auth probe (`/dashboard/api/auth/me`) returns `{ user_id, scopes: [] }`.

## Success Criteria

- Operator with 5 simultaneously-running spawn tasks can see all 5 in the kanban view at once, drill into any one's task detail in under 2 clicks, and post a steering message that reaches the child agent on its next turn — without opening Slack.
- Slack/Discord remains the source-of-truth audit log: every steer for a `native_thread` child is visible in the channel, prefixed and attributed.
- A live spawn that transitions `pending → running → completed` is reflected on the kanban board within 1 second of each transition (200ms chokidar debounce + sub-1s SSE round-trip).
- Memory: with 5 concurrent live tasks for 60 minutes, host process RSS increment over a no-dashboard baseline stays under 150MB.
- Auth: a scoped admin user logged into the dashboard sees zero tasks for groups they don't admin, and steer attempts on out-of-scope tasks return `403`.
- Rate-limit: 31st steer in a minute on the same child session returns `429` with `Retry-After`.
- All standard host quality gates (`pnpm run build` / `pnpm test --run` / `cd container/agent-runner && bun test`) pass after merge.
