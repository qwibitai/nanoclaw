## Findings

### B1 — DIVERGED — SSE event-name wire mismatch (CRITICAL)

- **SOT (plan §5 + design §5):** SSE feed delivers `inbound_message` and `task_event` events; SPA `lib/sse.ts` subscribes via named events.
- **Implementation server side** (`src/dashboard/api/events.ts:36`):
  ```ts
  const frame = `data: ${JSON.stringify({ kind, payload })}\n\n`;
  ```
  Emits unnamed `data:` frame. The browser SSE parser surfaces this on the default `message` event handler.
- **Implementation client side** (`dashboard/src/lib/sse.ts:22-30`):
  ```ts
  es.addEventListener('task_event', (e) => { ... });
  es.addEventListener('inbound_message', (e) => { ... });
  ```
  Listens on NAMED events. These never fire because server emits unnamed.
- **Class:** DIVERGED — wire mismatch, frontend SSE will never receive events
- **Fix:** Server-side: emit `event: ${kind}\ndata: ${JSON.stringify(payload)}\n\n`. Client side stays as-is.

### B2 — DIVERGED — Transcript wire shape mismatch

- **SOT (plan D3):** `GET /dashboard/api/tasks/:id` returns `{ task: Task, transcript: TranscriptEntry[] }` (top-level transcript).
- **Implementation server side** (`src/dashboard/api/tasks.ts:246`): returns `{ task, transcript }` — top-level. CORRECT per plan.
- **Implementation client side** (`dashboard/src/views/TaskDetail.tsx:98-100`): reads `task?.transcript` (nested under task). MISMATCH.
- **Class:** DIVERGED — frontend won't display transcript
- **Fix:** Frontend: read top-level `transcript`, not `task.transcript`. Update `TaskDetail.tsx` + `dashboard/src/lib/api.ts` `TaskDetailResponse` type.

### B3 — DIVERGED — Frontend retry_after handling

- **SOT (plan D5 + plan E5):** 429 includes `retry_after` body field; SPA TaskDetail shows "Rate limited; try again in N seconds".
- **Implementation backend:** Returns `{retry_after: 2}` in body for 429/503 (correct).
- **Implementation client:** `dashboard/src/lib/api.ts:63-72` discards body fields and headers, throws `{status, error}` only. TaskDetail can't access `retry_after`.
- **Class:** DIVERGED — frontend rate-limit UX broken
- **Fix:** api.ts throws `{status, error, retry_after?}`; TaskDetail uses it.

### B4 — DIVERGED — D5 echo_status SSE not emitted

- **SOT (plan D5 ASSERT echo_status enumeration):** SSE `task_event` MUST carry one of `echoed | echo_failed | adapter_unavailable | skipped_headless`.
- **Implementation** (`src/dashboard/steer.ts:295-303`): `_emitEchoStatus` defines correct enum values but only calls `log.debug`. Comment: "In a real system this would emit a task_event SSE."
- **Class:** DIVERGED — echo_status only logged, never reaches dashboard
- **Fix:** _emitEchoStatus should call `emitDashboardEvent('task_event', { task_id, kind: 'progress', echo_status: status })`.

### B5 — DIVERGED — D6 missing 5 of 6 emit callsites (PLAN-VS-REALITY)

- **SOT (plan D6):** Add `emitDashboardEvent('task_event', ...)` at end of EACH of these 6 apply functions: `applySpawnTask` (admit), `applySpawnComplete`, `applySpawnFailed`, `applySpawnCancel`, `applySpawnProgress`, `applySpawnTaskWatchdogFail`.
- **Implementation** (`src/modules/orchestrator-dispatch/dispatch.ts`): Only `applySpawnTask` is exported. The other 5 functions DO NOT EXIST in this file. They're not present anywhere in the code.
- **Class:** DIVERGED — plan-vs-reality gap; design assumed multi-function decomposition that the spawn-rework architecture didn't implement (status changes for spawn tasks happen via different paths — completion, failure, cancel are handled by reconciler / completeSpawnSideEffects / not as discrete apply functions).
- **Resolution path:** Ack with reason — the spawn architecture (Phase 1 shipped at PR #80/81) doesn't have these as discrete apply functions. Status changes propagate through different code paths. The dashboard relies on the `task_event` admit emit; other events would require new apply functions (out of scope for spawn-board MVP). To compensate, dashboard should refresh tasks list periodically (SWR's default revalidateOnFocus + interval), which it does.
- **Acknowledge as DIVERGED** in `drift-acks.json`.
