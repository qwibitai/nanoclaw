# Brief: Orchestrator Self-Spawn + Memory-Efficient Dashboard

> Generated by `/team-brief` — 2026-05-09
> Saved to: `docs/specs/orchestrator-dispatch/brief.md`
>
> **REVISION 2026-05-10:** Phase 1 originally framed as cross-group dispatch (orchestrator → other agent group). Reframed to self-orchestration after live use revealed the design intent: the orchestrator spawns N parallel sessions of ITSELF in the same group, sharing workspace / memory / CLAUDE.md / channels. Group is the trust boundary; session is the work-unit boundary. Cross-group capability removed; `dispatch_*` MCP tools and system actions renamed to `spawn_*`. See `spawn-rework-plan.md` for the full correction trail. Phase 1 cross-group capability is recorded under "Originally built but removed" so the audit trail is preserved.

## Requirements

### Orchestrator self-spawn (Phase 1, revised)

- Designate an existing agent group as an "orchestrator" via a new role (analogous to `user_roles` for users). Owner-approved at first grant.
- An orchestrator agent has a new `spawn_task` MCP tool that creates a child session in its OWN agent group. Each spawn produces its own session/thread — sharing workspace, memory, CLAUDE.md, channels, container config with the parent. Only the conversation/thread is isolated.
- Each child session runs as a normal NanoClaw session in the parent's agent group, indistinguishable to the agent code from a session created by an inbound chat message.
- The owner can chat with each child task directly in its own native channel thread (Slack/Discord) to steer or discuss it, exactly as the owner chats with any NanoClaw session today.
- A durable host-side `tasks` table tracks every dispatch with full provenance (task id, parent session, child session, target agent group, status, deadlines, timestamps, summary).
- The orchestrator can cancel its own dispatched tasks. The owner can cancel any task. Cancellation is cooperative-first (soft) with a fixed grace window before hard kill.
- A watchdog in `host-sweep.ts` detects hung children (deadline elapsed, container exited unexpectedly, no progress beyond a configurable timeout) and fails the task with a structured reason posted back to the parent session.
- The reverse completion signal flows from child → host → parent session as a new system action (`task_complete` / `task_failed`) — not via the existing `channel_type='agent'` pathway.
- Orchestrator can list, inspect, cancel, and re-dispatch tasks via MCP tools. List scope is "tasks I (this orchestrator session) dispatched."

### Memory-efficient dashboard (Phase 2 — read-only)

- A web UI served from the existing Node host on a localhost port. No new long-lived service.
- Single SPA (React + Vite or Next.js — design phase decides) that reads `data/v2.db` and per-session inbound/outbound DBs directly. No separate API tier; same Node process serves the bundle and the data endpoints.
- Owner authentication via existing `user_roles` (only owner sees the dashboard initially; admin scoping is a future extension).
- Three primary views:
  1. **Kanban board** — tasks grouped by status (pending / running / completed / failed / cancelled), with target-agent-group affinity.
  2. **Task detail** — full message transcript (read-only), child session metadata, parent linkage, dispatch + completion timestamps.
  3. **Session list** — every live session across all agent groups, last-active timestamp, current container status.
- Live updates via a file watcher (chokidar) on session DBs, pushed to the browser over a long-lived stream (WebSocket or SSE — design decides based on memory).
- Visual quality: invoke `bootstrap-domain:jony-ive` and `impeccable:impeccable` skills during the implementation phase. The UI must feel intentional and inevitable — not just a functional admin panel.

## Constraints

- **[HARD]** Total additional resident memory for the dashboard process must not exceed **150MB** under normal load (no live tasks: ~80MB target; under load: ≤150MB ceiling). Multica-style stack (Postgres+pgvector + Go + Next.js) is rejected on this constraint alone.
- **[HARD]** No new long-running service. The dashboard is served from the existing Node host process or piggybacks on it (no new daemon, no Postgres, no Go binary).
- **[HARD]** `dispatch_task` is additive infrastructure. It must not modify or replace `src/modules/agent-to-agent/` (`create_agent`, `routeAgentMessage`). The two primitives coexist for different use cases (hierarchical subagents vs flat orchestration fanout).
- **[HARD]** Synthetic thread IDs must never reach a channel adapter that would attempt to deliver into a real platform thread. Verified failure mode: `session-manager.ts:340` writes `session.thread_id` to `session_routing.thread_id`, `delivery.ts:358-364` passes that to `adapter.deliver`. The Slack adapter's `chat.postMessage({ thread_ts: <synthetic> })` errors or posts to the channel root. Resolution: child sessions either run with `mgId = null` (no channel surface) OR use a real platform thread returned by a new adapter capability `createThread(...)`.
- **[HARD]** Reverse signal (`task_complete` / `task_failed`) must use a new registered system action in `delivery.ts`, NOT `channel_type='agent'`. Reusing agent-to-agent for the return path drags in `agent_destinations` ACL and per-pair name assumptions that the new primitive intentionally bypasses.
- **[HARD]** Cancellation authorization: `applyCancelTask` must verify the calling session is either (a) the task's `parent_session_id` (orchestrator can cancel own children) or (b) an owner/admin override path with audit. Codex-flagged finding from prior cycle: orchestrator-role check alone is insufficient — any prompt-injected orchestrator session could otherwise cancel any other orchestrator's tasks.
- **[HARD]** Watchdog timer that measures "no progress" must be reset by genuine progress signals (e.g. heartbeat or `task_progress` MCP call). A long-running task that actively reports must not be auto-killed. Codex-flagged finding from prior cycle.
- **[HARD]** Owner-authentication on the dashboard must reuse the existing `user_roles` resolution path (`src/modules/permissions/access.ts`). Do not introduce a parallel auth scheme.
- **[HARD]** Phase 1 ships with `adapter.createThread` implementations for **Slack and Discord**. Channels without thread primitives (Telegram, iMessage, WhatsApp, Email) degrade gracefully to `mgId=null` mode for orchestrator-spawned children — children are visible only via the orchestrator's task list, not as standalone channel threads.
- **[SOFT]** Visual quality of the dashboard. Invoke `jony-ive` and `impeccable` skills during build. Don't ship a default-MUI admin panel; the dashboard should feel premium and considered.

## Non-Goals (Explicitly Out of Scope)

- **Per-task git worktree isolation.** Code-touching tasks (e.g., 22 XZO items editing the same repo in parallel) require worktree-per-task to avoid git corruption. This is a separate, larger problem. Phase 1 assumes orchestrators sequence code-touching tasks (no parallel writes to the same repo). A worktree manager is a future feature, not part of this brief.
- **Refactoring `agent-to-agent`.** The existing module continues to serve the hierarchical-subagent pattern for `create_agent`. No changes to `agent-route.ts` or `create-agent.ts` beyond what's strictly needed (e.g., destination projection adjustments) to avoid breakage.
- **Multi-host orchestration.** All dispatch is on a single NanoClaw host. Cross-host federation is not in scope.
- **Chat-from-dashboard.** The Phase 2 dashboard is read-only. Posting messages from the dashboard into child sessions is deferred to a future Phase 2B. Steering happens via native Slack/Discord threads.
- **Auto-retry of failed tasks.** Failed tasks remain failed in the `tasks` table. The owner or orchestrator decides whether to re-dispatch. No automatic retry policy in v1.
- **Dashboard for non-owner users.** Initial dashboard is owner-only. Admin / scoped-admin access is a future extension.
- **Dependency-graph orchestration in v1.** The orchestrator agent (not the host) is responsible for sequencing tasks with dependencies. The `tasks` table can record dependency hints, but the host does not schedule based on them.
- **Replacing existing scheduling MCP tools.** `schedule_task`, `list_tasks`, `cancel_task` in `container/agent-runner/src/mcp-tools/scheduling.ts` continue to operate on per-session scheduled occurrences. The new `dispatch_task` is a separate primitive for orchestrator fanout, with its own list / cancel / progress tools (potentially sharing names if disambiguated by scope).

## Key Concepts

- **Orchestrator:** An existing agent group with the `orchestrator` role. Has access to the `dispatch_task` MCP tool. Talks to the user in its own session(s); each dispatch creates a separate child session in a target group. The orchestrator role is a property of the agent group, granted once by the owner.
- **dispatch_task:** New MCP tool, mounted only in orchestrator agent containers. Takes `(target_agent_group, task_content, task_id, optional deadline)`. Idempotent on `task_id`.
- **Child task session:** A normal NanoClaw session created by `dispatch_task`. Lives in the target agent group. Has a real platform thread (Slack/Discord — Phase 1) or `mgId=null` (other channels). Indistinguishable from a chat-spawned session from the agent's perspective.
- **task_complete / task_failed:** New system actions registered in `delivery.ts`. Child session emits one of these on its outbound; host updates the `tasks` row and notifies the parent session.
- **Parent session:** The orchestrator session that issued the dispatch. The reverse signal targets `tasks.parent_session_id`, not just the orchestrator agent group (an orchestrator may have multiple concurrent sessions across different chat threads).
- **Watchdog:** Periodic sweep in `host-sweep.ts` that scans `tasks` rows for: (a) deadline elapsed, (b) child container exited, (c) no progress past timeout. Reaper-style: non-event-driven, idempotent, recovers correctly across host restarts.
- **adapter.createThread:** New optional capability on the channel adapter contract. `(messagingGroupId, parentMessageId, title, firstMessage) → { threadId, messageId }`. Required for Phase 1 only on adapters that have native threads (Slack, Discord). Other adapters omit the capability; orchestrator falls back to `mgId=null` for those targets.
- **Lookback extractor:** A separate utility (`scripts/lookback.ts`, restored) that scans recent session DBs and surfaces unresolved commitments. Phase 0 — already complete.

## Open Questions (defaulted — flag if wrong)

- **Concurrency cap per orchestrator:** Defaulting to **5 parallel running tasks per orchestrator session**, configurable via `agent_roles.config_json` at role-grant time. Rationale: enough parallelism for typical workflows, low enough that 11 active agent groups × 5 parallel × container overhead doesn't melt a single host. Override at role grant.
- **No-progress timeout default:** Defaulting to **30 minutes** (1800 seconds) for tasks without an explicit deadline. Codex-flagged: this timer must be reset by `task_progress` calls or any host-observed child activity. Configurable via `agent_roles.config_json`.
- **Cancellation grace window:** Defaulting to **2 minutes** between cooperative cancel signal and hard container kill. Mirrors existing patterns in NanoClaw container shutdown semantics.
- **Eligible target agent groups:** Defaulting to **all existing agent groups**, with the constraint that an orchestrator cannot dispatch to itself or another orchestrator (prevents loops). Configurable allowlist/denylist on `agent_roles.config_json` if owner wants tighter scoping.
- **Live updates transport for dashboard:** Defaulting to **Server-Sent Events (SSE)** over WebSocket. Lower memory, simpler reconnection, sufficient for one-way push. Design phase can revisit if bidirectional is needed (it isn't for read-only).
- **Dashboard framework:** Defaulting to **Vite + React** over Next.js. Memory budget is the primary driver — Next.js dev/runtime adds non-trivial overhead, and SSR is unnecessary when the only client is the owner on localhost. Design phase confirms.
- **`task_id` derivation:** Defaulting to **caller-supplied idempotency key** (`task_id = task-${idempotency_key}`). Rationale: deterministic, allows orchestrator to reference task before host round-trip. Trade-off: `task_id` and `idempotency_key` namespaces are coupled — retention TTLs cannot diverge. Document the coupling in schema.
- **Dispatch authorization model:** Defaulting to **role-grant gives blanket dispatch authority across all non-orchestrator groups**. The orchestrator role is the grant. No per-target ACL row required (intentionally diverges from `agent_destinations`). Owner-approved once at role grant; thereafter orchestrator dispatches without per-call approval. Cancellation requires `parent_session_id` match (Codex finding).

## Success Criteria

1. **Orchestrator can dispatch a 22-item task list** like the XZO example pasted in conversation. Each item becomes its own session in its target agent group. The orchestrator's own thread receives a structured launch summary (task IDs and their target groups).
2. **Owner can click into any child task's Slack/Discord thread** and converse with the child agent natively. Replies in the child thread are read by the child session, not the orchestrator's session.
3. **Child task completion is reflected in the orchestrator's session within ~60 seconds** of the child emitting `task_complete`. The orchestrator's session sees a structured system message with the child's summary and result.
4. **The watchdog correctly fails a hung task** within `no_progress_timeout` after the child stops reporting (verified by killing a child container manually and observing the parent's notification).
5. **A long-running task that calls `task_progress` periodically does NOT get auto-killed.** The watchdog timer resets on each progress signal. Codex-flagged regression test.
6. **Dashboard launches at < 200MB peak resident memory** with 5 active live tasks. Day-to-day idle resident < 100MB. Verified by `ps` / `htop` over a 1-hour test session.
7. **Dashboard live updates appear within ~1 second** of a child session writing to its outbound DB. Verified by writing a test message and observing in the browser.
8. **Visual quality bar:** the dashboard passes a single `jony-ive` audit pass with no MUST-FIX findings. Spacing, hierarchy, typography, and motion meet the audit's bar without compromise.
9. **Orchestrator role grant requires owner approval.** Verified by attempting to grant from a non-owner session and observing rejection. First grant of orchestrator role to a non-owned agent group goes through the existing approvals primitive.
10. **The 4 outstanding items from the lookback extractor (Phase 0)** are visible and clickable in the eventual dashboard once it's built.
