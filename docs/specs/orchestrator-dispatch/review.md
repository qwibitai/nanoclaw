## MUST-FIX Findings

### [M3-1] Cancelled internal-only tasks leak `child_summary_posted_at = NULL` permanently

- **Raised by:** Reviewer A (A-3-1)
- **Finding:** `applyCancelTask` step 2 sets `status='cancelled'` and `cancelled_at=now` but does not set `completed_at`. §Reverse signal step a's UPDATE has `WHERE ... AND status IN ('dispatched','running')`, so for an already-cancelled task it's a no-op and `completed_at` stays NULL. Step c for internal-only tasks does `UPDATE ... SET child_summary_posted_at=completed_at WHERE child_thread_id IS NULL` — when `completed_at IS NULL`, this writes NULL into `child_summary_posted_at`. The `idx_tasks_finalization` predicate happens to skip these rows (because `child_thread_id IS NULL` makes the inner AND false), so the watchdog doesn't loop — but any external invariant "finalized → all four markers non-null" is now silently false.
- **Effort:** Trivial · **Impact:** High (data invariant violation; future code using `child_summary_posted_at IS NOT NULL` as a finalization check will misbehave)
- **Resolution:** In `applyCancelTask` step 2, also set `completed_at=now`. Or, in §Reverse signal step c's internal-only branch, fall back to `COALESCE(completed_at, cancelled_at, now)`.
- **Status:** Open

### [M3-2] Cap-promotion atomic UPDATE specifies `dispatch_state='creating_thread'` (or `'session_created'` for non-thread fallback) — choosing between them requires adapter resolution never scoped inside the transaction

- **Raised by:** Reviewer A (A-3-2)
- **Finding:** §Dispatch flow line 237-239 says cap-aware promotion sets `dispatch_state='creating_thread' (or 'session_created' for non-thread fallback)`. The "(or)" requires resolving `agent_groups → messaging_groups → adapter.supportsThreads`. If that resolution happens outside the transaction, two concurrent dispatches racing on the cap could read different snapshots; if inside, the transaction grows to span application logic.
- **Effort:** Low · **Impact:** High (ambiguous starting state for builders + reconciler resume correctness depends on the persisted state being right)
- **Resolution:** Either (a) resolve adapter capability BEFORE the cap transaction and pass the dispatch path in (acceptable — the agent_group + mg pair's adapter capability is stable); or (b) introduce a single neutral starting state `dispatch_state='admitted'` and let step 5 branch on adapter capability, decoupling cap admission from path selection.
- **Status:** Open

### [M3-3] `dispatch_approval_mode='session-token'` (the DEFAULT) references a token concept with no schema, no storage table, no integration with `src/modules/approvals/primitive.ts`

- **Raised by:** Reviewer A (A-3-3), Codex C-3-6 (merged)
- **Finding:** §Dispatch authorization scope introduces session-tokens with N=30 dispatches and M=60min expiry, but no migration adds `dispatch_run_tokens`, no column on `sessions`, no field on `pending_approvals`. The flow says "issuing a `request_token` system row to the orchestrator if the token is missing/expired, and blocking the dispatch until the owner approves." But `src/modules/approvals/primitive.ts:188` `requestApproval` is fire-and-forget — no token concept. With `'session-token'` as the DEFAULT, the unspecified path is the one that runs.
- **Effort:** Medium-High (requires either a real token substrate OR scope reduction)
- **Impact:** High (the default mode is unimplementable as written)
- **Resolution:** Two options:
  - **(a)** Add a real substrate: `dispatch_run_tokens(token_id, parent_session_id, agent_group_id, dispatches_remaining, expires_at, granted_by)` table + migration + `consumeDispatchToken()` helper + a registered approval handler that creates the token on approve.
  - **(b)** **Drop `'session-token'` from v1.** Ship `'none' | 'per-dispatch'` only. Default to `'none'` for orchestrator groups (matches brief R13 — owner approves the role grant once). Defer the run-token feature until the storage and approval-handler shape are designed in a follow-up spec.
  Recommend (b) — keeps v1 minimal; (a) is a real ~2-day spike that bloats the MVP.
- **Status:** Open

### [M3-4] Cap check uses `≤ 6` and `≤ 3` instead of `< 6` and `< 3` — off-by-one allows 7th task per orchestrator / 4th per target group

- **Raised by:** Codex C-3-1
- **Finding:** §Dispatch flow CAP CHECK + atomic transition: `COUNT(*) ≤ 6` passes when current count is exactly at 6, then promoting adds one more to reach 7. Same for the per-target-group `≤ 3` check.
- **Effort:** Trivial · **Impact:** High (cap discipline broken, R5 mitigation off by one in both initial dispatch and watchdog promotion paths)
- **Resolution:** Change `≤ 6` to `< 6` (and `≤ 3` to `< 3`). Or equivalently `COUNT(*) + 1 <= 6`.
- **Status:** Open

### [M3-5] Initial dispatch promotion checks caps but does NOT require dependencies complete before promoting `pending → dispatched`

- **Raised by:** Codex C-3-2
- **Finding:** §Dispatch flow CAP CHECK only checks the two cap conditions. A task submitted with `blocked_on` will start immediately if cap allows. That breaks dependency ordering — the §Reverse signal fan-out step never gets a chance to unblock it because it's already dispatched.
- **Effort:** Low · **Impact:** High (dependency model broken for tasks with `blocked_on` set at dispatch time)
- **Resolution:** Add `NOT EXISTS (SELECT 1 FROM task_dependencies td JOIN tasks t ON t.task_id = td.blocked_on_task_id WHERE td.task_id = ? AND t.status != 'complete')` to the cap-promotion transaction. The single shared promotion path (initial dispatch + watchdog responsibility 4) checks both caps AND dependencies in one atomic step.
- **Status:** Open

### [M3-6] Authorization integrity gap — `tasks.child_session_id` can drift from `sessions.dispatch_task_id`; authorization should bind to session-side capability

- **Raised by:** Codex C-3-3
- **Finding:** §Authorization model says "(task_id ↔ child_session_id) is the capability" and authorize is `task_id = ? AND child_session_id = <source session>`. But the schema only enforces UNIQUE on `sessions.dispatch_task_id`; nothing enforces that `tasks.child_session_id` matches `sessions.dispatch_task_id` symmetrically. A migration backfill, retry bug, or ad-hoc write could leave them out of sync — and authorization trusts the stale `tasks.child_session_id`.
- **Effort:** Low-Medium · **Impact:** High (defense-in-depth gap; cycle-2 M2-E removed the nonce on the assumption that the binding is unforgeable, this finding shows it's not strictly enforced)
- **Resolution:** Authorize from the session-side capability: `WHERE source_session.dispatch_task_id = ? AND source_session.id = <source session id>` (then JOIN to tasks for status/agent_group). Add a SQLite trigger `BEFORE UPDATE OF child_session_id ON tasks WHEN NEW.child_session_id IS NOT NULL AND (SELECT dispatch_task_id FROM sessions WHERE id = NEW.child_session_id) != NEW.task_id BEGIN SELECT RAISE(ABORT, 'tasks.child_session_id binding mismatch') END`. This makes the binding self-defending.
- **Status:** Open

### [M3-7] `dependents_released_at` marker is set but the released list itself is not persisted — crash between fan-out and parent notification produces inaccurate parent message

- **Raised by:** Codex C-3-4
- **Finding:** §Reverse signal step b computes `dependents_unblocked` and persists `dependents_released_at=now` but doesn't persist the list. Step d (parent notification) needs the list. If the host crashes between b and d, the watchdog sees `dependents_released_at IS NOT NULL` and trusts that the work is done — but step d never ran, and on resume there's no persisted list to put into the parent message. Recomputing later can produce a different list (more dependencies have completed in the meantime), so the parent system row's `dependents_unblocked` field becomes stale or misattributed.
- **Effort:** Low · **Impact:** High (cycle-2 M2-F's reordering only solved the in-memory ordering — not crash durability)
- **Resolution:** Persist the release set transactionally. Either (a) new column `dependents_unblocked_json TEXT` on `tasks`, set in step b alongside `dependents_released_at`; or (b) new table `task_dependency_releases(completed_task_id, released_task_id)` populated in step b. Step d reads from the persisted set, never recomputes. (a) is simpler and matches the rest of the schema's denormalized markers.
- **Status:** Open

### [M3-8] `parentPlatformMessageId` and `threadId` persisted only after adapter returns — crash mid-call leaves task in `creating_thread` with no durable platform IDs

- **Raised by:** Codex C-3-5
- **Finding:** §Dispatch flow step 5a says `adapter.createThread` owns both platform parent post and thread creation, returning `{ parentPlatformMessageId, threadId }`. The host persists both AFTER the adapter returns. A crash after the platform parent/thread is created but before the DB UPDATE leaves the task in `dispatch_state='creating_thread'` with `parent_platform_message_id IS NULL` AND `child_thread_id IS NULL`. The reconciler can't call `getThreadId(parentPlatformMessageId)` because the parent ID isn't durable. Re-calling `createThread` produces duplicate parent messages and duplicate threads.
- **Effort:** Medium · **Impact:** High (M1's "crash idempotency" closure was incomplete — the adapter call itself isn't idempotent under the current contract)
- **Resolution:** Split the adapter contract into two staged operations:
  1. `adapter.postParent(messagingGroupId, localParentMessageId, title): Promise<{ parentPlatformMessageId, messageId }>` — host persists `parent_platform_message_id` IMMEDIATELY after this returns.
  2. `adapter.createThreadFromParent(messagingGroupId, parentPlatformMessageId, firstMessage): Promise<{ threadId }>` — host persists `child_thread_id` after this returns.
  On reconciler retry from `creating_thread`: if `parent_platform_message_id IS NULL`, re-call `postParent` (PK-protected on local message id via `writeSessionMessage`); if `parent_platform_message_id IS NOT NULL` but `child_thread_id IS NULL`, call `getThreadId(parent_platform_message_id)` first, adopt if found, else call `createThreadFromParent`. Both adapter methods individually idempotent; staged persistence makes mid-call crashes recoverable.
- **Status:** Open

## SHOULD-FIX Findings

### [S3-A] `createDispatchedSession` insert-or-select pattern under concurrent retries is under-specified

- **Raised by:** A-3-4
- **Resolution:** Specify as `INSERT ... ON CONFLICT(dispatch_task_id) DO NOTHING; SELECT ... WHERE dispatch_task_id = ?` (better-sqlite3 supports this). Or document the catch-and-re-SELECT idiom.

### [S3-B] No dependency-cycle detection in `applyDispatchTask` — multi-hop cycles will hang `pending` forever

- **Raised by:** A-3-5
- **Resolution:** Add recursive-CTE check in `applyDispatchTask` that, before INSERTing `task_dependencies` rows, walks the closure of `blocked_on` and rejects if the new task_id is reachable.

### [S3-C] `BEGIN IMMEDIATE` / `busy_timeout` discipline not called out for cap-promotion transaction (Reviewer B MEDIUM drift)

- **Raised by:** Reviewer B
- **Resolution:** Add an explicit "Transaction discipline" subsection in §Dispatch flow specifying `BEGIN IMMEDIATE` for the cap-promotion + initial-INSERT path, and document the `busy_timeout` value the host opens its better-sqlite3 connection with.
