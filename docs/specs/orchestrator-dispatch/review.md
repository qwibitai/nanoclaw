## MUST-FIX Findings

### [M1] `cancel_task` MCP tool name collision with shipping scheduling module

- **Raised by:** Reviewer A
- **Finding:** The new `cancel_task` orchestrator-only MCP tool collides with the existing `cancel_task` tool exported by `container/agent-runner/src/mcp-tools/scheduling.ts:222`. The scheduling module also registers a delivery action named `cancel_task` (`src/modules/scheduling/index.ts:31`). The MCP tool barrel imports both with no per-session gating; `registerTools` either silently overwrites the existing tool or throws on duplicate registration. The dispatcher would steer any `cancel_task` call to whichever handler registered last.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (silently breaks shipping scheduling)
- **Suggested resolution:** Rename the orchestrator tool to `cancel_dispatched_task` (and the system action to `cancel_dispatched_task` to match). `extend_deadline` is also orchestrator-coined; verify no collision with scheduling — currently no collision but pin the namespace explicitly.
- **Fact-check:** Verified against `container/agent-runner/src/mcp-tools/scheduling.ts:222`, `src/modules/scheduling/index.ts:31`, `src/delivery.ts:558`.
- **Status:** Open

### [M2] No primitive exists for conditional per-session MCP tool mounting

- **Raised by:** Reviewer A
- **Finding:** Design says `dispatch_task` is "Mounted only when the agent group has the `orchestrator` role" and `task_complete` is "auto-mounted into any session whose `tasks.child_session_id` matches the session id." But `container/agent-runner/src/mcp-tools/index.ts:9-22` is an unconditional barrel — every tool registers at module scope. There's no host→container per-session tool allowlist channel.
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (security regression — tools register globally, leaving the orchestrator/completion gating only at the host's `applyDispatchTask`/`applyTaskComplete` handlers; container-side becomes an inviting attack surface for prompt-injected agents in ANY group)
- **Suggested resolution:** Spec the gating mechanism. Option (a, recommended): host writes a per-session `enabled_tools.json` to the session dir at spawn time; `mcp-tools/index.ts` reads it and skips registration of tools not in the allowlist. Option (b): tools register unconditionally but each handler's first action is a host capability check via a session-state file. Option (a) is cleaner.
- **Fact-check:** Verified against `container/agent-runner/src/mcp-tools/index.ts:9-22`.
- **Status:** Open

### [M3] SSE emission point not specified — host's existing read path doesn't have an observer registry

- **Raised by:** Reviewer A
- **Finding:** Design says "Piggyback an SSE event emitter on that same code path — when the host's existing read loop ingests a new `messages_out` row that's relevant to a watched task, push the event over SSE." But `src/delivery.ts:182-256` (`drainSession`) reads outbound rows and routes them; there's no `onOutboundRow` observer hook. State-relevant changes happen inside `applyTaskComplete` (after DB transaction commits), separate from the chat-row delivery path. Implementer can't "just piggyback."
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (implementer must invent a primitive; without spec they may get the emission semantics wrong — e.g. emit before commit, lose events)
- **Suggested resolution:** Spec two emission points concretely:
  - **State changes**: `applyTaskComplete` and watchdog handlers emit `task.completed` / `task.failed` / `task.orphaned` AFTER the DB transaction commits.
  - **Transcript streaming**: extend `drainSession` (or add an `onDeliveredHook`) that emits `task.message` after a chat row delivers — only when the session has a corresponding `tasks.child_session_id` row.
  The SSE bus is a new module with a small subscriber registry, not a delivery-loop addendum.
- **Fact-check:** Verified against `src/delivery.ts:182-256`.
- **Status:** Open

### [M4] `model`/`effort` per-task overrides have no plumbing path to child container args

- **Raised by:** Reviewer A
- **Finding:** `dispatch_task` accepts `model` and `effort` and the schema has columns for them, but the existing per-spawn model/effort lookup (`container-runner.ts:199-207`) reads from `messaging_group_agents.default_model` / `default_effort` keyed by `(messaging_group_id, agent_group_id)`. Orchestrator-dispatched children with `mg_id IS NULL` (fallback) have no wiring; children with `mg_id IS NOT NULL` would consult the orchestrator's wiring (wrong). Design says "pass-through to child container's runtime config" but doesn't say how.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (the per-task model/effort feature visibly does nothing for fallback children and silently uses orchestrator's wiring for thread children)
- **Suggested resolution:** `spawnContainer` looks up the task by `child_session_id` and applies a precedence chain: `tasks.model` → `messaging_group_agents.default_model` → `container.json` → DEFAULT. Same for `effort`. Document the chain in the spec.
- **Fact-check:** Verified against `src/container-runner.ts:199-207` (existing per-spawn model lookup).
- **Status:** Open

### [M5] Reconciler replay safety + `idempotency_key` adapter parameter both missing

- **Raised by:** Reviewer A (#7), Reviewer C (#3, merged)
- **Finding:** The reconciler description says steps are idempotent ("re-posting a parent message uses idempotency_key on the adapter, re-resolving a session is a lookup") OR the task is marked `failed` "if resumption isn't safe." But: (a) `adapter.createThread(...)` has no idempotency_key parameter in its signature — there's no way to dedupe a retried call. (b) The first external side effect is the parent message post; if the host crashes between adapter call success and persisting `external_thread_id`, replay can't re-find the thread (Slack threads are addressed by `(channel, ts)`; if the `ts` was never persisted, the orphan thread can't be cleaned up by the reconciler). G6 ("a crash mid-flow leaves no orphaned platform threads") is **not actually met**.
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (G6 violated; orphan platform threads on crash require manual cleanup)
- **Suggested resolution:** **Reorder the dispatch flow** — do all internal-only steps (insert task, resolve session, inject prompt) BEFORE any external side effect. Only post the parent message + `createThread` once the child session is durably created. A crash before the platform call leaves zero external state. Combined with making `idempotency_key` mandatory on `dispatch_task` (host generates one if orchestrator omits) and accepting that the adapter call itself is "post once, persist once" with the persist happening synchronously after the post returns, the orphan window narrows to "host crashed between adapter call returning and DB write" — which is unavoidable without an idempotent adapter API. For that residual window: the reconciler logs an "untracked external thread possibly created" warning and surfaces to the dashboard; manual cleanup is required for that specific crash window only.
- **Fact-check:** Verified against design § Dispatch flow (no idempotency on adapter call) and `src/channels/adapter.ts` (no createThread idempotency parameter is feasible — would need a multi-call adapter API).
- **Status:** Open

### [M6] Cap-released pending tasks have no scheduler trigger

- **Raised by:** Reviewer A (#8), Reviewer C (#2, merged)
- **Finding:** Step 3g says cap-hit tasks "stay `pending` until a slot frees." Watchdog has 6 steps; none scan pending tasks for cap availability. Step 5e of `applyTaskComplete` (dependency unblock) is the only path from pending → dispatch — but a cap-released task with no dependents waits forever. The 22-task XZO triage with per-target-group cap=3 is the documented use case and would deadlock.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (documented use case deadlocks)
- **Suggested resolution:** Add watchdog step 0 (or fold into existing step): "Pending-task scheduler — for each `tasks` row with status='pending' AND no unmet dependencies AND under both concurrency caps, re-enter dispatch flow at step 5." Single transaction with the rest of the sweep. Also fire this same scheduler check from `applyTaskComplete`'s end (when a slot frees from completion).
- **Fact-check:** Verified against design § Watchdog (6 steps, none scan pending).
- **Status:** Open

### [M7] Bearer token in `logs/dashboard-token.log` — wrong storage for a credential

- **Raised by:** Reviewer A
- **Finding:** Design says "random 32-byte token... written to `logs/dashboard-token.log`." But `logs/` is the standard troubleshooting directory (`logs/nanoclaw.error.log`, `logs/nanoclaw.log` per CLAUDE.md). Token rotation on every restart pushes the credential through every log-rotation policy and tail tool. The token is the entire dashboard auth boundary; it should not live in the same directory the spec says to share when debugging.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (security regression; token leakage via log sharing)
- **Suggested resolution:** Write to `data/dashboard-token` (chmod 600), print the path to setup output once on each generation, never log the value. References standard precedent: OneCLI vault, OAuth token files.
- **Fact-check:** Verified against CLAUDE.md § Troubleshooting (logs/ is debug-share territory).
- **Status:** Open

### [M8] Reverse signal "child subthread write" violates the two-DB invariant

- **Raised by:** Reviewer A (#12), Reviewer B (MEDIUM drift, partial), Reviewer C (#7, merged)
- **Finding:** `applyTaskComplete` step 2c says "write an outbound chat message to the child session containing the summary text." But the host is not a writer of `outbound.db` — that's the container's exclusive role per CLAUDE.md ("Exactly one writer per file"). Writing to the child's `outbound.db` from the host violates the invariant.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (invariant violation; cross-mount cache semantics break)
- **Suggested resolution:** Host calls `deliveryAdapter.deliver(channelType, platformId, threadId, 'chat', summary)` directly from `applyTaskComplete` — same path as the dispatch flow's parent-message post. Bypasses any session DB, doesn't violate the invariant. Spec this explicitly: "Completion summary is delivered to the child's subthread via direct adapter call, NOT via writing to the child's outbound.db."
- **Fact-check:** Verified against CLAUDE.md two-DB invariant section.
- **Status:** Open

### [M9] `extend_deadline` MCP tool has no caller-identity authorization check

- **Raised by:** Reviewer A
- **Finding:** Design specs `extend_deadline(task_id, new_deadline)` for use by the child but doesn't include an auth check. Same pattern as M5 from cycle 1 (which closed `task_complete`) but unaddressed here. Any agent that learns a task_id can extend any task's deadline (capped only by `MAX_EXTENSIONS=3`).
- **Effort to Fix:** Low
- **Impact if Ignored:** High (security gap — same severity as cycle-1 M5)
- **Suggested resolution:** Apply the same authorization check as `task_complete`: handler verifies `task.child_session_id == source session id` AND `task.target_agent_group_id == source agent_group_id` AND status is non-terminal. Reject otherwise.
- **Fact-check:** Verified against design § Watchdog `extend_deadline` paragraph.
- **Status:** Open

### [M10] `dispatch_state` mutates while `status='pending'` — invisible to reconciler

- **Raised by:** Reviewer C
- **Finding:** Dispatch flow step 5 persists intermediate `dispatch_state` values (`thread_created`, `session_created`, `prompt_injected`, `wake_sent`) before step 5g sets `status='dispatched'`. But the reconciler and `idx_tasks_dispatch_state` scan ONLY `WHERE status='dispatched'`. A crash after any of those intermediate steps leaves a task in `status='pending'` with real platform/session side effects, invisible to recovery.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (G6 violated; orphan side effects untracked by recovery loop)
- **Suggested resolution:** Add explicit `status='dispatching'` BEFORE the first side effect; reconciler scans `status IN ('dispatching', 'dispatched')`. Concurrency caps count 'dispatching' too. Only transition to `dispatched` after step 5g (wake succeeds). Paired with M5's reorder, the dispatch state machine becomes:
  - `pending` → `dispatching` (commit before any external side effect; reconciler can see)
  - `dispatching` → `dispatched` (after wake)
  - `dispatched` → `running` (after first non-status outbound row)
  - `running` → `complete | failed | cancelled` (terminal)
- **Fact-check:** Verified against design § Data model index spec (`WHERE status='dispatched'`) and § Watchdog step 1.
- **Status:** Open

### [M11] Bearer token cannot be sent on SSE — browser EventSource doesn't allow custom headers

- **Raised by:** Reviewer C
- **Finding:** Dashboard § Auth requires `Authorization: Bearer <token>` "on every SSE connection." But the native browser `EventSource` API does NOT allow setting arbitrary headers. The authenticated SSE design will not work as written.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (auth model is incoherent; SSE will either be unauthenticated or won't connect)
- **Suggested resolution:** Switch SSE auth to a cookie-based path. Owner enters the token once via a setup endpoint that issues an `HttpOnly`, `SameSite=Strict`, `Secure=false` (localhost) session cookie scoped to `/`. SSE connections authenticate via the cookie; mutating endpoints accept either bearer token (for CLI/curl) OR the session cookie (for browser). This is the standard pattern for browser-issued SSE auth.
- **Fact-check:** Verified against MDN EventSource documentation.
- **Status:** Open

### [M12] Failed/cancelled prerequisites don't propagate to dependents

- **Raised by:** Reviewer C
- **Finding:** Reverse signal step 2e fans out only when prerequisites are `complete`. A dependent task with no deadline whose prerequisite `failed` or was `cancelled` remains `pending` forever. The state machine has no terminal path for "blocked by failed dependency."
- **Effort to Fix:** Low
- **Impact if Ignored:** High (state machine has a dead-end; tasks accumulate as zombies)
- **Suggested resolution:** Extend `applyTaskComplete`/`applyTaskCancel` step 2e: when a prerequisite transitions to `failed` or `cancelled`, scan dependents and transition them to `failed` with `failure_reason='blocked dependency <task_id> failed: <prerequisite's failure_reason>'`. Add the propagation to the watchdog as a fallback (cycle through pending tasks whose prerequisites are terminal-non-success).
- **Fact-check:** Verified against design § Reverse signal step 2e.
- **Status:** Open

### [M13] Watchdog overwriting `complete`/`failed` → `orphaned` destroys the terminal outcome

- **Raised by:** Reviewer C
- **Finding:** Watchdog step 5: "tasks where status IN ('complete','failed') with completed_at recent AND parent_session is no longer active. Mark `status='orphaned'`." But this overwrites the actual completion outcome. A task that completed stops counting as complete for later dependency fan-out; dashboard/history queries lose the primary result state.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (data integrity — terminal status gets clobbered)
- **Suggested resolution:** Don't overwrite `status`. Add separate columns: `parent_delivery_state` (`delivered` | `orphaned`), `orphaned_at`, `owner_dm_sent_at`. Status stays `complete`/`failed`/`cancelled`. Watchdog step 5 sets the new fields, never touches status.
- **Fact-check:** Verified against design § Watchdog step 5.
- **Status:** Open

### [S1] `writeSessionMessage` API not specified for the parent system row write

- **Raised by:** Reviewer A
- **Finding:** Reverse signal step 2d "direct-write a kind='system' inbound row" doesn't specify the API. Should be `writeSessionMessage(parent_session.agent_group_id, parent_session_id, { kind: 'system', trigger: 1, ... })`. Going through `writeSessionMessage` ensures correct seq numbering and cross-mount cache semantics.
- **Effort to Fix:** Low
- **Impact if Ignored:** Low (impl-time confusion, possible direct insertMessage that breaks invariants)
- **Suggested resolution:** Spec the call shape explicitly. Note the recall pipeline is no-op for kind='system'.
- **Status:** Open

### [S2] C4 partial-UNIQUE constraint is tautological for the spec'd flow — real hazard is reconciler replay

- **Raised by:** Reviewer A
- **Finding:** C4 fires only on `thread_id IS NULL AND mg_id IS NOT NULL`; the dispatch flow's two paths can't produce that combination by construction. The constraint is real but has no path to violation under the spec'd flow. The actual hazard surfaced by cycle-1 M4 is reconciler replay safety (now M5 in cycle 2).
- **Effort to Fix:** Low
- **Impact if Ignored:** Low (spec hygiene)
- **Suggested resolution:** Keep C4 as documentation. Add a note that C4 is a defense-in-depth invariant, not an active failure mode of the spec'd flow. The active failure mode is reconciler replay (covered by M5).
- **Status:** Open

### [S3] `completion_nonce` adds prompt-injection surface area for marginal benefit

- **Raised by:** Reviewer A (#9), Reviewer B (LOW, partial — recommends hash-storage instead)
- **Finding:** Cycle-1 M5 added `completion_nonce` as defense in depth. But the nonce is transmitted to the child via the inbound prompt, putting it in plaintext inside the LLM's context. A prompt-injection attack from third-party content (e.g. a triage task that reads attacker-controlled text) could exfiltrate the nonce. The four other server-side checks (`child_session_id == source`, `target_agent_group_id == source.agent_group_id`, status guard, single-call `task_complete`) are server-side and don't depend on child-supplied data. The nonce adds attack surface without adding security. Reviewer B's alternative — hash-store the nonce — only mitigates DB exfil (not the threat model here).
- **Effort to Fix:** Low (drop the field) OR Low (hash-store)
- **Impact if Ignored:** Medium (security simplification; current state is over-engineered defense in depth that introduces a new vector)
- **Suggested resolution:** **Drop `completion_nonce`** from the schema, banner, and `task_complete` payload. Rely on the four server-side checks which already meet brief R14. If defense in depth is wanted, store the nonce in a host-only sidecar file mounted into the child's `/workspace` ROOT (not in chat context) — but this is over-engineering for a single-operator threat model.
- **Status:** Open (drop vs hash is a small judgment call; recommend drop)

### [S4] Vite dashboard memory budget excludes install-time footprint

- **Raised by:** Reviewer A
- **Finding:** Memory table claims ~15-20MB additional resident. Correct at runtime. But install footprint is much larger: `vite` + `esbuild` + `rollup` is ~150-200MB in `node_modules`. The migrate-v2 script and `pnpm install` time both hit this.
- **Effort to Fix:** Low (clarify scope)
- **Impact if Ignored:** Low (operational, not architectural)
- **Suggested resolution:** Clarify in the spec: "G5 budget is runtime resident only. Install footprint adds ~200MB to node_modules; vite is a `devDependency` so production install via `--prod` or container builds skip it."
- **Status:** Open

### [S5] `task_dependencies` same-parent-session enforcement is application-only, not DB-level

- **Raised by:** Reviewer A
- **Finding:** Cycle-1 S8 said add a CHECK that both rows share `parent_session_id`. Cycle-2 schema has `CHECK (task_id != blocked_on_task_id)` but dropped the same-parent constraint, with the comment that it's enforced in `applyDispatchTask`. Future code paths (test fixtures, migrations, scripts) can violate it silently.
- **Effort to Fix:** Low
- **Impact if Ignored:** Low (cross-tenant linkage possible from non-handler paths)
- **Suggested resolution:** Add a SQLite row trigger: `CREATE TRIGGER task_deps_same_parent BEFORE INSERT ON task_dependencies BEGIN SELECT RAISE(ABORT, 'cross-parent dependency forbidden') WHERE (SELECT parent_session_id FROM tasks WHERE task_id = NEW.task_id) != (SELECT parent_session_id FROM tasks WHERE task_id = NEW.blocked_on_task_id); END;`. better-sqlite3 honors row triggers.
- **Status:** Open

### [S6] Watchdog deadline-expiry path bypasses authorization → code-duplication risk

- **Raised by:** Reviewer A
- **Finding:** Watchdog step 3 says "Single completion code path — same as voluntary completion above but bypasses authorization (host-emitted)." But `applyTaskComplete` has the auth check at its top; calling it from the watchdog requires either fake-auth or duplicating the state-mutation/fan-out logic.
- **Effort to Fix:** Low
- **Impact if Ignored:** Medium (drift hazard or introduction of `skipAuth: true` parameter that's ripe for misuse)
- **Suggested resolution:** Refactor into `_completeTaskCore(task, status, summary, reason)` (does state mutation + fan-out, no auth) and `applyTaskComplete` (thin auth wrapper that calls _completeTaskCore). Watchdog calls _completeTaskCore directly. Standard "core + permission shim" pattern.
- **Status:** Open

### [S7] No rate limit on `dispatch_task` allows runaway pending-row creation

- **Raised by:** Reviewer A
- **Finding:** Concurrency caps gate running tasks; `dispatch_task` itself has no rate limit. An orchestrator (or prompt-injected one) can call `dispatch_task` 10,000 times. All 10,000 rows insert as `pending`. Each writes an outbound system action and is processed.
- **Effort to Fix:** Low
- **Impact if Ignored:** Medium (DoS surface; even a benign orchestrator with a retry bug pollutes the ledger)
- **Suggested resolution:** Add a per-orchestrator-session pending-task cap (default 50). `dispatch_task` returns an error to the orchestrator when the cap is full; no row inserted. Distinct from the running cap.
- **Status:** Open

### [S8] TanStack Query `setQueryData` from SSE can race with active queries on reconnect

- **Raised by:** Reviewer A (#17), Reviewer B (LOW backpressure, merged conceptually)
- **Finding:** On reconnect, dashboard re-queries authoritative state (per S7 from cycle 1). If an SSE event arrives mid-fetch, `setQueryData` from the event will be overwritten by the fetch's resolution. Plus: no spec for SSE backpressure — if a client's TCP buffer fills, server doesn't drop or close.
- **Effort to Fix:** Low
- **Impact if Ignored:** Medium (race results in momentarily-stale data on reconnect; backpressure unhandled at scale)
- **Suggested resolution:** "On reconnect: drain SSE events into a buffer until the authoritative `tasks` requery resolves, then replay buffer through `setQueryData`. Server-side: detect slow consumers via TCP write-buffer fullness and close the connection (client reconnects with backoff)."
- **Status:** Open

### [S9] No host-startup reconciler scan — restart latency 60-120s per in-flight task

- **Raised by:** Reviewer A
- **Finding:** Reconciler runs on the 60s host-sweep tick. After a host restart, in-flight tasks wait up to 60s before the first scan, more before resumption. G3 ("survives host restart") is partially met.
- **Effort to Fix:** Low
- **Impact if Ignored:** Medium (cosmetic — restart adds latency, doesn't break correctness)
- **Suggested resolution:** On `startHostSweep()`, immediately scan tasks with non-terminal `dispatch_state` regardless of age. Document the worst-case restart latency in R9.
- **Status:** Open

### [S10] SSE emitter filter on `tasks.child_session_id` not specified

- **Raised by:** Reviewer A
- **Finding:** Without a filter, the SSE emitter receives events for every active session, not just orchestrator-dispatched children. Easy to mis-implement; would dump non-task agent traffic to the dashboard.
- **Effort to Fix:** Low
- **Impact if Ignored:** Low (performance + scope creep)
- **Suggested resolution:** Spec: "SSE emitter checks `tasks.child_session_id = ?` for each outbound row's session and only pushes if a row exists. Lookup is a single indexed point query (`idx_tasks_child`)."
- **Status:** Open

### [S11] Banner-instruction reliability — child agents can forget to call `task_complete`

- **Raised by:** Reviewer A (#20), Reviewer B (LOW, merged)
- **Finding:** Design relies on the agent reading the banner instruction and calling `task_complete`. A forgetful or misbehaving child silently wedges fan-out. Host fallback only catches the *exit* case, not "still running indefinitely without calling `task_complete`." Tasks with no deadline run forever.
- **Effort to Fix:** Low
- **Impact if Ignored:** Medium (G4 reverse-signal promise fails for forgetful children)
- **Suggested resolution:** Make `deadline` required on `dispatch_task` (with a generous default, e.g. 4 hours) so deadline-expiry is universal. Plus: add a watchdog "no chat output for N minutes" detector that prompts the agent before deadline fires ("Are you still working? If complete, call task_complete now.").
- **Status:** Open
