## MUST-FIX Findings

### [M1] Cycle-2 dispatch flow reorder is a regression â€” placeholder thread_id violates C1 + wake-before-thread is a guaranteed cost leak

- **Raised by:** Reviewer A (#1 + #2), Reviewer C (#1) â€” merged
- **Finding:** Cycle-2 moved internal-only steps (resolveSession, writeSessionMessage, wakeContainer) BEFORE external steps (parent_post, createThread). This requires a placeholder `thread_id='pending-<task_id>'` on the child session row during the window. Two distinct problems:
  1. **C1 violation**: Per `Verified constraints` C1, "synthetic thread_ids fail Slack/Discord delivery when `messaging_group_id IS NOT NULL`." A `'pending-<task_id>'` value with `messaging_group_id` set IS a synthetic thread_id. The child can emit outbound rows during the window; those go to `delivery.ts:358-364` and either fail or get delivered to a literal-string thread_id. Constraint violated by construction.
  2. **Cost leak**: The reorder wakes the child container BEFORE the external thread exists. The child starts running, receives the prompt, calls Claude, and may emit chat outbound â€” all before the platform has anywhere to deliver them. This is a 30-second-ish cost-leak window on **every** dispatch, not just on crash. The cycle-2 reorder traded "platform orphans on crash" for "guaranteed cost leak in normal operation," which is strictly worse.
- **Effort to Fix:** Low (revert reorder)
- **Impact if Ignored:** High (G6 violation + per-dispatch cost waste + potential delivery to literal-string thread_ids)
- **Suggested resolution:** Revert the reorder. New order: (1) `status='dispatching'` (visible to reconciler), (2) parent_post (external), (3) `createThread` (external), (4) `resolveSession` with the REAL platform thread_id (no placeholder), (5) `writeSessionMessage`, (6) `wakeContainer`, (7) `status='dispatched'`. The orphan window narrows to "adapter call returned, DB write committing" (~ms â€” unavoidable without full transactional outbox; same residual G6 the design admits). Wake fires only AFTER thread exists, so child's first outbound has a delivery target.
- **Fact-check:** Verified against `src/db/sessions.ts` (session lookup via thread_id), `src/db/migrations/001-initial.ts:97` (`idx_sessions_lookup`), `src/delivery.ts:358-364` (synthetic thread_id failure path). C1 from cycle-2 design is unambiguous.
- **Status:** Open (revision applied inline â€” see cycle-3 design)

### [M2] `_completeTaskCore` status guard is too narrow â€” pending/dispatching deadline expiry silently fails

- **Raised by:** Reviewer C
- **Finding:** Reverse signal step 3a guards: `UPDATE tasks SET status, ... WHERE task_id = ? AND status IN ('dispatched','running')`. But the watchdog calls `_completeTaskCore` for:
  - Step 4 (running deadline expiry): tasks in `dispatching`/`dispatched`/`running` â€” partially covered, fails silently for `dispatching`.
  - Step 5 (pending deadline expiry): tasks in `pending` â€” not in guard, completely fails silently.
  - Cancellation propagation (`applyCancelDispatchedTask`): tasks in any non-terminal state â€” partially covered.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (deadline expiry on pending/dispatching tasks doesn't fire; dependency chains stay blocked)
- **Suggested resolution:** Refactor `_completeTaskCore` to accept an `allowed_source_states` parameter. Callers specify per use case:
  - Voluntary completion (`applyTaskComplete`): `['dispatched','running']`
  - Watchdog timeout: `['pending','dispatching','dispatched','running']`
  - Cancellation: any non-terminal: `['pending','dispatching','dispatched','running']`
  - Failed-prereq propagation: `['pending']` (only propagate to tasks not yet in flight; in-flight tasks finish on their own)
  Guard becomes `WHERE task_id = ? AND status IN (<param>)`. Abort side effects when the guarded update affects zero rows.
- **Fact-check:** Verified against design Â§ Reverse signal step 3a guard and Â§ Watchdog step 4/5 callers.
- **Status:** Open (revision applied inline)

### [M3] Dependency-propagated failures bypass parent notification, parent_delivery_state, and SSE fan-out

- **Raised by:** Reviewer C
- **Finding:** Step 3e says propagated failures "transition that task to 'failed' with failure_reason='blocked dependency...'" but doesn't say those propagated failures pass through the same parent system row write, parent_delivery_state update, or SSE event emission as voluntary completions. A downstream task can become terminal silently â€” orchestrator unaware, dashboard shows mismatched state.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (state desync; orchestrator and dashboard show inconsistent task states)
- **Suggested resolution:** Make every terminalization path call the same `_completeTaskCore` (including propagated failures). One transition implementation handles: status update, child subthread write (skip if no channel surface), parent system row write (with parent_delivery_state update), SSE event emission, dependency fan-out. Propagated failures recurse through `_completeTaskCore` â€” every transition goes through the canonical path.
- **Fact-check:** Verified against design Â§ Reverse signal step 3e (propagation logic) vs Â§ _completeTaskCore (canonical path).
- **Status:** Open (revision applied inline)

### [M4] Markdown XSS in dashboard via agent-authored summaries â€” same-origin script can use cookie

- **Raised by:** Reviewer C
- **Finding:** Dashboard renders agent-authored markdown (task summaries, transcripts, error reasons). Cookie auth (M11 cycle-2) is `HttpOnly`, which prevents JS reading the cookie value â€” but does NOT prevent same-origin `fetch` calls from automatically including the cookie. If the dashboard renders agent markdown with raw HTML enabled (or any sanitization gap), prompt-injected content can run inline `<script>` that calls `fetch('/api/tasks/X/cancel', {method: 'POST'})` â€” the browser sends the cookie automatically, the call succeeds.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (XSS allows compromised agent in any group to mutate task ledger via dashboard's same-origin context)
- **Suggested resolution:** Three layers:
  1. **Markdown sanitization with allowlist**: render via DOMPurify or rehype-sanitize with an explicit allowlist (no `<script>`, no `<iframe>`, no event handlers, no data: URLs, no javascript: URLs). Use a known-safe rehype/remark pipeline (no `dangerouslySetInnerHTML`).
  2. **Strict CSP**: response header `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'` (no `'unsafe-inline'` on script).
  3. **User-gesture confirmation** for dangerous mutating actions (cancel, retry, approve): require an active click within the last 5s, not auto-fire. Mitigates same-origin auto-mutation if sanitization fails.
- **Fact-check:** Verified against design Â§ Dashboard Auth (cookie auth via M11), and Mozilla MDN HttpOnly + same-origin policy docs.
- **Status:** Open (revision applied inline)

## SHOULD-FIX Findings

### [S1] `enabled_tools.json` is RW from container â€” security boundary should be documented explicitly (A3)

- **Suggested resolution:** Add note to M2 mounting section: "The allowlist file is RW from the container; the role re-check at applyDispatchTask step 3a and child_session_id/target_agent_group_id checks at task_complete/extend_deadline ARE the security boundary. The allowlist is hint-level."

### [S2] Setup-token endpoint CSRF gap (A4)

- **Suggested resolution:** Setup endpoint enforces same Origin/Host allowlist as mutating endpoints. Rate-limit 5 attempts/minute per IP. Token entry is via paste of file contents, not URL parameter.

### [S3] Recursive fail-propagation + transaction-scope incompatibility (A5)

- **Suggested resolution:** Split:
  - DB state mutation: BFS over `task_dependencies` in one SQLite transaction; mark every transitively-blocked task as `failed`. No external calls inside the transaction. Worklist queue (no recursion); max-depth 50.
  - Child subthread delivery: post-commit, only the ROOT failure posts a chat. Transitive failures are dashboard-only (avoids notification storm).
  - Parent system row + SSE: every transitively-failed task emits its own (orchestrator wants to see the propagation pattern in fan-out events).

### [S4] `extend_deadline` should block deadline shrinkage (A6)

- **Suggested resolution:** Handler validates `new_deadline > task.deadline + MIN_EXTENSION (5 min)`. Reject otherwise.

### [S5] Idempotency-key TTL not specified (B1)

- **Suggested resolution:** Idempotency keys expire after 7 days (matches Stripe's 24h-7d range). Daily cleanup pass deletes `tasks` rows where `status IN ('complete','failed','cancelled')` AND `completed_at < now - 7d` â€” but only for the idempotency-key check; archival policy is separate (S8).

### [S6] Idempotency-key fingerprint mismatch behavior (B2)

- **Suggested resolution:** Same `(parent_session_id, idempotency_key)` with different `target_group`/`title`/`prompt` returns the existing task_id without inserting (first-wins). Document this in the dispatch_task description so orchestrator-authors know retry semantics.

### [S7] Idle-child ping inserts a system message into agent context â€” document the trade-off (B3)

- **Suggested resolution:** Add note to watchdog Â§ step 6: "The idle-ping is a kind='system' inbound, which the agent reads in its next turn. This consumes tokens and may interrupt mid-task reasoning. The 30-min threshold is set conservatively to bound how often this fires; future revision could add a dedicated heartbeat MCP tool the agent calls voluntarily, decoupling liveness signals from the conversation."

### [S8] No archival policy for terminal tasks (B4)

- **Suggested resolution:** Daily watchdog pass: tasks where `status IN ('complete','failed','cancelled')` AND `completed_at < now - 30d` AND `parent_delivery_state IN ('delivered','orphaned')` move to a `tasks_archive` table (same schema, no indexes). Reduces working-set scan size as the install ages.
