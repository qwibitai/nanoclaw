## MUST-FIX Findings

### [M1] State-machine dispatch is not crash-idempotent — `createThread` and `writeSessionMessage` are non-idempotent operations called before persisting state, and `child_thread_id` is referenced but absent from schema

- **Raised by:** A6, C1, A2 (merged)
- **Finding:** §Dispatch flow step 5a writes the parent message and calls `adapter.createThread()` BEFORE persisting `dispatch_state='thread_created'`. A host crash between the post and the DB update leaves the platform thread created but the DB unaware → reconciler retries → duplicate Slack/Discord threads. Compounding: §step 5a writes `child_thread_id=<real id>` but the `tasks` schema has no such column. `writeSessionMessage` (step 5c) re-injects the dispatch banner; design doesn't pin a deterministic message id, so retry produces duplicate banners.
- **Effort:** Medium · **Impact:** High (duplicate platform side effects on every host crash)
- **Resolution:**
  1. Add `child_thread_id TEXT` to `tasks` schema (or denormalize through `child_session_id → sessions.thread_id`; pick one).
  2. Reorder step 5a: persist intent (e.g. `dispatch_state='creating_thread'`) → call `createThread` → persist `child_thread_id` + `dispatch_state='thread_created'` in a single UPDATE. On reconciler retry, if `child_thread_id` is already set, skip re-creation.
  3. Pin dispatch-banner message id to `dispatch-banner-${task_id}` so `writeSessionMessage`'s `INSERT OR IGNORE` on PK protects against duplicates.
- **Fact-check:** Verified. `src/db/migrations/024-sessions-channel-root-unique.ts` defines the partial-UNIQUE invariant; existing `session-manager.ts:writeSessionMessage` uses primary-key insert.
- **Status:** Open

### [M2] Per-session MCP-tool mounting / unmounting doesn't exist in NanoClaw's container architecture

- **Raised by:** A1, C2 (merged)
- **Finding:** §`task_complete` MCP tool says "Auto-mounted into any container whose currently-running session matches a `tasks.child_session_id` row in `dispatched` or `running` state. Auto-unmounted after the first successful call (one-shot)." But `container/agent-runner/src/mcp-tools/index.ts` is a barrel that imports every tool unconditionally at boot; `loadConfig()` runs once, then each module's `registerTools` fires from top-level. There is no per-session mounting hook in this architecture (verified against `capabilities.ts:69`, `scheduling.ts`). The same architectural fiction applies to `dispatch_task` ("Mounted only when the agent group has the `orchestrator` role") and `cancel_task`. Codex C2 also notes that even if the mounting existed, `dispatched | running` is the wrong condition because the child wakes in `wake_sent`.
- **Effort:** Medium · **Impact:** High (gives reviewers and implementers a false guarantee)
- **Resolution:** Drop the "auto-mounted/unmounted" framing entirely. Make all three MCP tools (`dispatch_task`, `task_complete`, `cancel_task`) **always present in every container**; correctness is purely the host-side authorization check (orchestrator-role validation in `applyDispatchTask`; session+nonce match in `applyTaskComplete`). Document the host as the **sole** authority. Tool descriptions should set agent expectations ("only call this when you're a dispatched child" / "only call if you have the orchestrator role"), but those are guidance, not gates.
- **Fact-check:** Verified against `container/agent-runner/src/mcp-tools/index.ts` and `capabilities.ts`.
- **Status:** Open

### [M3] Completion finalization is not atomic — host crash between status update and downstream writes leaves orchestrator and dependents stranded

- **Raised by:** Codex C3
- **Finding:** §Reverse signal step 2b updates `tasks.status='complete'/'failed'` BEFORE step 2c (post to child subthread), 2d (write to parent system), 2e (fan out to dependents), 2f (wakeContainer). A crash after 2b but before 2d-2f leaves a terminal task that never informs the orchestrator and never unblocks downstream tasks.
- **Effort:** Medium · **Impact:** High (silent orchestration failure)
- **Resolution:** Split completion into authorize → finalize state machine. Add intermediate columns: `parent_notified_at TEXT`, `child_summary_posted_at TEXT`, `dependents_released_at TEXT`. Reconciler resumes incomplete finalization on the next host-sweep tick. Each side effect is idempotent (PK-protected child summary message, watchdog-guarded parent notification, `task_dependencies` join table is naturally idempotent under SELECT-then-trigger).
- **Fact-check:** Verified against `design.md` §Reverse signal step 2.
- **Status:** Open

### [M4] Watchdog deadline-expiry path cannot satisfy the authorization check it relies on

- **Raised by:** Codex C4
- **Finding:** §Watchdog responsibility 2 says deadline expiry "emits a synthetic `task_complete` (status=`failed`, failure_reason=`timeout`) which then flows through `applyTaskComplete`." But §Reverse signal step 2a's authorization requires source `child_session_id` + `target_agent_group_id` + `completion_nonce`. A host-generated timeout has no child outbound row source and shouldn't possess the child nonce. As written, timeout completion either fails authorization or requires an undocumented privileged bypass.
- **Effort:** Low · **Impact:** High (deadline expiry is broken)
- **Resolution:** Factor into `authorizeChildCompletion()` (called only from `applyTaskComplete` for child outbound rows) and `finalizeTaskCompletion()` (called by both, internal). Watchdog calls `finalizeTaskCompletion()` directly with explicit host authority, no synthetic-row charade.
- **Fact-check:** Verified against §Reverse signal step 2a + §Watchdog responsibility 2.
- **Status:** Open

### [M5] Reverse-signal "post into child subthread via delivery path" requires host to write to outbound.db — must explicitly invoke the existing `writeOutboundDirect` bypass to preserve C5

- **Raised by:** A19
- **Finding:** §Reverse signal step 2c says "post the summary as a chat message into the child's outbound, going through the normal delivery path (`delivery.ts`)." NanoClaw's two-DB invariant (C5, CLAUDE.md): host writes inbound only; container writes outbound only. The existing escape valve is `session-manager.ts:writeOutboundDirect` (used by router for ack messages). The design re-introduces M8's resolution risk by saying "normal delivery path" without naming the bypass — a future reviewer will flag this as a C5 violation.
- **Effort:** Low · **Impact:** High (will cycle-2-flag)
- **Resolution:** Reference `writeOutboundDirect` (`session-manager.ts`) as the mechanism. Update C5 in the constraint table to explicitly mention this controlled bypass for host-initiated outbound writes (it already exists for router acks; M8's two-write fix is the second use case). Document why this isn't a new invariant violation.
- **Fact-check:** Verified against `src/session-manager.ts` and `src/router.ts:36`.
- **Status:** Open

### [M6] Dashboard auth contradictions — `ui-design.md` §7 says "no external auth, you're on localhost"; design.md says token + Origin + Host

- **Raised by:** A8, C6 (merged)
- **Finding:** Cycle-1 M3 hardened design.md to a 3-layer auth (token + Origin + Host allowlist). But `ui-design.md` §7 Implementation Posture still says "No external auth — owner check is 'you're on localhost,' same posture as the rest of NanoClaw." Whichever doc the implementer reads last wins. Compounding (Codex C6): even within design.md, "read endpoints can be unauthenticated for simplicity" undercuts the token guarantee — read surfaces leak transcripts, prompts, and embedded `completion_nonce` if M16 isn't fixed. `ui-design.md` §7 also still references `undici`-based HTTP server (cycle-1 M1's fix didn't propagate).
- **Effort:** Low · **Impact:** High (auth is contradicted in ship docs)
- **Resolution:** Update `ui-design.md` §7 to mirror design.md's 3-layer auth + Node `http` server choice. Tighten design.md: token required on every API request including reads; only static-asset serving is unauthenticated. Origin/Host checks always apply.
- **Fact-check:** Verified against `ui-design.md` §7 and `design.md` §Dashboard.
- **Status:** Open

### [M7] Native `EventSource` cannot send `Authorization: Bearer <token>` header — SSE auth model is incompatible with the API surface

- **Raised by:** Codex C7
- **Finding:** §Dashboard auth requires `Authorization: Bearer <token>` on every API request; §Real-time updates uses a single browser `EventSource` for SSE. The native `EventSource` API has no header API, so the live dashboard either cannot connect or requires an undocumented exception.
- **Effort:** Low · **Impact:** High (SSE breaks under cycle-1 M3)
- **Resolution:** Two options, both established practice (record judgment under `auto_judgments`):
  - **(a)** Use a fetch-based SSE client (e.g. `@microsoft/fetch-event-source` or a thin wrapper over `fetch` with `ReadableStream`) that can attach headers. Modern, supported, ~1KB.
  - **(b)** Establish a session cookie (HttpOnly, SameSite=Strict, scoped to `127.0.0.1:7457`) at token-validation time; SSE endpoint accepts the cookie. EventSource sends cookies natively.
  Option (b) is closer to OWASP's CSRF cheat sheet recommendation for cookie + Origin-double-defense. Pick (b) unless there's reason not to.
- **Fact-check:** Verified — MDN documents `EventSource` does not accept headers other than via the constructor's `withCredentials` flag (cookies only).
- **Status:** Open

### [M8] Reverse-signal Write 2 wire format is unspecified, but `formatSystemMessage` requires fixed `action` / `status` / `result` keys

- **Raised by:** A3
- **Finding:** §Reverse signal step 2d says "direct-write a kind='system' inbound row carrying the result summary + task metadata." `container/agent-runner/src/formatter.ts:307-317` renders `kind='system'` as `<system_response action="X" status="Y">RESULT_JSON</system_response>` — `content.action`, `content.status`, `content.result` are mandatory. Without a pinned shape, host writer and orchestrator-prompt author will diverge. `result_summary` ≤16k chars markdown embedded in JSON also risks size/escaping issues.
- **Effort:** Low · **Impact:** High (builder ambiguity → broken integration)
- **Resolution:** Pin the wire format. Add subsection "Wire format for parent system write": `{ action: 'task_complete', status: 'complete' | 'failed', result: { task_id, target_agent_group, summary_markdown, failure_reason?, completed_at, dependents_unblocked: string[] } }`. Note that the orchestrator's CLAUDE.md must instruct the agent to interpret `<system_response>` elements with `action='task_complete'`.
- **Fact-check:** Verified against `container/agent-runner/src/formatter.ts:307-317`.
- **Status:** Open

### [M9] Concurrency cap algorithm is missing — recursive fan-out from §Reverse signal step 2e silently violates caps; pending tasks have no specified promotion path

- **Raised by:** A9
- **Finding:** §Concurrency caps section states the caps (6 per orchestrator, 3 per target group) but the dispatch flow has no cap-check step, watchdog has no "promote pending → dispatched when slot frees" responsibility, and §Reverse signal step 2e fan-out triggers dispatch recursively without checking caps.
- **Effort:** Medium · **Impact:** High (R5 mitigation does not exist as written)
- **Resolution:**
  1. Add explicit cap check in §Dispatch flow before step 5a: `SELECT COUNT(*) FROM tasks WHERE parent_session_id = ? AND status IN ('dispatched','running')` ≤ 6, AND `... WHERE target_agent_group_id = ? AND status IN ('dispatched','running')` ≤ 3. If either violates, status stays `pending`.
  2. Add 4th watchdog responsibility: "Promote pending → dispatched when caps allow." FIFO by `spawned_at`; for each pending task whose dependencies are complete AND caps now allow, run §Dispatch flow step 5a-5d.
  3. Update §Reverse signal step 2e to enqueue (not directly dispatch) — dependents transition `pending → eligible` and watchdog promotes per cap budget.
- **Fact-check:** Verified against §Concurrency caps + §Risks R5.
- **Status:** Open

### [M10] Non-thread-channel fallback path collides with `agent-shared` session resolution — silently coalesces all dispatched tasks into one shared session

- **Raised by:** A10
- **Finding:** §Phasing fallback says "messaging_group_id = NULL on the child session row, thread_id = task-<id>." Fine in isolation. But `session-manager.ts:262-266` forces `messaging_group_id=NULL` for `agent-shared` mode sessions. If the target agent group has any agent-shared wirings, `findSessionByAgentGroup(target_agent_group_id)` returns the shared session before considering whether to create a new fallback session — silently fanning all dispatched non-thread tasks into ONE session, exactly the failure mode dispatch is designed to avoid.
- **Effort:** Medium · **Impact:** High (per-task isolation lost on the fallback path)
- **Resolution:** Orchestrator-dispatch must NOT use the standard `resolveSession` for the fallback path — it must call `createSession()` directly with explicit `session_mode='per-thread'` and bypass agent-shared coalescence. Add a new helper `createDispatchedSession(target_agent_group_id, mg_id, thread_id)` in `src/modules/orchestrator-dispatch/` that always creates a fresh row.
- **Fact-check:** Verified against `src/session-manager.ts:262-266`.
- **Status:** Open

### [M11] `task_progress` system action and `original_deadline` column are referenced but unspecified

- **Raised by:** A13
- **Finding:** §Watchdog deadline-extension says "child can extend its deadline by emitting a `task_progress` system action with a new `deadline` value... Bounded by a config max (default: 4× the original deadline)." But: (a) `task_progress` is not in any MCP tool block, not in the delivery action registry, never specified; (b) the dashboard SSE event types include `task.progress` (different name, different concept); (c) `tasks` table has no `original_deadline` column to compute the 4× bound.
- **Effort:** Medium · **Impact:** High (deadline extension impossible as written)
- **Resolution:** Spec the `task_progress` MCP tool (input: `task_id`, `completion_nonce`, optional `deadline_extension_seconds`, optional `progress_message`). Spec `applyTaskProgress` delivery handler. Add `original_deadline TEXT` column on `tasks` (set at insert, never mutated). Rename dashboard SSE event to `task.heartbeat` to avoid namespace collision with the system action, OR keep both with a clear naming distinction documented.
- **Fact-check:** Verified against §Watchdog deadline-extension + §Dashboard SSE event types.
- **Status:** Open

### [M12] `cancel_task` references nonexistent `scheduling.ts:174` and an undefined "terminate signal" mechanism — LLM-turn interruption is unsolved

- **Raised by:** A11
- **Finding:** §`dispatch_task` MCP tool last paragraph says cancellation "mirrors existing scheduled-task cancellation (`scheduling.ts:174`)." That file path doesn't exist (`src/modules/scheduling/db.ts` has `cancelTask`; not at line 174). The "terminate signal into the child's session" is novel — NanoClaw's existing model is "container polls inbound, finishes Claude turn, processes batch on next wake"; a `cancel_task` arriving mid-turn cannot stop the in-flight LLM call. `killContainer` (host-side) is SIGKILL — loses any in-flight work.
- **Effort:** Medium · **Impact:** High (cancellation, lifted to MUST in cycle 1 / S9, doesn't work as written)
- **Resolution:** Define cancellation as **best-effort soft + watchdog hard kill** (record judgment):
  - Host writes `cancel-${task_id}` marker into child's `inbound.db` on cancel + immediate wake.
  - Child checks marker on next poll, exits gracefully if it's between turns.
  - If marker not consumed within 2-minute grace period (configurable), watchdog `killContainer`s the child.
  - Update file reference to `src/modules/scheduling/db.ts:cancelTask`.
- **Fact-check:** Verified against `src/modules/scheduling/db.ts` (no line 174 cancel) + `src/container-runner.ts:killContainer`.
- **Status:** Open

### [M13] `idempotency_key` is nullable; `UNIQUE(parent_session_id, idempotency_key)` doesn't enforce dedup on NULL (SQLite NULL semantics)

- **Raised by:** A4
- **Finding:** `tasks` schema declares `idempotency_key TEXT,` (nullable) and `UNIQUE(parent_session_id, idempotency_key)`. SQLite treats NULLs as distinct in UNIQUE — the dedup contract relies on UNIQUE actually firing. dispatch_task tool says `idempotency_key` is `required: true`, but the column allows NULL. A future delivery handler, migration backfill, or test fixture inserting NULL bypasses the dedup contract.
- **Effort:** Trivial · **Impact:** High (silent integrity failure)
- **Resolution:** `idempotency_key TEXT NOT NULL` matches the MCP tool's `required: true`. Application validates non-NULL before insert anyway; the schema should match.
- **Fact-check:** Verified — SQLite docs explicitly state UNIQUE treats NULLs as distinct.
- **Status:** Open

### [M14] `task_dependencies` lacks DB-level enforcement of "both rows share `parent_session_id`" — cross-session dependency chains silently allowed

- **Raised by:** A5
- **Finding:** Schema declares `task_id REFERENCES tasks(task_id)` and `blocked_on_task_id REFERENCES tasks(task_id)` with no enforcement that both reference tasks with the same `parent_session_id`. Comment says "enforced at insert time in application code" — fragile. A buggy or malicious orchestrator could chain a task in session A as `blocked_on` a task in session B, fanning out to wake an unrelated parent. Cycle-1 M5 closed cross-session forgery via `completion_nonce`; this leaves another cross-session vector.
- **Effort:** Low · **Impact:** High (security gap analogous to M5 pre-fix)
- **Resolution:** Add `parent_session_id TEXT NOT NULL` to `task_dependencies` (denormalized but verifiable). Add a CHECK constraint or SQLite trigger that both `task_id` and `blocked_on_task_id` reference rows with this same `parent_session_id`.
- **Fact-check:** Verified against §`task_dependencies` schema.
- **Status:** Open

### [M15] Reconciler stuck-detection uses `spawned_at` but `spawned_at` is creation time — false-positives tasks legitimately held by dependencies or caps

- **Raised by:** Codex C5
- **Finding:** §Watchdog responsibility 1 retries dispatches when `spawned_at < now - 60s`. But a task can sit in `pending` behind dependencies or caps for minutes/hours, then legitimately enter `session_created` or `wake_sent`; on the next sweep, `spawned_at` is already old → classified as stale → false retry.
- **Effort:** Low · **Impact:** High (false retries cause duplicate side effects per M1)
- **Resolution:** Add `dispatch_state_updated_at TEXT` and `dispatch_retry_count INTEGER DEFAULT 0` columns. Update `dispatch_state_updated_at` on every state transition. Stuck detection becomes: `dispatch_state IN ('thread_created','session_created','prompt_injected','wake_sent') AND dispatch_state_updated_at < now - 60s`.
- **Fact-check:** Verified against §Watchdog responsibility 1.
- **Status:** Open

### [M16] `completion_nonce` is delivered inside the dispatch banner (chat-kind) — observable in chat history and extractable via prompt injection

- **Raised by:** A15, A20 (merged)
- **Finding:** Cycle-1 M5 added `completion_nonce` as authorization-grade secret. But §Dispatch flow step 5c says the banner is written via `writeSessionMessage(child)` as a `kind='chat'` message — the nonce ends up in conversation history, formatted into Claude's input on every wake, and (in the Slack/Discord subthread case) **delivered to the platform**. Any agent reading the thread, any human with channel access, or any prompt-injection successfully extracting "your initial instructions" can grab the nonce.
- **Effort:** Medium · **Impact:** High (M5's security guarantee is undermined)
- **Resolution:** Two options:
  - **(a)** Inject the nonce as a `kind='system'` row instead (formatter strips system rows from chat-platform delivery). Agent still sees it via system context, but it never reaches Slack/Discord.
  - **(b)** Stash the nonce in `/workspace/.task-context.json` (host-written, container-readable file). Banner instructs the agent: "When you call `task_complete`, read your task context from `/workspace/.task-context.json` and include the `completion_nonce` field."
  Option (b) is cleaner: the secret never enters the conversation data plane at all. Pick (b) unless `/workspace/` mount semantics block it.
- **Fact-check:** Verified against §Dispatch flow step 5c (banner is `kind='chat'`).
- **Status:** Open
