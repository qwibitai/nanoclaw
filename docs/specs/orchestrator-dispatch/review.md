### [M3] Dashboard auth — "you're on localhost" insufficient against browser-based local attacks

- **Raised by:** Reviewer A, Reviewer C (merged)
- **Finding:** § Dashboard says "Owner-only auth: localhost-bound + a single owner check"; § 7 says "No external auth — owner check is 'you're on localhost', same posture as the rest of NanoClaw." These contradict each other. More importantly, the dashboard exposes mutating endpoints (steer task, retry, approve, cancel) — any malicious browser tab visiting a page that probes 127.0.0.1:7457 can issue cross-origin POSTs. DNS rebinding attacks defeat the localhost-only assumption entirely. The rest of NanoClaw checks `user_roles.role = 'owner'` (`src/modules/permissions/access.ts`) — the dashboard's claim of matching that posture is wrong.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (security blocking)
- **Why it blocks:** Owner-mediated mutating control surface accessible by any local browser context = drive-by task hijack from any visited site. Not a hypothetical — DNS rebinding is well-known.
- **Suggested resolution:** Three layers: (1) random bearer token generated at host startup, printed once to setup logs and required on every API request; (2) `Origin` header allowlist on mutating endpoints (only `http://127.0.0.1:7457` accepted); (3) `Host:` header allowlist with literal `127.0.0.1:7457` to defeat DNS rebinding. Bind to `127.0.0.1`, not `0.0.0.0`. Read endpoints can be unauthenticated for simplicity.
- **Fact-check:** Verified against `src/modules/permissions/access.ts` (project's owner-check pattern) and the OWASP DNS rebinding documentation.
- **Status:** Open

### [M4] Synthetic `task-<id>` thread_id ambiguous about `messaging_group_id` — risks colliding with migration 024 partial-UNIQUE invariant

- **Raised by:** Reviewer A
- **Finding:** § Phasing's fallback path: "messaging_group_id = NULL on the child session row, thread_id = task-<id>." Migration 024 (`src/db/migrations/024-sessions-channel-root-unique.ts`) enforces `UNIQUE(agent_group_id, messaging_group_id) WHERE thread_id IS NULL AND messaging_group_id IS NOT NULL`. The dispatch flow step 5a says `resolveSession(target_agent_group_id, mg_id, real_thread_id, 'per-thread')` but never says which `mg_id` is "the orchestrator's mg" vs NULL when the createThread call fails partway through. Two distinct insertion paths with race-prone collision risk.
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (silent failure under concurrent dispatch)
- **Why it blocks:** Second concurrent dispatch in the same MG could fail the UNIQUE index and surface as a confusing "task could not start" error.
- **Suggested resolution:** Spell out the exact `(messaging_group_id, thread_id)` tuple per branch:
  - **Adapter supports threads, createThread succeeds**: `mg_id = orchestrator_mg`, `thread_id = real_platform_thread_id`. Never NULL on either field. No collision risk.
  - **Adapter supports threads, createThread fails after parent message posted**: explicitly fail the dispatch (don't fall back to internal-only mid-flow); leave the parent message visible with an error indicator, surface to orchestrator.
  - **Adapter doesn't support threads (Telegram, iMessage, email)**: `mg_id = NULL`, `thread_id = task-<id>`. Internal-only; never collides with platform sessions.
  Add a unit test covering the "second concurrent dispatch in same MG" case to lock the partial-UNIQUE behavior.
- **Fact-check:** Verified against `src/db/migrations/024-sessions-channel-root-unique.ts`.
- **Status:** Open

### [M5] `task_complete` handler trusts `task_id` alone — no authorization check

- **Raised by:** Reviewer C
- **Finding:** § Reverse signal step 2a: "UPDATE tasks SET status, completed_at, result_summary, failure_reason WHERE task_id = ?" The handler does not validate that the outbound system action came from the task's recorded `child_session_id` and `target_agent_group_id`. Any agent that learns or guesses a task_id can mark another task complete, fail another task, or unblock dependents prematurely.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (security/integrity)
- **Why it blocks:** Trust boundary not enforced. Compromised agent or prompt-injected agent in any agent group could corrupt the task ledger.
- **Suggested resolution:** Bind completion to source identity in the handler: `UPDATE tasks SET ... WHERE task_id = ? AND child_session_id = ? AND target_agent_group_id = ? AND status IN ('dispatched', 'running')`. The `child_session_id` is implicit in the outbound's source session (host knows which session emitted the system action). Optionally add a per-task `completion_nonce` generated at dispatch and required in the `task_complete` payload as defense in depth.
- **Fact-check:** Verified against `design.md` § Reverse signal — handler spec confirms task_id-only update.
- **Status:** Open

### [M6] Child-side `task_complete` MCP tool not specified — reverse-signal protocol depends on an undefined component

- **Raised by:** Reviewer C
- **Finding:** § Reverse signal describes the flow ("Child container writes outbound system action `task_complete` with task_id, status, summary, failure_reason"), but the `dispatch_task MCP tool` section only specifies the dispatch tool. There's no spec for the child-side tool that emits `task_complete`, nor for how it's mounted into a child session, nor for whether it's automatically present or requires explicit instruction in the child's CLAUDE.md.
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (functional gap)
- **Why it blocks:** Without spec, child agents have no defined way to signal completion. If the child finishes normally, crashes, or produces a final answer without the system action, the task remains `dispatched`/`running` and dependents never unblock.
- **Suggested resolution:** Add a specification block for the `task_complete` MCP tool: input schema (`task_id`, `status` ∈ `complete | failed`, `summary` (markdown), `failure_reason?`), mounting rules (auto-mounted into any child session that has a `tasks.child_session_id` matching its session, auto-unmounted after the first call), and host-side fallback detection — when the child container exits without emitting `task_complete`, host-sweep should mark the task `failed` with `failure_reason='child exited without completion signal'`. The child's first inbound (the task prompt) should include explicit instruction: "When you finish this task, call `task_complete` with status=complete and a summary, or status=failed with a reason."
- **Fact-check:** Verified against `design.md` — only `dispatch_task` is specified in the MCP tool section.
- **Status:** Open

### [M7] Dispatch flow not retry-safe — multi-step procedure has no idempotency or resumable state machine

- **Raised by:** Reviewer C, Reviewer B (merged with idempotency drift)
- **Finding:** Dispatch flow is 8 steps: insert task row, post Slack parent message, call `adapter.createThread`, call `resolveSession` (creates child session), `writeSessionMessage` (inject prompt), update tasks row, `wakeContainer`. A crash or adapter error between any two steps can leave: duplicate platform threads, child sessions without task rows, pending tasks that already launched, or tasks that can't be safely retried because the orchestrator might re-dispatch the same logical task. The design has no idempotency key on `dispatch_task`, no recorded "where did this dispatch get to" state, and no reconciler.
- **Effort to Fix:** Medium
- **Impact if Ignored:** High (operational correctness under failure)
- **Why it blocks:** Real production failure mode (LLM retry, container restart mid-tool-call, network blip during adapter call). A multi-step procedure that's not resumable produces orphaned platform state that requires manual cleanup. Inngest's "queues guarantee delivery, durable execution guarantees completion" taxonomy is exactly what this design needs and currently lacks.
- **Suggested resolution:** Two changes: (1) add `idempotency_key TEXT UNIQUE` to the `tasks` schema, accepted as an optional argument to `dispatch_task`; orchestrator generates one per logical task; second call with the same key returns the existing task_id. (2) Add a `dispatch_state` column with values `pending | thread_created | session_created | prompt_injected | wake_sent | dispatched`. Each step persists the new state before continuing. A reconciler in host-sweep detects `tasks` rows stuck in non-terminal dispatch_state for >60s and either resumes from the last persisted step (idempotent) or marks the task `failed` with an explicit reason.
- **Fact-check:** Verified against `design.md` Dispatch flow and § Data model (no idempotency key, no dispatch state column).
- **Status:** Open

### [M8] Reverse signal must land in BOTH child subthread AND parent — current spec only writes to parent's inbound

- **Raised by:** Reviewer A
- **Finding:** § Reverse signal step 2c: "Direct-write a kind='system' inbound row into the orchestrator's session (parent_session_id) carrying the result summary. NOT a channel_type='agent' message." Correct in isolation. But this kind='system' inbound row does NOT route to chat (verified: `src/modules/memory/recall-injection.ts:613` and other system writes are for the agent to read, not for delivery). The orchestrator can post a user-visible "task X complete" message in its OWN thread after processing — but the UI/UX contract says "each task = its own chattable thread." If the completion summary doesn't land in the **child's subthread**, the subthread sits empty at completion, which is exactly when the owner is most likely to scroll back to read what happened.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (UX contract violation, silent)
- **Why it blocks:** Promised behavior ("you can chat with each task in its own thread, see updates there") is not delivered for the most important moment (completion).
- **Suggested resolution:** Two writes per `task_complete`:
  - (1) A user-visible chat message posted to the **child session's** outbound, routed through the normal delivery path to the child's Slack/Discord subthread. Body = the summary text. This is what the owner reads.
  - (2) A `kind='system'` inbound write to the **parent session** (the existing spec's behavior). This wakes the orchestrator so dependent tasks can fan out.
  Both writes share the same `task_complete` system action; the host's `applyTaskComplete` handler emits both.
- **Fact-check:** Verified against `src/modules/memory/recall-injection.ts:613` (system rows aren't delivered) and `src/delivery.ts:286-296` (delivery routing).
- **Status:** Open

### [M9] chokidar/fsevents transport choice is unreliable on macOS — silently misses completion events

- **Raised by:** Reviewer A (×2 findings), Reviewer B (HIGH drift), Reviewer C (merged)
- **Finding:** § Dashboard memory budget table mentions chokidar watching `outbound.db`; § Real-time says SSE; the design also uses "WebSocket" in one paragraph. Cross-cutting issues: (a) FSEvents on macOS does not reliably fire on SQLite WAL writes — documented at `anthropics/claude-code#16523`. The host runs on multiple platforms and Dave develops on macOS. (b) For 22 active sessions × WAL write frequency, chokidar fires constantly; the 15MB budget ignores per-fire query CPU cost. (c) The spec is internally inconsistent on SSE vs WebSocket — different reconnection, backpressure, and proxying behavior.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (silent data loss on Dave's actual platform)
- **Why it blocks:** Dashboard claims live updates; on macOS those updates can silently miss `task_complete`. The "is this thing alive" promise fails exactly where it's most valuable.
- **Suggested resolution:** Drop chokidar. Drop file-watching entirely. The host process is already the sole reader of `outbound.db` (via `delivery.ts` polling); piggyback an SSE event emitter on that same code path — when the host reads a new `messages_out` row that's relevant to a watched task, push the event over SSE. Single read code path, no second observer of the file system, no platform divergence. Pick **SSE** consistently in the spec; remove all WebSocket references. `tanstack/query`'s SSE-into-cache pattern is the React side (see also S6).
- **Fact-check:** Verified against `anthropics/claude-code#16523` (T2 source from Reviewer B), `container/agent-runner/src/db/connection.ts` ("journal_mode=DELETE is load-bearing for cross-mount visibility"), and `src/delivery.ts` polling pattern.
- **Status:** Open

### [M10] `createThread` adapter capability must be optional + capability-flag, not a required interface method

- **Raised by:** Reviewer A
- **Finding:** § Verified constraint C2 acknowledges the gap; § Implementation Order treats it as two adapter additions. But the `ChannelAdapter` interface already follows an optional-method convention for capabilities (`setTyping?`, `deleteMessage?`, `subscribe?` are all optional in `src/channels/adapter.ts`). If `createThread` is added as a required method, every other adapter living on the channels branch (Telegram, WhatsApp, Teams, Linear, GitHub, iMessage, Webex, Resend, Matrix, Google Chat, WhatsApp Cloud — 11 adapters) breaks compilation on its next merge.
- **Effort to Fix:** Low
- **Impact if Ignored:** High (build break across the entire channels branch on next merge)
- **Why it blocks:** The change is a public contract change; making it required is a multi-adapter migration burden that the design doesn't acknowledge.
- **Suggested resolution:** Spec the method as optional (`createThread?: (...) => Promise<...>`) with a `supportsCreateThread: boolean` capability flag (parallel to the existing `supportsThreads` capability). Host calls `adapter.createThread?.()` and falls back to internal-only mode when missing. Mirror the existing capability-flag pattern.
- **Fact-check:** Verified against `src/channels/adapter.ts` (existing optional method convention) and CLAUDE.md (channels branch adapter list).
- **Status:** Open
