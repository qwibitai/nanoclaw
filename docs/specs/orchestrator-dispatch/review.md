## MUST-FIX Findings

### [M2-A] Fallback path doesn't actually isolate dispatched sessions — `findSessionByAgentGroup` ignores `thread_id`, allowing a2a callers to land in dispatched sessions

- **Raised by:** A1, C3 (merged)
- **Finding:** `createDispatchedSession` creates `messaging_group_id=NULL, thread_id=task-<id>`. But `findSessionByAgentGroup(agentGroupId)` (`src/db/sessions.ts:62`) selects by `agent_group_id AND messaging_group_id IS NULL` ordered `created_at DESC LIMIT 1` — `thread_id` is ignored. Any subsequent caller hitting `resolveSession(targetAgentGroupId, null, ?, 'agent-shared')` (including `agent-route.ts:242` whenever caller's `effectiveMgId` is null) lands in the most-recently-created dispatched session and contaminates its inbound. Codex C3 adds: even ignoring contamination, `createDispatchedSession`'s "naturally PK-protected" claim is unsubstantiated — a host crash between session insert and `tasks.child_session_id` UPDATE produces duplicate sessions on retry.
- **Effort:** Medium · **Impact:** High (M10 doesn't actually solve the isolation problem; cross-traffic contamination of orchestrated work)
- **Resolution:** Add `sessions.dispatch_task_id TEXT UNIQUE REFERENCES tasks(task_id)` column. `findSessionByAgentGroup` adds `AND dispatch_task_id IS NULL` to its WHERE. `createDispatchedSession(task_id, ...)` becomes insert-or-select on `dispatch_task_id`, idempotent on retry. `tasks.child_session_id` and `sessions.dispatch_task_id` are bidirectional FKs, set in the same transaction.
- **Fact-check:** Verified against `src/db/sessions.ts:62` and `src/modules/agent-to-agent/agent-route.ts:242`.
- **Status:** Open

### [M2-B] `cancelled_at` column referenced by cancel watchdog but not defined in schema — hard-kill timer can't fire

- **Raised by:** A2, C8 (merged)
- **Finding:** §`cancel_task` step 5 says "if `tasks.status='cancelled'` AND `dispatch_state IN ('dispatched','wake_sent')` AND `cancelled_at < now - 120s`" but `tasks` schema has no `cancelled_at` column. M12 cancellation as written is soft-only — hard-kill never triggers.
- **Effort:** Trivial · **Impact:** High (M12 broken)
- **Resolution:** Add `cancelled_at TEXT` to `tasks` schema. `applyCancelTask` step 2 sets it alongside `status='cancelled'`. Update watchdog responsibility to use the column.
- **Fact-check:** Verified against `tasks` schema lines 89-123 in revised design.md.
- **Status:** Open

### [M2-C] `dispatch_state` advances through side-effect steps but `status` is never set to `'dispatched'` — concurrency cap counts skip in-flight dispatching tasks

- **Raised by:** Codex C1
- **Finding:** §Dispatch flow inserts `status='pending'`, advances `dispatch_state` (`creating_thread → thread_created → session_created → prompt_injected → wake_sent`), and only sets `status='running'` on first heartbeat. There is no transition that sets `status='dispatched'`. Cap checks count `status IN ('dispatched','running')` — so tasks actively creating threads/sessions/waking are invisible to the cap → orchestrator can blow past the cap during burst. Compounding: `authorizeChildCompletion` accepts only `dispatched|running`, so a fast completion before heartbeat is dropped.
- **Effort:** Low · **Impact:** High (cap bypass + dropped fast completions)
- **Resolution:** Set `status='dispatched'` atomically when the cap-aware promoter begins the dispatch (BEFORE step 5a). Heartbeat refines display state to `'running'` but cap and authorization checks already include `'dispatched'`. Update §Watchdog responsibility 4 to set status atomically with the promotion.
- **Fact-check:** Verified against §Dispatch flow steps 4 and 5e.
- **Status:** Open

### [M2-D] `dispatch-parent-${task_id}` is a NanoClaw DB message id, not a Slack/Discord platform parent ID — `getThreadId` cannot use it for adoption

- **Raised by:** A8, C2 (merged)
- **Finding:** §Adapter idempotency contract says "Slack: `conversations.replies` with the parent ts" and §Dispatch flow step 5a uses `dispatch-parent-${task_id}` as the deterministic parent message id. But Slack needs a real `thread_ts` (the platform-side message timestamp), and Discord needs a real Message/Thread object. NanoClaw's local message id is not platform-resolvable — the reconciler can't actually adopt a partial thread. Codex C2 also notes: even with `conversations.replies`, the API returns the parent regardless of whether a thread exists; thread-existence requires checking `thread_ts == ts` AND `reply_count > 0`.
- **Effort:** Medium · **Impact:** High (M1 idempotency contract incomplete)
- **Resolution:** `adapter.createThread` should own platform parent-message creation AND thread creation, returning `{ parentPlatformMessageId, threadId }`. Persist both into `tasks` (add `parent_platform_message_id TEXT` column) before continuing. Reconciler's adoption path uses `parent_platform_message_id` — and only adopts if `getThreadId` returns a value (semantics: "thread exists when parent's `thread_ts==ts` is set OR `reply_count > 0`"). If neither, re-call `createThread` with the persisted parent id (idempotent because the parent post was deduplicated by NanoClaw's `writeSessionMessage` on the deterministic local id but the platform post happened once).
- **Fact-check:** Verified Slack `conversations.replies` API semantics via official docs.
- **Status:** Open

### [M2-E] `completion_nonce` should be removed entirely — source-session binding is sufficient authorization, and the nonce creates a tampering/extraction surface

- **Raised by:** Codex C4
- **Finding:** Cycle-1 M16 added the nonce to close prompt-injection extraction (M5). Cycle-2 reveals the deeper truth: `createDispatchedSession` ensures one task ↔ one child session (verified — A1's resolution adds `sessions.dispatch_task_id UNIQUE`). The host's authorize check is `task_id = ? AND child_session_id = <source session>` — this binding alone is the capability. The nonce adds nothing the source-session check doesn't already provide. Meanwhile the nonce exists in `/workspace/.task-context.json` (RW-mountable per A9), in agent's reading context, in the host's `tasks.completion_nonce` column (plaintext per B-DRIFT-1) — three exposure surfaces that disappear if we drop it.
- **Effort:** Low (deletion) · **Impact:** High (security-model simplification + closes A9 / B-DRIFT-1 / C4 simultaneously)
- **Resolution:**
  1. Drop `tasks.completion_nonce` column.
  2. Drop `/workspace/.task-context.json` mechanism entirely. Children read `task_id` from a `kind='system'` banner row (formatter strips system rows from chat-platform delivery — exposed only to the agent's prompt context, not to Slack/Discord).
  3. `task_complete` MCP tool input drops `completion_nonce` field.
  4. `authorizeChildCompletion` becomes: `WHERE task_id = ? AND child_session_id = <source session> AND target_agent_group_id = <source agent group> AND status IN ('dispatched','running','complete')` (the `complete` addition closes Codex C1's "fast completion before heartbeat" gap; the row remains the row — there's nothing to forge).
  5. M16's "out-of-band delivery" stops being a thing — the threat model the nonce was protecting against (chat-history extraction of an authorization secret) doesn't exist when there is no secret.
- **Fact-check:** Verified against §Out-of-band nonce delivery paragraph + §Reverse signal step 2a.
- **Status:** Open

### [M2-F] Finalization markers have gaps — internal-only tasks never finalize; parent-wake has no marker; dependent-release ordering inverts the parent message contents

- **Raised by:** A6, Codex C5, Codex C6 (merged)
- **Finding:** Three related issues:
  1. (C5) Internal-only fallback tasks have `child_thread_id IS NULL`, so child-summary posting is skipped, so `child_summary_posted_at` stays NULL. §Watchdog responsibility 3 will retry every sweep forever.
  2. (A6) `wakeContainer(parent)` has no persisted marker. If wake fails (parent stopped), watchdog has no signal to retry.
  3. (C6) §Reverse signal step 2c writes the parent system row containing `dependents_unblocked` BEFORE step 2d computes which dependents are eligible. The deterministic parent-message id + `parent_notified_at` suppresses rewrites, so the orchestrator gets a permanently incomplete `dependents_unblocked` list.
- **Effort:** Low-Medium · **Impact:** High (M3 finalization story incomplete)
- **Resolution:**
  - C5: change watchdog responsibility 3 predicate. For internal-only tasks (`child_thread_id IS NULL`), set `child_summary_posted_at` to the same value as `completed_at` immediately when the task transitions to terminal status (it's "not applicable"), or change the predicate to `(child_thread_id IS NOT NULL AND child_summary_posted_at IS NULL)`.
  - A6: Add `parent_woken_at TEXT` marker. Watchdog responsibility 3 also resumes on NULL parent_woken_at. `wakeContainer(parent)` updates it on success; failure leaves it NULL for retry.
  - C6: Reorder finalize steps. Compute and persist `dependents_released_at` (step 2d → 2c in new ordering) BEFORE writing the parent system row (now last). Parent message contains the final, accurate `dependents_unblocked` list.
- **Fact-check:** Verified against §Reverse signal steps 2b-2e + §Watchdog responsibility 3 predicate.
- **Status:** Open

### [M2-G] Dashboard auth has no `/api/login` unauth exception specified; SSE Origin handling under-specified

- **Raised by:** Codex C7
- **Finding:** §Dashboard auth says "every API endpoint requires auth" but doesn't specify which endpoint accepts the bearer token to issue the cookie — that endpoint must be unauthenticated by definition. Origin checks on SSE need precise spec (browsers may omit Origin on same-origin GET; behavior must be defined).
- **Effort:** Low · **Impact:** High (auth flow won't actually work)
- **Resolution:** Specify exactly one unauthenticated endpoint: `POST /api/login`. Body `{ token }`. On match, sets the session cookie and returns 204. Every other endpoint requires the cookie. SSE: require valid cookie + Host allowlist. Origin allowlist enforced on mutating methods (POST/PUT/PATCH/DELETE) only — relax on GET/SSE because browsers may send empty Origin on same-origin same-tab navigations.
- **Fact-check:** Verified against §Dashboard auth section.
- **Status:** Open

### [M2-I] 4× extension cap formula breaks when `original_deadline IS NULL` — task_progress extension semantics undefined for tasks dispatched without a deadline

- **Raised by:** A10
- **Finding:** §Watchdog deadline-extension says `original_deadline + 4 × (original_deadline - spawned_at)`. If `original_deadline IS NULL` (no deadline set on dispatch — `deadline` is optional), the formula is undefined.

- **Resolution:** State explicitly that `task_progress.deadline_extension_seconds` is rejected with an error when `original_deadline IS NULL` (no deadline → no extension semantics). Document in the `task_progress` MCP tool description.
- **Fact-check:** Verified against §Watchdog deadline extension.

### [S2-D] Dispatch authorization is persistent agent-group authorization — prompt-injection on a stale orchestrator session can dispatch real work without owner intent

- **Raised by:** Codex C9
- **Finding:** "Once an agent group has the orchestrator role, any prompt that reaches that group can attempt to dispatch."
- **Resolution:** Two options (record judgment):
  - **(a)** Make per-dispatch approval mandatory: every `dispatch_task` call surfaces an approval card to the owner. Effective but operationally heavy.
  - **(b)** Owner-approved dispatch run-token: orchestrator must obtain a token from the owner (one approval per "dispatch session" — owner clicks "approve this orchestrator's next 22 dispatches"). Token bound to `parent_session_id` + expiry + max-task-count.
  Default per the brief's R13 ("role grant requires owner approval — one-time per orchestrator group, not per dispatch") leans (a) is too heavy. Pick (b) as a configurable option: `agent_groups.dispatch_approval_mode IN ('per-dispatch', 'session-token', 'none')`. Default `'session-token'` for orchestrator groups exposed to chat input; `'none'` for orchestrator groups with no chat exposure.
- **Status:** Open (user input may be needed on default — flag for `auto_judgment` if applied)
