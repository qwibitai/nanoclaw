## MUST-FIX Findings

### [M17] Container has only a per-agent PROJECTION of central.db; `sessions`, `agent_group_capabilities`, and `tasks` are NOT in the projection allow-list

- **Raised by:** Reviewer A (F1) — confirmed by direct reading of `src/db/per-agent-projections.ts`
- **Finding:** Design §3 line 298 says container reads `sessions.dispatch_task_id` via "the existing channel-inbound DB mount (sessions table is in `data/v2.db`, mounted RO)." Verified at `src/db/per-agent-projections.ts:128-131,142`: container-runner builds a per-agent **projection** of central.db (only `backlog_items`, `ship_log` are projected, filtered to the agent's rows). `sessions` table is explicitly excluded with the comment "the container has no need to see them" (lines 122-127). The container does NOT see `data/v2.db` directly — it sees a snapshot at spawn time built by `buildCentralProjection(centralSrc, centralDst, agentGroupId)` (`container-runner.ts:723-725`).
- **Effort to Fix:** Medium (relocate mechanism to inbound.db OR extend projection allow-list)
- **Impact if Ignored:** High — the M12 bifurcated tool mounting cannot work as designed. `getSessionDispatchTaskId()` would query a table that doesn't exist in the mounted DB.
- **Suggested resolution:** Move `dispatch_task_id` from `sessions` (central.db) to inbound.db. Either:
  - **(A)** Add `dispatch_task_id` column to inbound.db's `session_routing` table (writes via host's `writeSessionRouting` already called on every wake).
  - **(B)** Add a new `session_metadata` table to inbound.db schema with `(key, value)` rows.
  - **(C)** Pass `dispatch_task_id` as the first inbound message envelope only (drops the DB-column path entirely).
  Option A is cleanest — minimal new code, reuses existing helper, host already writes `session_routing` on every container wake. Container reads via new `getSessionDispatchTaskId()` that opens `getInboundDb()` (NOT central.db). Aligns with CLAUDE.md's "two-DB IO surface" invariant.
- **Fact-check:** **VERIFIED.** `per-agent-projections.ts:128-131,142` shows allow-list is `{'backlog_items', 'ship_log'}`. `container-runner.ts:723-725` confirms the projection-then-mount pattern.
- **Status:** Open

### [M18] §3 line 301 task_id prefix is `'task-'` but everywhere else uses `'dispatch-'`

- **Raised by:** Reviewer C (#1)
- **Finding:** Line 301 of design says `'task-' + sha256(...)` for the locally-computed task_id. Lines 227, 324, D8 commitment, and MVP IN-scope all say `'dispatch-'` (per cycle-2 fix S16). The orchestrator MCP tool would compute a task_id with the wrong prefix; host's lookups by `dispatch-<hash>` would never find it.
- **Effort to Fix:** Low (one-line text fix)
- **Impact if Ignored:** High — fundamental cancellation/correlation breakage; orchestrator and host produce different IDs for the same task.
- **Suggested resolution:** Change line 301 to `'dispatch-' + sha256(...)`. Add a contract test to /team-tdd that asserts both the host-side derivation and the container-side computation produce identical IDs for the same `(parent_session_id, idempotency_key)`.
- **Fact-check:** **VERIFIED via grep — line 301 still has `'task-'` prefix.**
- **Status:** Open

### [M19] `completeDispatchSideEffects(taskRow)` needs the full task row but the atomic-admit `INSERT ... RETURNING` returns only `task_id, request_hash`

- **Raised by:** Reviewer C (#2)
- **Finding:** Design §4 line 320 schedules `setImmediate(completeDispatchSideEffects, taskRow)`. But the INSERT in step 7 (lines 333-338) has `RETURNING task_id, request_hash` only. `completeDispatchSideEffects` needs `parent_messaging_group_id`, `target_agent_group_id`, `task_content`, `parent_session_id`, `parent_agent_group_id`, `deadline`, plus all artifact columns. The implementer is left to either invent `RETURNING *` (not stated) or re-SELECT.
- **Effort to Fix:** Low (text fix)
- **Impact if Ignored:** High — the side-effect completion path's input shape is undefined.
- **Suggested resolution:** Specify in §4: `completeDispatchSideEffects(task_id: string)` reloads the full row at entry via `getTaskById(task_id)` AND re-reads before each artifact step (so reconciler-vs-setImmediate concurrent runs see fresh state). This also coincides with M23 fix.
- **Status:** Open

### [M20] Concurrency cap check (step 2) precedes idempotency replay detection (step 3); valid retry rejected when at cap

- **Raised by:** Reviewer C (#3)
- **Finding:** §4 step 2 (line 323) counts pending+running and rejects at cap. Step 3 (line 324) computes `task_id` and checks for existing row. A valid retry of an already-admitted task can be rejected as "cap exceeded" when the parent session is at cap — violating Brandur Leach / Stripe canonical idempotency contract (replays MUST be idempotent regardless of current state).
- **Effort to Fix:** Low (reorder steps)
- **Impact if Ignored:** Medium-High — silent retry rejection under load defeats the idempotency guarantee.
- **Suggested resolution:** Reorder: step 1 auth → NEW step 2: compute `task_id`, attempt SELECT by `(parent_session_id, idempotency_key)`. If existing → idempotency replay path (re-emit current status). If new → proceed to step 3 (concurrency cap), step 4 (target validation), step 5 (wiring), step 6 (request_hash), step 7 (atomic INSERT).
- **Status:** Open

### [M21] Race: child container is woken BEFORE `tasks.child_session_id` and `status='running'` are written; child's first `dispatch_progress` rejected, completion CAS fails

- **Raised by:** Reviewer C (#4)
- **Finding:** §4 step 8 threaded path order: `wakeContainer(child_session)` then `UPDATE tasks SET child_session_id = ?, started_at = now, ...`. mgId=null path same pattern. A fast child can run, call `dispatch_progress({task_id, message})`, hit the host's auth check `WHERE task_id=? AND child_session_id=callerSession.id` — but `child_session_id` isn't set yet → reject. Or worse: child calls `dispatch_complete`, host's CAS `WHERE task_id=? AND status='running'` updates 0 rows because the prior `UPDATE ... status='running'` hasn't committed.
- **Effort to Fix:** Low (reorder writes)
- **Impact if Ignored:** High — race-class progress rejections AND race-class completion losses on fast children.
- **Suggested resolution:** Reorder step 8 last sub-step: write tasks columns FIRST (child_session_id, started_at, last_progress_at, status='running', and `sessions.dispatch_task_id`), THEN `writeSessionMessage` first inbound, THEN `wakeContainer`. Container can't run until wake; host writes complete before child has a chance to call back.
- **Status:** Open

### [M22] Side-effect artifact updates in step 8 are NOT status-guarded; cancel/watchdog can race with completion

- **Raised by:** Reviewer C (#5)
- **Finding:** §4 step 8 updates artifact columns with `UPDATE tasks SET parent_platform_message_id = ? WHERE task_id = ?`. No status guard. Meanwhile `dispatch_cancel` or watchdog can flip `status='cancelled'` or `status='failed'` between calls. A subsequent step-8 update would overwrite back to `running` (line 372: `status = 'running'`). End state: cancelled task observably running.
- **Effort to Fix:** Low-Medium (add CAS guards)
- **Impact if Ignored:** High — terminal state can flip back; cancel/watchdog don't actually terminate.
- **Suggested resolution:** Every artifact-write UPDATE in step 8 must CAS on `status='pending'`: `UPDATE tasks SET parent_platform_message_id = ? WHERE task_id = ? AND status='pending'`. Final transition uses `WHERE task_id = ? AND status='pending' AND child_session_id IS NULL`. If 0 rows updated mid-way, abort `completeDispatchSideEffects` (the task was terminated).
- **Status:** Open

### [M23] `setImmediate` job AND reconciler can run `completeDispatchSideEffects` concurrently for the same task

- **Raised by:** Reviewer C (#6)
- **Finding:** §4 says `setImmediate(completeDispatchSideEffects, taskRow)` runs after admit. §5 reconciler runs every 60s and calls the same function for any `status='pending' AND child_session_id IS NULL` row. If the setImmediate job is still in flight (e.g., adapter call slow) when the 60s sweep fires, both run concurrently, both observe `parent_platform_message_id IS NULL`, both call `adapter.postParent` → DUPLICATE Slack/Discord parent message even without a host crash. The S12-acknowledged "rare host crash" failure mode is now a routine race.
- **Effort to Fix:** Medium (in-process lease/lock + durable claim)
- **Impact if Ignored:** High — non-rare duplicate-message UX degradation; the entire S12 acceptance is invalidated.
- **Suggested resolution:** Two-layer guard:
  - **In-process:** add a `completionInFlight: Map<task_id, Promise>` on the dispatch module. setImmediate path acquires the entry; reconciler skips any task in the map.
  - **Durable:** add `tasks.completion_lease_at TEXT` column; before running completion, atomic UPDATE sets `completion_lease_at = now WHERE task_id = ? AND (completion_lease_at IS NULL OR completion_lease_at < now - 60s)`. If 0 rows updated, another worker has the lease; skip. Clears on success or failure.
  Both layers needed: in-process for steady-state; durable for crash-recovery on a stale in-process lease.
- **Status:** Open

### [M24] §5 drain-first grace window measured against PROGRESS timestamps, not against the pending TERMINAL-outbound row's timestamp

- **Raised by:** Reviewer C (#7)
- **Finding:** §5 line 480: `drain_age = now - parse(task.last_progress_at OR task.started_at OR task.admitted_at)`. A quiet long-running child (no progress reports) can enqueue `dispatch_complete` after `now - last_signal > no_progress_timeout`. drain-first computes `drain_age` from old timestamps → exceeds 120s grace → falls through to no-progress fail. Watchdog terminates the task while the legitimate completion is sitting one row away in outbound.
- **Effort to Fix:** Low (different timestamp source)
- **Impact if Ignored:** Medium-High — race-class false failures on quiet long-running tasks.
- **Suggested resolution:** Change `has_pending_terminal_dispatch_outbound(child_session_id)` to return `terminal_outbound_seen_at` (the actual timestamp of the pending row). Compute `drain_age = now - terminal_outbound_seen_at`. If a terminal row appeared in outbound just before the sweep, grace starts NOW, not from the last progress signal.
- **Status:** Open

### [M25] §6 Slack `createThread` returns the in-thread reply `ts` as `threadId`, but `parentMessageId` IS the Slack thread identifier — semantic confusion

- **Raised by:** Reviewer C (#9)
- **Finding:** §6 line 547 says: "Slack `createThread`: `chat.postMessage({ channel: platformId, thread_ts: parentMessageId, text: firstMessage })` returns `ts` of the in-thread message; `parentMessageId` IS the thread identifier in Slack semantics." But the design's threaded-path step 8 (line 369) persists the returned `threadId` as `child_platform_thread_id`. Subsequent Slack delivery uses `child_platform_thread_id` as `thread_ts` — but in Slack, that should be the PARENT message's `ts`, not the in-thread reply's `ts`. Posting with `thread_ts = <reply_ts>` would either error or post to the wrong location.
- **Effort to Fix:** Low (return-shape fix)
- **Impact if Ignored:** High — every Slack-threaded task delivers child outbound to the wrong place.
- **Suggested resolution:** Slack `createThread` returns `{ threadId: parentMessageId, messageId: reply_ts }`. Both fields point to the right things: `threadId` is the thread-root timestamp (Slack `thread_ts`); `messageId` is the in-thread reply's timestamp (for editing/deletion of that specific message). The wrapper-layer `createThread` signature already returns `{threadId, messageId}` — Slack adapter just needs to populate `threadId` from `parentMessageId`, not from the new reply.
- **Status:** Open

### [S23] `request_hash = sha256(target_group + ':' + content + ':' + (deadline ?? ''))` admits structural collisions

- **Raised by:** Reviewer B (Q4 MEDIUM drift)
- **Finding:** User-supplied `content` can contain `:`. Three distinct tuples like `('A', 'B:C', '')`, `('A', 'B', 'C:')`, `('A:B', 'C', '')` all produce identical `request_hash` strings — colliding under canonical hash-of-concatenation guidance (RFC 8785, AWS SigV4, Brandur Leach references via Reviewer B). Operational exposure bounded by `(parent_session_id, idempotency_key)` namespace, but the foot-gun is well-documented.
- **Suggested resolution:** Use length-prefixing: `request_hash = sha256(len(target_group) || ':' || target_group || len(content) || ':' || content || len(deadline) || ':' || deadline)` OR JCS-canonical JSON object `{target_group, content, deadline}`. Trivial fix; closes the canonical drift.
- **Status:** Open

- **Finding:** §4 step 7 checks `typeof adapter.createThread === 'function'` on the wrapper. Once any adapter (Slack/Discord) implements it, the wrapper exposes the method — non-thread channels appearing through the same wrapper could be mis-classified as thread-capable.
- **Suggested resolution:** Per-channel check: `getChannelAdapter(channelType).createThread` exists. Or add `supportsCreateThread(channelType): boolean` to the wrapper.

### [S25] Surface mode (threaded vs mgId=null) not persisted; happy-path and recovery paths can disagree

- **Raised by:** Reviewer C (#10)
- **Finding:** §4 chooses threaded vs mgId=null based on adapter capability at admit time. If adapter availability changes between admit and recovery (e.g., adapter version downgrade, plugin uninstall), reconciler decides differently than the original admit — could split a task across both modes or fail with `adapter_unavailable` for a task that admitted in mgId=null mode.
- **Suggested resolution:** Persist `surface_mode TEXT NOT NULL CHECK (surface_mode IN ('native_thread', 'headless'))` on tasks at admit time. Both happy-path completion and reconciler use the stored mode, not re-evaluate.
- **Status:** Open

### [S26] `dispatch_cancel_request` system action: container-side protocol unspecified

- **Raised by:** Reviewer C (#11)
- **Finding:** §4 cancel handler sends `kind='system'` action `dispatch_cancel_request` to child's inbound. But there's no container-runner handler spec for this action — does the agent see it as a chat message? Get a special signal? Just have the container killed at the 2-min timer? Builders cannot implement.
- **Suggested resolution:** Either (a) make cancellation host-only (skip the inbound write; arm the kill timer immediately on `dispatch_cancel`), or (b) add a §4 sub-section specifying container-runner's handler: receive system action, propagate to agent as a system note "task cancelled by orchestrator," allow agent to flush state, then host kills at 2-min mark regardless.
- **Status:** Open
