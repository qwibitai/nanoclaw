## Overview

Phase 1 ships the orchestrator-dispatch primitive: an existing agent group designated as orchestrator gets new MCP tools (`dispatch_task`, `list_dispatched_tasks`, `dispatch_cancel`) that spawn child sessions in target groups. Each dispatched task becomes its own NanoClaw session with its own real Slack/Discord platform thread (or `mgId=null` headless session for non-thread channels). The host runs a synchronous atomic admit + asynchronous side-effect-completion + 60s reconciler-recovery model. Children report progress and terminal state via `dispatch_progress`/`dispatch_complete`/`dispatch_failed` MCP tools, gated by a bifurcated mounting condition keyed on `session_routing.dispatch_task_id`.

Phase 1 deliverables (per design "MVP Scope Boundary"):
- `agent_group_capabilities` table + capability-grant flow
- `tasks` table + `session_routing.dispatch_task_id` extension (migration 026)
- 5 system actions (`dispatch_task`, `dispatch_complete`, `dispatch_failed`, `dispatch_cancel`, `dispatch_progress`) registered via the existing `registerDeliveryAction` pattern
- 6 MCP tools mounted bifurcated by orchestrator-capability vs child-task-session
- `ChannelDeliveryAdapter` widening + `ChannelAdapter` per-adapter Slack/Discord implementations of `postParent`/`createThread`
- Watchdog `decideTaskAction` in host-sweep with drain-first scoped to terminal-dispatch outbound and bounded grace
- Reconciler for crash recovery on existing 60s sweep
- Runtime assertion `channel_type IS NULL ⟺ platform_id IS NULL` in delivery + session-manager

Phase 0 (lookback extractor) is already complete (`scripts/lookback.ts` restored).

## Known Risks

Carried forward from the cycle-3 review (`docs/specs/orchestrator-dispatch/review.md`) as known risks. None block /team-build but should be tracked through /team-qa and /team-ship.

- **Risk K1 — Rare-host-crash duplicate platform message** (review S12 / A14 — accepted limitation):
  If the host crashes between `adapter.postParent` succeeding (Slack/Discord side) and the host's `UPDATE tasks SET parent_platform_message_id` committing, the reconciler's next sweep will retry `postParent` and produce one duplicate "Launched dispatch:" message. Operationally bounded by host crash frequency. Mitigation deferred to v2 (metadata-correlation via Slack `metadata.event_payload` / Discord embed footer keyed on task_id). Mitigation strategy documented for v2.

- **Risk K2 — Adapter rate-limit blowback under fanout** (review S19 / A15 — defer to /team-tdd validation):
  22-task fanout fires 22 `setImmediate` callbacks; all race against Slack tier-1's 1 chat.postMessage/sec rate limit. Current default 5 retry attempts × `dispatch_completion_attempts` may produce false `completion_exhausted` markings. /team-tdd benchmark will surface this; if confirmed, implement per-channel sleep/queue in `completeDispatchSideEffects` before /team-ship. Tracked as A12 in design Assumptions Log.

- **Risk K3 — Per-agent-projection allow-list extension** (Group A / D3 dependency):
  Cycle-3 fix M17 keeps `dispatch_task_id` in inbound.db (avoids the projection issue), BUT D3's `list_dispatched_tasks` MCP tool still needs the central DB's `tasks` table available in the per-agent projection. Task A2 acceptance criterion includes adding `tasks` and `agent_group_capabilities` to the allow-list at `src/db/per-agent-projections.ts:128-131,142`. If this is missed, D3's tool fails silently (queries empty projection). Verified by D3's `test_list_dispatched_tasks_reads_central_db`.

- **Risk K4 — Memory budget reconciliation** (Phase 2 — design constraint C1):
  150MB resident memory for the dashboard process is provisional pending Phase 2 build measurement. Not in Phase 1 scope; reconciliation gate at Phase 2 /team-ship.

- **Risk K5 — Cross-cycle review fatigue** (cap-reached path 1 chosen):
  /team-review hit its 3-cycle cap. User explicitly approved Path 1 (proceed without 4th review) on the basis that cycle-3 findings were addressable inline with bounded scope. If /team-build surfaces issues that should have been caught in cycle 4, the recovery path is /team-design re-entry (rollback) — NOT another /team-review attempt.
