# Design: Orchestrator Dispatch + Memory-Efficient Dashboard

> Generated by `/team-design` — 2026-05-09
> Saved to: `docs/specs/orchestrator-dispatch/design.md`

## Brief Reference

Approved brief: `docs/specs/orchestrator-dispatch/brief.md`. Two-phase feature.

**Phase 1 (one shippable milestone):** new `dispatch_task` MCP primitive, additive to `agent-to-agent`. Owner-approved `orchestrator` role on an agent group grants blanket dispatch authority. Each dispatch creates its own NanoClaw session in a target group with its own real platform thread (Slack/Discord) so the owner can chat with each task natively. Channels without native threads (Telegram, iMessage, WhatsApp, email) degrade to `mgId=null` children visible only via the orchestrator's task list. Durable host-side `tasks` table with watchdog in `host-sweep.ts`. Reverse `dispatch_complete`/`dispatch_failed` system action on existing `delivery.ts` registry.

**Phase 2 (read-only dashboard):** Vite+React SPA mounted on the existing Node HTTP server (`webhook-server.ts`). Reads central + session DBs directly, owner-only auth via `canAccessAgentGroup`, SSE live updates via chokidar. Visual quality gated by `bootstrap-domain:jony-ive` and `impeccable:impeccable` skills during build.

- **C1 is numeric and provisional.** Per `/team-design` Step 2 grounding rule, every numeric HARD must cite empirical evidence from the most representative executable path. ## Research Summary

### Project patterns (from CLAUDE.md and `docs/project-scope.md`)

- **Two-DB session split:** `inbound.db` host-write/container-read, `outbound.db` container-write/host-read. Single writer per file. Cross-mount visibility relies on `journal_mode=DELETE`. Documented: `CLAUDE.md` "Two-DB Session Split" section, `container/agent-runner/src/db/connection.ts`. Implication: child sessions written by the host take effect through the same path as channel-inbound — no new IO surface needed.
- **System actions via `registerDeliveryAction`:** `delivery.ts:526-556` (action handler registry). Used by `agent-to-agent`, `backlog`, `bash-gate`, `channel-config`, `remote-control`. Pattern: container writes outbound `kind='system'` with JSON content `{action: '<name>', ...}`; host's deliveryMessage dispatches to registered handler.
- **Privilege lives on users, not groups.** Schema comment in `src/db/schema.ts:9` and `:73`: "All workspaces are equal; privilege lives on users, not groups." Implication: orchestrator role goes in a separate `agent_group_capabilities` table (mirrors `user_roles`), NOT as a column on `agent_groups`. Confirmed against codex's M7 from prior cycle.
- **Existing HTTP server:** `webhook-server.ts:24-26` runs `http.createServer` on port 3000 (default), maintains a `routes` map, used by Chat SDK adapter webhooks. Lazy-started on first adapter registration. **Implication: dashboard mounts as additional routes on this server. No new server.**
- **Module pattern for new system actions:** `src/modules/<name>/index.ts` does `registerDeliveryAction('action', handler)` at import time. Module is loaded by `src/modules/index.ts`. Same pattern for the new orchestrator-dispatch module.

### Library docs (verified)

- **chokidar** (file watcher) — already a transitive dep candidate; verified via Context7 that the v3 API supports `watch(paths, { ignoreInitial: true })` with debounced events. ~50KB on disk, ~10-20MB resident memory. Suitable for watching `data/v2-sessions/*/outbound.db` files. Source: chokidar README on GitHub.
- **Server-Sent Events in Node** — built-in via `http.ServerResponse` with `Content-Type: text/event-stream`, `Connection: keep-alive`, `Cache-Control: no-cache`, periodic heartbeat comments. No library needed. Source: MDN SSE docs.
- **Vite + React static build** — `vite build` produces `dist/` with hashed bundles, ~100-200KB JS gzipped for a small SPA. Static-served via Node `fs.createReadStream` and content-type mapping. No SSR; localhost-only consumer. Source: Vite docs (vitejs.dev/guide/static-deploy).

### Existing code (file:line citations)

- `src/db/schema.ts:78-86` — `user_roles` table pattern: composite PK `(user_id, role, agent_group_id)` with nullable `agent_group_id` for global vs scoped. We mirror this for `agent_group_capabilities`.
- `src/db/schema.ts:110-122` — `sessions` table; `messaging_group_id` already nullable (supports mgId=null children); migration 024 enforces channel-root uniqueness via partial index on `WHERE thread_id IS NULL`.
- `src/host-sweep.ts:61-78` — `SWEEP_INTERVAL_MS = 60_000`, `ABSOLUTE_CEILING_MS = 30 min`, `CLAIM_STUCK_MS = 60s`, `SPAWN_GRACE_MS = 60s`. Pure decision function `decideStuckAction` (line 90+). Adding a tasks-watchdog responsibility fits the same shape: pure decision + IO at top.
- `src/session-manager.ts:318-345` — `writeSessionRouting` writes `(channel_type, platform_id, thread_id)` to inbound.db's `session_routing` row; channel-side `delivery.ts:358-364` reads this and passes to `adapter.deliver`. **This is exactly the path that breaks for synthetic thread_ids.**
- `src/session-manager.ts:355-371` — `writeSessionMessage(agentGroupId, sessionId, {...})` is the host primitive for inbound writes. We use this directly from the dispatch handler to inject the task content into a fresh child session.
- `src/container-runner.ts:119-149` — `wakeContainer(session)` deduplicates concurrent wakes via in-flight promise map. Safe to call from dispatch handler.
- `src/delivery.ts:280-296` — system action dispatch path: `if (msg.kind === 'system') { ... handleSystemAction(content, session, inDb) }`. `handleSystemAction` looks up the action name in the registry filled by `registerDeliveryAction`.
- `src/modules/agent-to-agent/index.ts:23` — minimal module pattern: import handler, call `registerDeliveryAction('create_agent', handleCreateAgent)`. Three lines of bootstrap. The orchestrator-dispatch module mirrors this for `dispatch_task`, `dispatch_complete`, `dispatch_failed`, `dispatch_cancel`, `dispatch_progress`.
- `src/webhook-server.ts:24, 70-100` — `routes` map keyed by adapter name, lazy server start, request adapter from Node http to Web Request. New dashboard routes register via the same `routes` map (or via a parallel `dashboardRoutes` map; design decides — see Option A vs B).
- `src/modules/permissions/access.ts:21-28` — `canAccessAgentGroup(userId, agentGroupId)` returns `AccessDecision`. The dashboard auth check is just `isOwner(userId)` (function exported from `src/db/user-roles.ts`).
- `src/db/migrations/024-sessions-channel-root-unique.ts` — migration pattern: `{ version: N, name: 'kebab-case', up: (db) => { ... } }`. New migrations: 025 (`agent_group_capabilities`), 026 (`tasks`).
- `container/agent-runner/src/mcp-tools/scheduling.ts:80-100` — MCP tool pattern: write outbound with `kind: 'system'` and `content: JSON.stringify({ action: '<name>', ... })`. Host's registered handler picks it up. `dispatch_task`, `dispatch_cancel`, `dispatch_progress` follow the same pattern.

## Options

The brief locks the high-level architecture (additive primitive, agent_group_capabilities vs column, system action for reverse signal). The genuine architectural choice left to evaluate is the **task lifecycle representation** — i.e., how state and recovery work. This is also where the prior cycle 3 of the abandoned attempt accumulated its MUST-FIX findings (M-29, M-30, M-31), so the choice has historical weight.

### Option A: Coarse status + nullable artifact columns (single source of truth from data)

**Approach:** `tasks.status ∈ {pending, running, completed, failed, cancelled}` is the lifecycle column. Side-effect artifacts (`parent_platform_message_id`, `child_platform_thread_id`, `child_session_id`, `admitted_at`, `started_at`, `completed_at`, `last_progress_at`) are nullable columns filled in as side effects succeed. Recovery and reconciliation logic infers what's been done by checking which columns are NULL. Watchdog reads `last_progress_at` (not a state-machine timer) for no-progress decisions.

**Pros:**
- Single source of truth — no risk of `dispatch_state` column drifting from artifact columns
- Fewer states to enumerate, test, and verify
- Eliminates the entire class of "phantom column" bugs that bit cycle 3 (M-29, M-30) — the columns referenced by responsibilities are exactly the columns in the schema, by construction
- `last_progress_at` is a first-class column, so reset-on-progress (C7) is direct: `dispatch_progress` writes `last_progress_at = now`; watchdog reads `WHERE last_progress_at < now - timeout` → no risk of forgetting to reset (M-31)
- Simpler queries for the dashboard ("WHERE status='running'")
- Maps cleanly to Stripe's idempotent_requests pattern (status + artifact columns) which is widely deployed

**Cons:**
- Recovery code has to handle every NULL combination explicitly (e.g., admitted but no parent_platform_message_id → resume from postParent step)
- Less self-documenting than named states; a reader has to infer the pipeline from null-checks
- If we later need an explicit audit trail of state transitions, we'd need to add a `task_state_log` table

**Pattern alignment:** Matches the project pattern of "minimal nullable columns rather than separate state tables" (e.g., `sessions.last_active`, `sessions.container_status` — both in `src/db/schema.ts:117-118` are coarse with nullable timestamps rather than fine-grained state machines). Aligns with CLAUDE.md emphasis on "everything is a message" simplicity.

**Confidence:** **High.** All assumptions validated against existing code. The cycle 3 review noted Option B (below) was "structurally sound but had implementation gaps" — Option A eliminates the categories of gaps that bit cycle 3 by collapsing redundant state.

**Assumptions:** Recovery logic stays bounded (≤6 NULL-check branches per resume path). If the dispatch pipeline grows to >10 distinct artifact-creation steps, we should revisit.

---

### Option B: Fine-grained `dispatch_state` machine (cycle 3 pattern)

**Approach:** `tasks.status` is coarse but `tasks.dispatch_state` is fine: `pending → admitted → creating_thread → parent_posted → thread_created → running → completing → completed/failed/cancelled`. Each transition has a reconciler clause in host-sweep. `dispatch_state_updated_at` plus `last_progress_at` plus `last_progress_message` columns. Recovery scans by `dispatch_state` to find tasks needing resume action.

**Pros:**
- Self-documenting: state names describe exactly where we are in the pipeline
- Reconciler can use the state column as a single index for "tasks needing recovery action of type X"
- Reviewer B (best-practice review of the prior cycle) called this "textbook conformant" with Temporal/Conductor reaper patterns
- Cleaner separation between dispatch-pipeline state and child-execution state

**Cons:**
- Redundant with artifact columns: `dispatch_state='parent_posted'` ⟺ `parent_platform_message_id IS NOT NULL AND child_platform_thread_id IS NULL`. The state column is derivable, and any drift between them is a bug.
- Cycle 3 v1 hit exactly this drift class three times (M-29 phantom `container_status_changed_at`, M-30 phantom `last_progress_message`, M-31 timer not reset). Each is a state-vs-data sync bug.
- Watchdog responsibility 7's M-31 finding ("`dispatch_state_updated_at` never reset by `dispatch_progress`") is intrinsic to having `dispatch_state_updated_at` AND `last_progress_at` as separate timers — easy to forget to update one.
- More schema columns; more migration surface.

**Pattern alignment:** Matches Temporal/Conductor (per Reviewer B from prior cycle) but not the existing nanoclaw codebase, which uses the coarse-status-plus-artifact pattern (sessions table itself does this). Aligns less well with CLAUDE.md "two-DB minimal IO surface" simplicity.

**Confidence:** **Medium.** Architecturally sound (Reviewer B greenlit), but the implementation track record from cycle 3 shows it accumulates phantom-column / state-drift bugs faster than Option A.

**Assumptions:** Engineering discipline to keep `dispatch_state` synchronized with artifact columns at every transition. Past evidence (cycle 3) suggests this discipline is hard to sustain across reviews.

---

### Option C: Eventless inferred status (no `status` column at all)

**Approach:** `tasks.status` is removed entirely. Lifecycle is fully inferred from artifact columns: pending if `admitted_at IS NULL`, running if `admitted_at IS NOT NULL AND completed_at IS NULL AND failed_at IS NULL AND cancelled_at IS NULL`, etc. Most "purely data-modeled" of the three.

**Pros:**
- Maximally normalized — no chance of state column being out of sync with artifact columns because the state column doesn't exist
- Forces the schema to be self-describing

**Cons:**
- Common queries become long ("WHERE admitted_at IS NOT NULL AND completed_at IS NULL AND failed_at IS NULL AND cancelled_at IS NULL")
- Slightly slower queries unless we add a generated/virtual column or covering index
- Harder for a human reviewer to scan-read
- Doesn't materially improve correctness over Option A (the `status` column in A is updated atomically with artifact column writes, so drift is a single-transaction concern)

**Pattern alignment:** Doesn't match any existing nanoclaw table.

**Confidence:** **Medium-Low.** Theoretically clean but adds query and ergonomic friction without significant correctness benefit over A.

**Assumptions:** Reviewers and dashboard builders don't mind null-check WHERE clauses; future maintainers can read and write them without error.

---

## Recommendation

**Selected:** **Option A — coarse status + nullable artifact columns.**

**Justification:**

1. **Eliminates the cycle 3 failure class.** The three MUST-FIX findings from cycle 3 v1 (phantom `container_status_changed_at`, phantom `last_progress_message`, no-progress timer not reset) all stem from having parallel state/timestamp columns that have to be kept in sync manually. Option A makes the columns themselves the state, so there is no parallel to drift.
2. **Aligns with existing nanoclaw schema.** The `sessions` table (the most analogous existing structure) is exactly this shape: coarse status (`status`, `container_status`) plus nullable timestamps (`last_active`). Option B would introduce a pattern foreign to the rest of the codebase.
3. **C7 watchdog timer reset is structurally correct.** The watchdog reads `last_progress_at`. The `dispatch_progress` handler writes `last_progress_at = now`. There is no parallel `dispatch_state_updated_at` to forget. The bug class M-31 chased in cycle 3 is impossible by construction.
4. **Codex's prior best-practice review (Reviewer B, cycle 3 v1)** greenlit Option B but explicitly noted the trade-off with Option-A-style "separate idempotency-key + resource-id namespace" was a LOW-severity drift. Option A's drift in the other direction is also LOW. Choosing the simpler one is correct when both are conformant.
5. **Recovery logic stays bounded.** The dispatch pipeline has 4 artifact-creation steps (admit, postParent if threaded, createThread if threaded, openSession). 4 NULL-check branches in recovery, well below the "should reconsider" threshold I stated in Option A.

**Confidence note:** **High.** Validated against codebase patterns and against the failure history of the alternate option. No spike needed.

## Detailed Specification

This section translates the Option A recommendation into concrete primitives that drive `/team-plan`.

### 1. New host modules

```
src/modules/orchestrator-dispatch/
├── index.ts              — registerDeliveryAction calls
├── dispatch.ts           — applyDispatchTask (handler for 'dispatch_task')
├── completion.ts         — applyDispatchComplete, applyDispatchFailed
├── cancellation.ts       — applyDispatchCancel
├── progress.ts           — applyDispatchProgress
├── watchdog.ts           — task-related host-sweep responsibilities
├── reconciler.ts         — crash-recovery scan (called from index.ts on host start)
└── db/
    ├── tasks.ts                       — CRUD for tasks table
    └── agent-group-capabilities.ts    — CRUD for agent_group_capabilities table
```

Module bootstrap (`index.ts`) — following the `agent-to-agent/index.ts:23` pattern. **All actions use the `dispatch_` prefix to avoid collision with the existing scheduling module's `cancel_task` action (verified at `src/modules/scheduling/index.ts:31`)** — see review M1:

```ts
registerDeliveryAction('dispatch_task', handleDispatchTask);
registerDeliveryAction('dispatch_complete', handleDispatchComplete);
registerDeliveryAction('dispatch_failed', handleDispatchFailed);
registerDeliveryAction('dispatch_cancel', handleDispatchCancel);
registerDeliveryAction('dispatch_progress', handleDispatchProgress);
runReconcilerOnStartup();   // recover in-flight tasks from prior host restart
```

(Also recommended as a separate hardening: extend `registerDeliveryAction` in `src/delivery.ts` to throw on duplicate-key registration instead of silently overwriting. Logged for follow-up; not blocking this design.)

### 2. Schema migrations (025, 026)

Following the migration024 pattern (`src/db/migrations/024-sessions-channel-root-unique.ts`):

**Migration 025 — `agent_group_capabilities`:**

```sql
CREATE TABLE agent_group_capabilities (
  agent_group_id TEXT NOT NULL REFERENCES agent_groups(id),
  role           TEXT NOT NULL,        -- 'orchestrator' (extensible)
  config_json    TEXT,                 -- per-role config: concurrency cap, timeouts, denylist
  granted_by     TEXT REFERENCES users(id),
  granted_at     TEXT NOT NULL,
  PRIMARY KEY (agent_group_id, role)
);
CREATE INDEX idx_agent_group_capabilities_role ON agent_group_capabilities(role);
```

**Naming rationale (review S2):** the schema invariant at `src/db/schema.ts:9` declares "All workspaces are equal; privilege lives on users, not groups." This table is *not* a privilege grant on a group (would violate that invariant) — it is a **capability declaration** that this group plays an orchestrator role. Parallel-but-different from `user_roles` (which IS privilege on users). The rename makes the distinction explicit. `agent_group_id` is non-NULL because orchestrator is a per-group capability (unlike user_roles where NULL means "global"). `config_json` carries the orchestrator's concurrency cap and target denylist (decisions.yaml F1, F4).

**Revoke-while-running rule (review S2):** revoking the orchestrator capability MUST refuse if any tasks for that orchestrator have `status IN ('pending', 'running')`. Operator must cancel/wait first. Enforced at the revoke entry point (admin DM command or future dashboard action), not via DB trigger.

**Migration 026 — `tasks` + `sessions.dispatch_task_id` extension:**

```sql
CREATE TABLE tasks (
  task_id                       TEXT PRIMARY KEY,
                                -- derived: dispatch-<sha256(parent_session_id || ':' || idempotency_key)[0:16]>
                                -- (cycle-2 fix S16: 'dispatch-' prefix to disambiguate from scheduling's 'task-')
  idempotency_key               TEXT NOT NULL,
                                -- caller-supplied; namespaced by parent_session_id via the UNIQUE below.
  parent_session_id             TEXT NOT NULL REFERENCES sessions(id),
  parent_agent_group_id         TEXT NOT NULL REFERENCES agent_groups(id),
  parent_messaging_group_id     TEXT REFERENCES messaging_groups(id),
                                -- recorded at admit time so reconciler can recover channel surface
                                -- after host restart even when callerSession has been archived (review S6)
  target_agent_group_id         TEXT NOT NULL REFERENCES agent_groups(id),
  child_session_id              TEXT REFERENCES sessions(id),
                                -- NULL until child session opened
  status                        TEXT NOT NULL DEFAULT 'pending',
                                -- 'pending' | 'running' | 'completed' | 'failed' | 'cancelled'
  task_content                  TEXT NOT NULL,
                                -- the user-facing prompt the orchestrator dispatched
  request_hash                  TEXT NOT NULL,
                                -- LENGTH-PREFIXED hash (cycle-3 fix S23): sha256(
                                --   len(target_group) || ':' || target_group ||
                                --   len(content)      || ':' || content      ||
                                --   len(deadline??'')|| ':' || (deadline??'')
                                -- ) — avoids collisions when content contains the ':' separator.
                                -- Idempotency replay safety per Brandur Leach / Stripe canonical pattern + RFC-8785 / AWS-SigV4 canonicalization guidance.
                                -- On replay (existing task_id), if request_hash differs, return structured error.
  deadline                      TEXT,
                                -- optional ISO; absent → no-progress timeout governs
  -- Artifact columns (filled as side effects succeed; NULL means "not yet"):
  parent_platform_message_id    TEXT,    -- adapter.postParent return value (null on mgId=null)
  child_platform_thread_id      TEXT,    -- adapter.createThread return value (null on mgId=null)
  child_messaging_group_id      TEXT REFERENCES messaging_groups(id),
                                -- the mg the child thread lives under (null on mgId=null)
  -- Lifecycle timestamps:
  admitted_at                   TEXT NOT NULL,    -- (cycle-2 fix M10) set in atomic INSERT — never NULL after committed row
  started_at                    TEXT,    -- child session received first inbound
  completed_at                  TEXT,    -- dispatch_complete signal applied
  failed_at                     TEXT,    -- dispatch_failed or watchdog-fail
  cancelled_at                  TEXT,    -- dispatch_cancel applied
  last_progress_at              TEXT,    -- dispatch_progress writes this; watchdog reads this (sole timer)
  -- Audit / observability:
  last_progress_message         TEXT,    -- (cycle-2 fix S10) overwritten on each dispatch_progress; ≤500 chars; NOT an audit log
  fail_reason                   TEXT,    -- 'no_progress_timeout' | 'deadline_exceeded' | 'spawn_deadline' | 'container_exit' | 'cancelled' | 'agent_error' | 'adapter_unavailable' | 'completion_exhausted' | 'target_not_wired_to_caller_messaging_group' | etc.
  result_summary                TEXT,    -- final summary from the child on completion
  dispatch_completion_attempts  INTEGER NOT NULL DEFAULT 0,
                                -- (cycle-2 addition) bumped each time completeDispatchSideEffects throws on this task; at threshold (5), task is marked failed with completion_exhausted
  completion_lease_at           TEXT,
                                -- (cycle-3 fix M23) durable lease for completeDispatchSideEffects.
                                -- Acquired before running completion; cleared on success/failure.
                                -- Reconciler skips tasks where completion_lease_at > now - 60s.
                                -- Defends against setImmediate-vs-reconciler concurrent runs.
  surface_mode                  TEXT NOT NULL DEFAULT 'pending'
                                CHECK (surface_mode IN ('pending', 'native_thread', 'headless')),
                                -- (cycle-3 fix S25) chosen ONCE during atomic admit (step 7) based on adapter
                                -- capability at that moment. Both happy-path completion and reconciler use this
                                -- stored mode rather than re-evaluating capability (which can change mid-task).
  created_at                    TEXT NOT NULL
);
CREATE INDEX idx_tasks_parent_session ON tasks(parent_session_id);
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_target_group ON tasks(target_agent_group_id);
CREATE INDEX idx_tasks_pending_admitted ON tasks(status, admitted_at) WHERE status IN ('pending', 'running');
-- Cross-orchestrator collision safety (review M6): caller-namespacing encoded structurally,
-- not by convention. Two orchestrators using the same idempotency_key cannot collide.
CREATE UNIQUE INDEX uq_tasks_caller_idempotency ON tasks(parent_session_id, idempotency_key);
```

**Cycle-3 fix M17 — `dispatch_task_id` lives in inbound.db, NOT central.db:** the previous cycle-2 plan to add `dispatch_task_id` to `sessions` (central.db) cannot work — the container has only a per-agent projection of central.db (`src/db/per-agent-projections.ts:128-131,142` allow-list = `{'backlog_items', 'ship_log'}` only). Instead, extend the inbound.db `session_routing` schema (`src/db/schema.ts:221`) — host writes `dispatch_task_id` on every container wake via `writeSessionRouting()`, container reads via a new `getSessionDispatchTaskId()` helper that opens `getInboundDb()` (analogous to existing `getSessionRouting()` in `container/agent-runner/src/db/session-routing.ts:19`):

```sql
-- Add to INBOUND_SCHEMA in src/db/schema.ts (cycle-3 fix M17 supersedes the prior central-DB ALTER TABLE):
ALTER TABLE session_routing ADD COLUMN dispatch_task_id TEXT;
```

This relocation has cascading benefits: `getSessionDispatchTaskId()` truly IS analogous to `getSessionRouting()`; mgId=null path still works (inbound.db exists for every session, regardless of mg); the central-DB projection allow-list stays untouched; the container's "two-DB IO surface" invariant from CLAUDE.md is preserved. A child session's first wake writes `session_routing.dispatch_task_id = <task_id>`; subsequent wakes preserve it via the standard `writeSessionRouting()` flow.

The single `status` column is the lifecycle. The artifact columns describe HOW MUCH of the dispatch has succeeded. There is no `dispatch_state` column.

**`task_id` derivation (review M6 + cycle-2 S16, supersedes decisions F7):** `task_id = 'dispatch-' + sha256(parent_session_id + ':' + idempotency_key).slice(0, 16)`. This preserves F7's deterministic property (caller can compute task_id locally before host round-trip) while structurally encoding caller-namespacing — two orchestrators with the same `idempotency_key` produce different `task_id`s because their `parent_session_id`s differ. The `UNIQUE(parent_session_id, idempotency_key)` index is belt-and-suspenders. The `dispatch-` prefix avoids confusion with scheduling's `task-` prefix. Coupling between idempotency_key TTL and task retention TTL stays per F7's documented trade-off.

### 3. MCP tools (container-side)

**Cycle-2 fix M12: tool mounting is BIFURCATED into two roles, not one.** Children of dispatched tasks need to call `dispatch_progress`/`dispatch_complete`/`dispatch_failed` but are NOT orchestrators (step 4 below denylists orchestrator-targeting). Two mounting conditions:

| Mounting condition | Tools mounted |
|---|---|
| Agent's group has `agent_group_capabilities.role = 'orchestrator'` | `dispatch_task`, `list_dispatched_tasks`, `dispatch_cancel` |
| Session has `sessions.dispatch_task_id IS NOT NULL` (task-child mode) | `dispatch_progress`, `dispatch_complete`, `dispatch_failed` |

These two conditions are NOT mutually exclusive — an orchestrator session that itself was dispatched by a higher-level orchestrator (theoretical; not a Phase 1 use case but the schema supports it) would mount both sets. The MCP server checks both conditions at container startup and mounts the union.

**`session_routing.dispatch_task_id` mechanism** (cycle-3 fix M17 supersedes cycle-2 M12): the child's task_id is stored in **inbound.db's `session_routing` table** (NOT central.db's `sessions` table — the container can't read those tables). Host's `writeSessionRouting()` writes it on every container wake. Container reads via new helper `getSessionDispatchTaskId()` that opens `getInboundDb()` — truly analogous to existing `getSessionRouting()` at `container/agent-runner/src/db/session-routing.ts:19`. Set during `applyDispatchTask` step 8 (after host writes the new tasks row but before waking the child).

**Sync-RPC vs async system-action protocol (review M13):** All dispatch MCP tools write `kind: 'system'` outbound rows. The host processes them later in the delivery loop. There is NO synchronous return-value channel from host handler back to MCP tool call. Two consequences for the MCP tool surface:
1. **`task_id` is computed LOCALLY** by the `dispatch_task` MCP tool using D8's deterministic hash (`'dispatch-' + sha256(parent_session_id + ':' + idempotency_key).slice(0,16)` — cycle-3 fix M18). The orchestrator agent already knows `parent_session_id` (via session_routing) and chooses `idempotency_key` itself. The MCP tool returns this locally-computed `task_id` immediately to the agent. **Implementation contract:** a single shared helper `deriveDispatchTaskId(parentSessionId, idempotencyKey)` MUST be used by both host and container; /team-tdd asserts both call sites produce identical IDs for the same inputs.
2. **Accept/reject results are delivered as inbound** back to the orchestrator session via `writeSessionMessage`. The MCP tool's return value is "submitted; status arrives shortly" — the agent learns the outcome on its next turn when host has processed the system action.

| Tool | Caller | Args | Purpose / Return |
|---|---|---|---|
| `dispatch_task` | Orchestrator | `target_group: string, content: string, idempotency_key: string, deadline?: ISO` | Computes `task_id` locally and returns it. Writes `kind='system'` outbound; host handler admits/rejects asynchronously and notifies via inbound. |
| `list_dispatched_tasks` | Orchestrator | `status?: enum, limit?: number` | **Direct DB read** from mounted central DB. Returns rows synchronously without host round-trip. |
| `dispatch_cancel` | Orchestrator | `task_id: string, reason?: string` | Writes `kind='system'` outbound. Host handler verifies `parent_session_id` match and applies cancellation. |
| `dispatch_progress` | Child | `task_id: string, message: string` | Writes `kind='system'` outbound. Fire-and-forget — child does not gate on confirmation. |
| `dispatch_complete` / `dispatch_failed` | Child | `task_id: string, summary: string, fail_reason?: string` | Writes `kind='system'` outbound. Host handler updates terminal state via compare-and-set. |

**`list_dispatched_tasks` implementation (review S4):** the orchestrator container has `data/v2.db` mounted RO (verify mount during /team-plan; A2 covers this assumption). The MCP tool queries `tasks WHERE parent_session_id = <from session_routing> ORDER BY created_at DESC LIMIT ?` directly. NO host round-trip. Matches existing `list_tasks` pattern at `container/agent-runner/src/mcp-tools/scheduling.ts:103-168`.

**How children populate `task_id` in their tool calls:** the agent reads `dispatch_task_id` from session metadata at session start (via `getSessionDispatchTaskId()` helper). The `dispatch_progress`/`dispatch_complete`/`dispatch_failed` tool argument schemas DEFAULT `task_id` to this value (the agent doesn't have to pass it explicitly — the tool injection layer fills it in). For cycle-2 explicitness: the FIRST INBOUND message to the child is a structured envelope `{kind:'chat', content: JSON.stringify({_dispatch: {task_id: '<id>'}, text: '<task_content>'})}` — the formatter (`container/agent-runner/src/formatter.ts`) renders the `text` field as the user-facing prompt and surfaces `_dispatch.task_id` as a system fact. The two paths (DB column + first-inbound envelope) are redundant on purpose: DB column survives session restart; first-inbound provides the agent's user-visible context.

### 4. System action handlers (host-side)

#### `applyDispatchTask(content, callerSession)`

**Cycle-2 model commitment (review M10, M11, M13, M14):** **Happy-path inline + reconciler-recovery.** The handler admits the task **atomically and synchronously** (steps 1-7 in a single transaction), then **fires the side-effect completion as a non-blocking in-process job** (`setImmediate(completeDispatchSideEffects, task_id)`). The handler returns to the delivery loop in tens of milliseconds, so a 22-task fanout admits at admit-rate. The reconciler is **pure recovery**: it re-runs `completeDispatchSideEffects` for any task whose host process crashed mid-completion, on the existing 60s sweep tick. No new timer added; no "5s sub-tick" implied.

**Atomic admit (steps 1-7) — single SQLite transaction. ORDER MATTERS (cycle-3 fix M20: idempotency replay precedes concurrency cap):**

1. **Auth:** check `agent_group_capabilities WHERE agent_group_id=callerSession.agent_group_id AND role='orchestrator'`. Reject otherwise via `notifyAgent(callerSession, 'dispatch rejected: not an orchestrator')`.
2. **Compute task_id and check idempotency replay FIRST (cycle-3 M20 reorder, Stripe canonical pattern):** `task_id = deriveDispatchTaskId(callerSession.id, idempotency_key)` (shared helper — see §3 contract). SELECT existing row by `(parent_session_id, idempotency_key)`:
   - If existing row found AND `existing.request_hash != computed_request_hash`: notify orchestrator with `idempotency_key_reused_with_different_payload`. Skip remaining steps.
   - If existing row found AND request_hash matches: re-notify with existing row's status. Skip remaining steps. **Replay is idempotent regardless of current cap.**
   - If no existing row: proceed to step 3 (genuine new admission).
3. **Concurrency cap (cycle-3 M20: only applies to NEW admissions):** count `tasks WHERE parent_session_id=callerSession.id AND status IN ('pending', 'running')`. If ≥ cap from `agent_group_capabilities.config_json`, reject with structured error notification.
4. **Target validation:** verify target_group exists, is not the orchestrator's own group, has no `agent_group_capabilities.role = 'orchestrator'` (no orchestrator-targeting orchestrators in v1).
5. **Wiring check (review M16):** if `callerSession.messaging_group_id IS NOT NULL`, require a row in `messaging_group_agents WHERE agent_group_id = target_agent_group_id AND messaging_group_id = callerSession.messaging_group_id`. If absent, reject with structured error `'target_not_wired_to_caller_messaging_group'`. Task-scoped grants are deferred to v2.
6. **Compute `request_hash`** using length-prefixed canonicalization (cycle-3 fix S23):
   ```
   request_hash = sha256(
     len(target_group)        + ':' + target_group        +
     len(content)             + ':' + content             +
     len(deadline ?? '')      + ':' + (deadline ?? '')
   )
   ```
   Length-prefixing prevents structural collisions when `content` (user-supplied) contains the `:` separator. Matches RFC-8785 / AWS-SigV4 canonicalization guidance.
7. **Determine `surface_mode` (cycle-3 fix S25):**
   - Compute `mode = (parent_messaging_group_id IS NOT NULL && hasCreateThreadCapability(channel_type)) ? 'native_thread' : 'headless'`. The capability check is **per-channel** — `getChannelAdapter(channel_type).createThread` exists (cycle-3 fix S24), NOT a wrapper-level `typeof` check.
8. **Atomic INSERT with ON CONFLICT:**
   ```sql
   INSERT INTO tasks (
     task_id, idempotency_key, parent_session_id, parent_agent_group_id,
     parent_messaging_group_id, target_agent_group_id, status, task_content,
     request_hash, deadline, admitted_at, surface_mode, created_at
   ) VALUES (?, ?, ?, ?, ?, ?, 'pending', ?, ?, ?, now, ?, now)
   ON CONFLICT(parent_session_id, idempotency_key) DO NOTHING
   RETURNING *;
   ```
   - **`RETURNING *`** (cycle-3 fix M19): the full row is returned to the handler so `completeDispatchSideEffects` has every column it needs without a follow-up SELECT.
   - If RETURNING gives a row: brand new task admitted. Proceed to side-effect completion.
   - If no row returned: a parallel admit won the race; SELECT and treat as idempotency replay (already handled in step 2 for sequential calls; this branch covers concurrent-call windows where step 2's SELECT and step 8's INSERT race).
   - `admitted_at` is set in the same INSERT — never NULL after a committed row. `surface_mode` is set once and persists.

**Sync admit notification:** after successful INSERT (post-transaction), `writeSessionMessage(callerSession.agent_group_id, callerSession.id, {kind: 'chat', content: 'Task admitted: <task_id>'})` and `wakeContainer(callerSession)`. **Cycle-3 fix S18 — failure mode documented:** if this post-txn write fails, log and continue; the side-effect completion's later notification carries the platform thread URL and serves as both acknowledgments. The transaction is already committed regardless.

**Side-effect completion** (`completeDispatchSideEffects(task_id)`, runs via `setImmediate(fn, task_id)` from the post-admit handler AND from the reconciler):

**Cycle-3 fix M19, M23 — entry contract:**

1. **Acquire completion lease (M23 — durable lock):**
   ```sql
   UPDATE tasks SET completion_lease_at = now
     WHERE task_id = ?
       AND status = 'pending'
       AND (completion_lease_at IS NULL OR completion_lease_at < now - 60s)
   RETURNING *;
   ```
   - If 0 rows updated: another worker holds the lease (or task isn't pending). Skip — return immediately.
   - If row returned: this worker has the lease. Proceed.
2. **In-process guard (M23 — fast-path mutex):** `dispatchModule.completionInFlight` is a `Map<task_id, Promise>`. Set `completionInFlight[task_id] = thisPromise` immediately after lease acquisition. Reconciler's setImmediate fan-out checks the map first and skips entries already in flight. Cleared in `finally`.
3. **Reload row at every step (M19):** every artifact step starts by re-reading the task row to see latest state. This handles cancellation/watchdog races (M22): if `status != 'pending'` mid-completion, abort.

**Threaded path** (`task.surface_mode = 'native_thread'`):

For each artifact step, the UPDATE is **status-guarded (cycle-3 fix M22)** to prevent overwriting cancel/fail terminal states:

1. Resolve adapter: `mg = getMessagingGroup(task.parent_messaging_group_id)` → `{channel_type, platform_id}`. `adapter = getChannelAdapter(channel_type)` (per-channel resolution — cycle-3 fix S24). If `adapter.createThread` is undefined despite `surface_mode='native_thread'`: mark `status='failed', fail_reason='adapter_unavailable'`.
2. **postParent step:** if `task.parent_platform_message_id IS NULL` (re-read):
   - Call `adapter.postParent(channel_type, platform_id, "Launched dispatch: <truncated content>")` → `{messageId}`.
   - `UPDATE tasks SET parent_platform_message_id = ? WHERE task_id = ? AND status = 'pending' AND parent_platform_message_id IS NULL`. If 0 rows: terminate (cancelled/failed by another path).
3. **createThread step:** if `task.child_platform_thread_id IS NULL` (re-read):
   - Call `adapter.createThread(channel_type, platform_id, task.parent_platform_message_id, title, firstMessage)` → `{threadId, messageId}`.
   - For Slack (cycle-3 fix M25): the adapter returns `{threadId: parent_platform_message_id, messageId: reply_ts}`. The `threadId` IS the parent message's ts (Slack's `thread_ts` semantic). `messageId` is the reply for editing/deletion.
   - `UPDATE tasks SET child_platform_thread_id = ?, child_messaging_group_id = ? WHERE task_id = ? AND status = 'pending' AND child_platform_thread_id IS NULL`. If 0 rows: terminate.
4. **openSession step:** if `task.child_session_id IS NULL` (re-read):
   - `resolveSession(target_agent_group_id, parent_messaging_group_id, child_platform_thread_id, 'per-thread')` → fresh child session.
   - **Cycle-3 fix M21 — write order is critical:**
     a. **Host-side writes FIRST:** `UPDATE tasks SET child_session_id = ?, started_at = now, last_progress_at = now, status = 'running' WHERE task_id = ? AND status = 'pending' AND child_session_id IS NULL`. If 0 rows updated: terminate.
     b. Update `session_routing.dispatch_task_id = task_id` for the child session (cycle-3 fix M17 — inbound.db, NOT central.db).
     c. `writeSessionMessage(target_agent_group_id, child_session_id, {kind: 'chat', content: JSON.stringify({_dispatch: {task_id}, text: task_content})})`.
     d. **THEN** `wakeContainer(child_session)`. The child cannot run until wake; by the time it can call `dispatch_progress`, `tasks.child_session_id` is set and `status='running'` — auth and CAS succeed.
5. **Completion notification:** `writeSessionMessage(parent_session_id, ...)` with the platform thread URL/message link (Slack: `https://<team>.slack.com/archives/<channel>/p<thread_ts>`; Discord: thread URL). `wakeContainer(parent_session)`.
6. **Release lease:** `UPDATE tasks SET completion_lease_at = NULL WHERE task_id = ?` and clear `completionInFlight[task_id]`.

**`headless` path** (`task.surface_mode = 'headless'`):

Skip postParent/createThread. Use a per-task synthetic-thread session, safe per C4 verification:

1. `resolveSession(target_agent_group_id, null, '<task_id>', 'per-thread')` — synthetic thread_id is internal session-key identity only. **C4 invariants:**
   - When `messaging_group_id IS NULL`, `writeSessionRouting` writes `channel_type=NULL, platform_id=NULL, thread_id=session.thread_id`. The non-null thread_id is benign because `delivery.ts` skips `adapter.deliver` whenever `channel_type` or `platform_id` is null.
   - **Maintainer invariant:** `sessions.thread_id` MUST never be passed to `adapter.deliver` directly — always go through `session_routing` (which carries the channel_type guard).
   - **Runtime assertion (Phase 1 IN-scope, cycle-3 fix S22):** add to `delivery.ts` and `session-manager.ts`: `channel_type IS NULL ⟺ platform_id IS NULL`. Defends against split-state bugs.
2. **Cycle-3 fix M21 — write order:**
   a. `UPDATE tasks SET child_session_id = ?, started_at = now, last_progress_at = now, status = 'running' WHERE task_id = ? AND status = 'pending' AND child_session_id IS NULL`. If 0 rows: terminate.
   b. Update `session_routing.dispatch_task_id = task_id` for the child session (inbound.db).
   c. `writeSessionMessage(target_agent_group_id, child_session_id, {kind: 'chat', content: JSON.stringify({_dispatch: {task_id}, text: task_content})})`.
   d. **THEN** `wakeContainer(child_session)`.
3. Notify parent_session that the task is now running (no platform URL).
4. Release lease.

**Failure handling in `completeDispatchSideEffects`:** any step that throws (adapter call fails, DB write conflict, target session creation fails) leaves the corresponding artifact column NULL, releases the lease, and exits. The reconciler's next 60s sweep retries idempotently. After 5 consecutive failures (`dispatch_completion_attempts` counter), mark `status='failed'` with `fail_reason='completion_exhausted'`.

**Crash recovery (reconciler) — pure recovery role on the existing 60s sweep tick (review M11):**

- Scans `WHERE status='pending' AND admitted_at IS NOT NULL AND child_session_id IS NULL AND (completion_lease_at IS NULL OR completion_lease_at < now - 60s)` → for each, schedules `setImmediate(completeDispatchSideEffects, task_id)`. Lease acquisition in step 1 of `completeDispatchSideEffects` itself dedupes against any in-flight setImmediate from the original admit.
- If reconciler observes `surface_mode='native_thread'` but `getChannelAdapter(channel_type).createThread` is undefined (adapter version mismatch since admit): mark `failed` with `fail_reason='adapter_unavailable'`.
- The `admit_crashed` branch from the prior cycle is REMOVED — atomic INSERT in step 8 makes a partially-admitted row impossible.

**External side-effect retry safety (review S12, accepted limitation):** if the host crashes between `adapter.postParent` succeeding (Slack/Discord side) and the host's `UPDATE tasks SET parent_platform_message_id = ?` committing, the reconciler will see a NULL artifact column and retry `postParent` — producing one duplicate parent platform message. **Documented as acceptable for personal-scale single-host Phase 1.** Deduplication via metadata-correlation (Slack `metadata.event_payload`, Discord embed footer with `task_id`) is deferred to v2 hardening.

### 5. Watchdog (host-sweep extension)

**Where it runs in `host-sweep.ts` (review S3):** at the **top level of the sweep tick**, AFTER `sweepSession` has completed for all sessions in this tick (so a kill-by-stuck-container in the per-session pass is already reflected in `child_session.container_status='stopped'` when we read it). Order within the tick:

```
sweep tick (60s):
  for each session: sweepSession(session)        — existing per-session loop
  reconcileDispatchTasks()                       — new: complete admitted-but-incomplete tasks (idempotent)
  for each task WHERE status='running': decideTaskAction(task)  — new: terminal-state watchdog
  // each iteration in try/catch like sweepSession:163-169
```

Reconciler runs first so any newly-admitted tasks complete their side-effects before the watchdog reaper looks at them. Watchdog runs last so a watchdog-fail doesn't race with a reconciler-completion in the same tick.

Add one new pure decision function `decideTaskAction(args)` returning `{action: 'ok' | 'fail-no-progress' | 'fail-deadline' | 'fail-container-exit' | 'fail-spawn-deadline'}`. Inputs:

- `now` (sweep timestamp)
- `task` row (status IN ('pending', 'running'))
- `child_session.container_status` (null when child_session_id is null, e.g. tasks still mid-admit)
- `has_pending_outbound(child_session_id)` — read from outbound.db; true if `messages_out` rows exist that the host hasn't yet delivered/processed
- `task.last_progress_at`, `task.deadline`, `task.started_at`, `task.admitted_at`
- `no_progress_timeout`, `spawn_deadline` from orchestrator's `agent_group_capabilities.config_json` (defaults: 30 min, 5 min)

**Decision logic — drain-first scoped narrowly to PENDING TERMINAL system actions, with grace cap (review M15):**

The cycle 2 review found that an over-broad drain-first guard (any pending outbound suppresses all terminal failures) can cause indefinite starvation under stuck/undeliverable outbound. Cycle-2 fix narrows drain-first to only those outbound rows that could legitimately produce a terminal dispatch decision, and caps the grace window.

```
# Hard deadline ALWAYS overrides drain-first (deadlines are user contracts).
if task.deadline AND now > parse(task.deadline):
    return 'fail-deadline'

# Spawn deadline: status='pending' tasks that have admitted but never started
# within spawn_deadline window. Side-effect completion is the reconciler's job.
if task.status == 'pending' AND task.admitted_at IS NOT NULL AND task.started_at IS NULL:
    if (now - parse(task.admitted_at)) > spawn_deadline:
        return 'fail-spawn-deadline'
    return 'ok'   # still within spawn window, reconciler will complete

# Drain-first guard NARROWED + grace measured against pending row's own timestamp (cycle-3 fix M24):
# Only suppress terminal-fail when the child's outbound queue contains a pending
# dispatch_complete OR dispatch_failed system action. Grace window starts when
# THAT row appeared in outbound, NOT against last_progress_at. A long-quiet child
# that just emitted dispatch_complete deserves 120s of grace from THAT moment.
terminal_seen_at = pending_terminal_dispatch_outbound_seen_at(task.child_session_id)
if terminal_seen_at IS NOT NULL:
    drain_age = now - parse(terminal_seen_at)
    if drain_age <= drain_grace_window:    # default 120 seconds
        return 'ok'   # let the terminal system action land
    # else fall through to terminal evaluation; drain-first has been waiting too long.

# No-progress timer (review S1: triple fallback so tasks pre-heartbeat are still observed)
last_signal = task.last_progress_at OR task.started_at OR task.admitted_at
if (now - parse(last_signal)) > no_progress_timeout:
    return 'fail-no-progress'

# Container exit (only meaningful for status='running' children)
if task.child_session_id IS NOT NULL AND child_session.container_status == 'stopped':
    return 'fail-container-exit'

return 'ok'
```

**`pending_terminal_dispatch_outbound_seen_at(child_session_id)`** returns the earliest `messages_out.timestamp` for a pending terminal-dispatch system action, or NULL if none exists. SQL with **corrected precedence (cycle-3 fix S20):**

```sql
SELECT MIN(timestamp) FROM messages_out
 WHERE kind='system' AND (
   content LIKE '%"action":"dispatch_complete"%'
   OR content LIKE '%"action":"dispatch_failed"%'
 );
```

Note the parentheses — the cycle-2 design's SQL had `WHERE kind='system' AND ... OR ...` which would match any `dispatch_failed` row regardless of `kind`. (Implementation in production code MUST deserialize content JSON to match `action` exactly; LIKE is shown here for spec clarity, not for production use.) Generic chat outbound, status messages, and other system actions do NOT trigger drain-first.

**Scaling note (S20):** this SQL opens the child's outbound.db once per task per 60s tick. For Phase 1 (≤50 concurrent dispatched tasks), the cost is negligible. If task count exceeds 100, add a host-side cache populated by the existing delivery poll (which already opens outbound.db) keyed on `(session_id, terminal_outbound_seen_at | NULL)` and refreshed on the same poll tick.

Reaper applies failure with **compare-and-set guards** (review M7) so a race-arriving `dispatch_complete` system action can land cleanly:

```
UPDATE tasks SET status='failed', failed_at=now, fail_reason=<reason>
  WHERE task_id=? AND status IN ('pending', 'running')
```

If 0 rows updated, the task already terminated via another path — skip the parent notification. Otherwise: writeSessionMessage to parent with `kind='system'` summary, then wakeContainer(parent).

`dispatch_progress` is fire-and-forget (review S1): the child agent does NOT receive confirmation. The host writes the column update and returns; failures are logged but not propagated.

### 6. Adapter capability — `createThread` and `postParent`

**Interface boundary (review M3 + M4):** The dispatch handler runs in `src/modules/orchestrator-dispatch/` (host core). Core's existing adapter access point is `ChannelDeliveryAdapter` (`src/delivery.ts:69-80`), a NARROWER interface than `ChannelAdapter` (`src/channels/adapter.ts:127`). Existing system action handlers (e.g., `applyCreateAgent`) never call adapter methods directly — they write through `writeSessionMessage`. Two valid placements for the new methods:

- **(a) Widen `ChannelDeliveryAdapter`** to include the new methods. Core's `getDeliveryAdapter()` returns an interface that supports them. Cleanest from a dependency-direction POV, matches how `setTyping`/`deleteMessage` were added previously.
- **(b) Have dispatch handler import from `./channels/channel-registry.js`** to obtain the full `ChannelAdapter`. Introduces a NEW core→channels dependency.

**Choosing (a)** — preserves the existing core/channels boundary; matches existing precedent. Auto-judgment recorded.

The dispatch handler resolves `messaging_group_id → { channel_type, platform_id }` via `getMessagingGroup(mgId)` BEFORE calling the adapter (see §4 step 8).

**Per-adapter signatures (review M9):** Per-adapter `ChannelAdapter` implementations DROP `channelType` to match existing per-adapter convention (`deliver(platformId, threadId, message)` at `adapter.ts:149`, `setTyping(platformId, threadId)` at `adapter.ts:152`, `deleteMessage(platformId, threadId, messageId)` at `adapter.ts:159`).
```ts
// Per-adapter (ChannelAdapter in src/channels/adapter.ts widening + slack.ts/discord.ts implementations)
interface ChannelAdapter {
  // existing methods...
  postParent?(platformId: string, text: string): Promise<{ messageId: string }>;
  createThread?(platformId: string, parentMessageId: string, title: string, firstMessage: string): Promise<{ threadId: string; messageId: string }>;
}
```

**Slack implementation** (`src/channels/slack.ts`):
- `postParent(platformId, text)`: `chat.postMessage({ channel: platformId, text })` returns `{ messageId: response.ts }`.
- `createThread(platformId, parentMessageId, title, firstMessage)` (cycle-3 fix M25 — return semantics):
  - `const reply = await chat.postMessage({ channel: platformId, thread_ts: parentMessageId, text: firstMessage })`.
  - Returns `{ threadId: parentMessageId, messageId: reply.ts }`. The `threadId` IS the parent message's `ts` — that's what Slack uses as `thread_ts` for subsequent posts to the thread. The reply's own `ts` (`messageId`) is for editing/deleting THAT specific message. Persisting `parentMessageId` as `child_platform_thread_id` ensures all subsequent child outbound posts use the correct `thread_ts`.

**Discord implementation** (`src/channels/discord.ts`):
- `postParent(platformId, text)`: send a message to the channel by `platformId`, return the message id.
- `createThread(platformId, parentMessageId, title, firstMessage)`: `channel.threads.create({ name: title, startMessage: parentMessageId, ... })` returns the new thread's ID.

**Bridge wrapper** (`src/index.ts:215-243`): the existing pattern looks up the per-adapter `ChannelAdapter` via `getChannelAdapter(channelType)` and forwards the call without `channelType`. New methods follow this convention — the wrapper takes `(channelType, platformId, ...)`, looks up the adapter, calls `adapter.postParent(platformId, text)` or `adapter.createThread(platformId, ...)`. Identical pattern to existing `setTyping` and `deleteMessage` wiring.

Both adapter source files live on the `channels` branch (per CLAUDE.md "Channels and Providers (skill-installed)" section). Phase 1 ships these as additions to those adapters; the work happens on the channels branch and is merged back via the existing skill-update flow. The HOST-side `ChannelDeliveryAdapter` interface widening (wrapper signatures with `channelType`) lands in trunk; only the per-adapter implementations land on the channels branch.

Adapters that don't implement these methods (Telegram, iMessage, WhatsApp, email): `dispatch_task` checks `typeof adapter.createThread === 'function'` on the wrapper; absent ⇒ mgId=null path.

### 7. Dashboard topology (Phase 2 — read-only)

**Webhook-server refactor (review M8):** Phase 2 begins with a bounded refactor of `src/webhook-server.ts`, NOT a route-map extension. The current implementation (`webhook-server.ts:97-110`) has a routes Map keyed on adapter-name AFTER the literal `/webhook/` prefix and returns 404 for any path that doesn't match. Static-asset serving doesn't exist. SSE streams would be incompatible with the current `fromWebResponse` (lines 52-67), which reads a Web Response stream to completion before calling `nodeRes.end()` — fatal for an infinite SSE stream.

The refactor introduces a multi-route HTTP router (~200 LOC of routing logic) with three endpoint families:

1. **Webhook adapter routes** (preserves existing `/webhook/<adapter>` semantics; the existing routes Map continues to drive these).
2. **Static-asset routes** (`/dashboard/static/*`): serves Vite-built bundle from `dist/dashboard/`. Uses `fs.createReadStream` with content-type detection from extension. ETag headers for cache.
3. **API/SSE routes** (`/dashboard/api/*`): handlers write Node response streams directly, bypassing `fromWebResponse`. SSE handlers send `Content-Type: text/event-stream`, periodic `:keepalive\n\n` comment frames every 25s, and `event: <type>\ndata: <json>\n\n` payloads on chokidar events.

Constraint C2 ("piggyback on existing Node host") is preserved at the *process* level — still one Node process. Assumption A7 is updated below to reflect this is a real refactor, not just a route-map insert. Memory budget C1 incorporates the routing layer overhead (small — ~5MB Node code).

New routes registered after the refactor:

| Route | Purpose | Auth |
|---|---|---|
| `GET /dashboard/` | Serves SPA index.html | Owner cookie |
| `GET /dashboard/static/*` | Vite-built JS/CSS bundles | Owner cookie |
| `GET /dashboard/api/auth/me` | Returns user id and isOwner | None (auth probe) |
| `POST /dashboard/api/auth/exchange` | Owner DM token → cookie | Token in body |
| `GET /dashboard/api/tasks` | List tasks (filter by status, target_group) | Owner cookie |
| `GET /dashboard/api/tasks/:id` | Task detail incl. recent inbound/outbound from child session | Owner cookie |
| `GET /dashboard/api/sessions` | All live sessions across groups | Owner cookie |
| `GET /dashboard/api/events` | SSE stream for live updates | Owner cookie |

**Auth flow:**
1. Owner sends a `/dashboard-token` command in DM with the orchestrator agent (or `nanoclaw` core).
2. Host generates a single-use token (24h TTL, hex-encoded), DMs it back.
3. Owner pastes token at `https://localhost:3000/dashboard/?token=<>` (or via UI box).
4. Server exchanges token for an `httpOnly` `Secure` cookie bound to user id.
5. All `/dashboard/api/*` requests require the cookie.

**SSE feed:** chokidar watches `data/v2-sessions/*/outbound.db` and `data/v2.db`. On change events, emit summarized JSON over SSE. Debounced 200ms to coalesce burst writes during agent turns.

**SPA structure** (Vite + React):
- Three top-level views: `KanbanBoard`, `TaskDetail`, `SessionList`.
- State: SWR for fetching, EventSource for SSE.
- No router beyond URL hash anchors; dashboard is a single page with tab switching.
- Visual styling — gated by `bootstrap-domain:jony-ive` and `impeccable:impeccable` audits during build.

**Memory reconciliation gate (C1):** at /team-ship, run `ps -o rss= -p $NODE_HOST_PID` snapshots over 60 minutes with 5 active live tasks. If >150MB resident incremental over a baseline (no dashboard) Node process, optimize chokidar watch scope or revisit framework.
