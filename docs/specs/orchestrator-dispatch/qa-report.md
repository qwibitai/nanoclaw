## Carry-Forward Known Risks

These findings are real but not blocking Phase 1 ship. Each has a documented mitigation or is consistent with design intent.

### K7 — Orphaned-running task on host crash mid-completion
**Codex E2 (HIGH, conf 0.92), Concurrency #3.** If host crashes between `dispatch.ts` step (a) (status='running' UPDATE) and step (c) (`writeSessionMessage`), the task is `status='running'` with `child_session_id` set, but the child has no first-inbound and was never woken. `acquireCompletionLease` (`status='pending'`) and `getOrphanedTasks` (`child_session_id IS NULL`) both skip. Watchdog catches via no-progress timeout within ~30 min and reaps. **Mitigation:** capacity slot held for up to 30 min, but no data loss; user-visible failure with clear `fail_reason='no_progress_timeout'`. Architectural fix (per-step recovery markers + idempotent re-run) deferred to follow-up.

### K8 — 2-min hard-kill timer lost on restart
**Concurrency #4, Adversarial #5, Security #3.** `setTimeout` for the cancellation hard-kill is in-process. If host restarts within the 2-min window, the timer is dropped — child container runs until natural exit or until watchdog reaps via `container_exit` detection (next 60s sweep). **Mitigation:** Watchdog handles orphaned-running containers within 60s. Durable replacement (cancellation-aware watchdog branch) is small follow-up.

### K9 — Prompt injection via `result_summary` / `reason` into orchestrator context
**Security #5, #6.** Child `dispatch_complete` summary and orchestrator `dispatch_cancel` reason are interpolated into the parent session's chat content. A malicious child or compromised orchestrator could inject instructions. **Mitigation:** Orchestrator's `CLAUDE.md` must treat task summaries as untrusted external data. Structural mitigation (XML-style wrapping `<task_result task_id="...">...</task_result>`) deferred.

### K10 — Cross-tenant within-group via raw SQLite (Codex E3 HIGH, conf 0.9)
The per-agent projection copies the full `tasks` table filtered by `parent_agent_group_id`. Containers have raw SQLite access via shell — they can `SELECT *` and see all rows for their agent_group, not just their session's. **This matches design intent** (agent_group is the documented trust boundary; the MCP-layer filter by parent_session_id is for UX, not isolation). **Mitigation:** Defensive tightening to session-scoped projection (filter by current session_id, not just agent_group) is a session-isolation enhancement. Deferred — out of scope for Phase 1's "agent_group = trust boundary" model.

### K11 — Schema cleanup: `surface_mode = 'pending'` is dead surface
**Adversarial #3, Domain #3.** `Task.surface_mode` type union and migration 026's CHECK constraint include `'pending'`, but application code always sets `'native_thread'` or `'headless'` at admit. The dead value is never written. **Mitigation:** Cleanup is non-functional (zero runtime impact). Defer to follow-up migration that narrows the union and CHECK.
