# Skill 設計原則

參考 Anthropic 官方 skill-creator 的設計哲學。

## 核心原則

### 1. 簡潔設計至關重要

> "The context window is a public good."

只包含 Claude 尚未具備的資訊。挑戰每一個添加的必要性。

**好的做法：**
- 專注於 Claude 不知道的特定領域知識
- 提供清晰、可操作的指令
- 避免重複基礎概念

**不好的做法：**
- 解釋基本程式設計概念
- 包含 Claude 已知的通用知識
- 冗長的背景說明

### 2. 適當的具體性

根據任務複雜度匹配細節層級：

- **文字指令** - 用於需要靈活方法的任務
- **腳本** - 用於確定性操作
- **參考文件** - 用於領域知識

### 3. 描述觸發使用時機

frontmatter 的 description 是關鍵：
- 應該指定「何時使用」
- 不要只說「是什麼」
- 簡潔但具體

**好的範例：**
```yaml
description: Generate UUID v4 identifiers for unique resource identification
```

**不好的範例：**
```yaml
description: A UUID generator
```

## Skill 結構

### 必需的元素

1. **SKILL.md** - 包含 frontmatter 和指令
   ```yaml
   ---
   name: skill-name
   description: When to use this skill
   ---
   ```

### 可選的資源

2. **scripts/** - 可執行程式碼
   - 用於需要可靠性或重複任務
   - 提供確定性結果
   - 減少上下文使用

3. **references/** - 文檔
   - 按需載入
   - 領域特定知識
   - API 規格、標準等

4. **assets/** - 模板和檔案
   - 用於最終輸出的模板
   - 配置檔案範例
   - 靜態資源

## 開發流程

1. **理解** - 收集具體使用案例
2. **規劃** - 決定可重用的內容
3. **初始化** - 使用 init_skill.py
4. **編輯** - 完善 SKILL.md 和資源
5. **驗證** - 使用 validate_skill.py
6. **迭代** - 基於實際使用改進

## 最佳實踐

### 清晰的指令

- 使用命令式語氣
- 提供具體範例
- 說明預期輸出

### 模組化

- 每個 skill 專注於一個領域
- 相關功能組合在一起
- 避免過度細分

### 可測試性

- 包含測試案例
- 提供驗證方法
- 確保可重現性

### 文檔化

- 說明使用時機
- 提供參數說明
- 包含範例輸出

## 常見模式

### 工具類 Skill

提供具體工具或功能：
- UUID 生成器
- 日期格式轉換
- 文字處理

### 工作流程 Skill

引導完成複雜任務：
- 文件協作
- 程式碼審查
- 測試流程

### 知識類 Skill

提供領域專業知識：
- 品牌指南
- API 規格
- 設計系統

## 避免的陷阱

1. **過度通用** - Skill 太廣泛，缺乏焦點
2. **過度具體** - 只適用於極少數情況
3. **冗長說明** - 浪費上下文空間
4. **缺少範例** - 使用者不知道如何使用
5. **未經測試** - 可能無法正常運作

## 持續改進

- 觀察實際使用情況
- 收集用戶反饋
- 優化常用路徑
- 移除未使用的部分
- 合併相似功能
