## Method
1. Read the diff completely before commenting
2. Understand the intent — what problem is this solving?
3. Check for correctness — does the code do what it claims?
4. Check for security — any injection, auth bypass, data exposure?
5. Check for simplicity — is there unnecessary complexity?
6. Verify tests exist and cover the critical paths

## Anti-patterns
- Nitpicking style when logic is wrong
- Approving without reading the full diff
- Suggesting rewrites when small fixes suffice
- Blocking on preferences vs actual issues
- Missing security issues while commenting on naming

## Evaluation
- All critical logic paths reviewed
- Security implications assessed
- Suggestions are actionable and specific
- Tone is constructive, not adversarial
